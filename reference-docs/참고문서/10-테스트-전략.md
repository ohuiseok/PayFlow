# 테스트 전략

PayFlow의 테스트 전략은 "API가 성공 응답을 주는가"보다 "돈이 중복으로 움직이지 않는가", "실패 후 복구 가능한 상태가 남는가"에 초점을 둡니다.

## 테스트 목표

- 회원가입, 로그인, 지갑 생성의 기본 사용자 흐름 검증
- 지갑 입금/출금의 잔액 정합성 검증
- 송금 요청의 멱등성 검증
- 출금 이후 실패 시 보상 필요 상태 검증
- outbox 이벤트 저장과 발행 상태 검증
- ledger-service의 이벤트 중복 소비 방어 검증
- 프론트엔드 API adapter와 화면 흐름 검증

## 테스트 계층

| 계층 | 목적 | 예시 |
| --- | --- | --- |
| Unit Test | 도메인 규칙과 서비스 로직 검증 | 잔액 부족, 상태 전이, request hash |
| Slice/Controller Test | API 계약과 에러 응답 검증 | controller status code, DTO validation |
| Integration Test | DB transaction, repository, service 협력 검증 | wallet transaction 저장, idempotency unique |
| Contract/Smoke Test | 실제 API 흐름 검증 | `scripts/api-smoke.ps1` |
| Frontend E2E | 사용자 화면 흐름 검증 | Playwright |
| Load Test | latency, 실패율, 병목 확인 | k6 시나리오 |

## 서비스별 검증 포인트

### user-service

- 같은 email로 중복 가입 방지
- password hashing 적용
- login 성공 시 access token 반환
- role이 `PARENT`, `CHILD`로 저장되는지 확인

### wallet-service

- 사용자별 지갑 1개 생성
- 입금/출금 후 `wallet_transactions` 기록
- 잔액 부족 시 출금 실패
- 같은 `referenceType` + `referenceId` 요청이 중복 반영되지 않음
- 동시 요청에서 최종 잔액이 틀어지지 않음

### banking-service

- 계좌 등록 시 masking/hash 정책 적용
- 같은 idempotency key와 같은 body는 기존 결과 반환
- 같은 idempotency key와 다른 body는 conflict
- Open Banking ambiguous response가 즉시 성공/실패로 단정되지 않음
- compensation API가 지갑 환불을 한 번만 반영

### transfer-service

- 송금 요청 생성과 상태 전이
- sender wallet 출금, receiver wallet 입금 순서
- 같은 idempotency key 중복 요청 방어
- 출금 후 입금 실패 시 `COMPENSATION_REQUIRED`
- refund 성공 시 `COMPENSATED`
- 송금 완료/실패 시 outbox event 저장
- outbox relay retry, stale processing recovery

### reward-service

- 부모-자녀 연결 생성
- 부모만 미션 생성 가능
- 자녀 제출, 부모 승인/반려 상태 전이
- 승인된 미션만 보상 지급 가능
- 보상 지급 idempotency key: `reward-payment-{missionId}`

### ledger-service

- `transfer.completed` 이벤트 소비 시 원장 기록 생성
- 같은 transferId 이벤트 중복 소비 방어
- `transfer.failed` 이벤트 소비 시 실패 추적 저장

## 검증 명령

백엔드 서비스별 테스트:

```powershell
.\gradlew.bat test
```

프론트엔드 검증:

```powershell
cd sample-react
npm run check
```

Docker Compose 설정 검증:

```powershell
docker compose config --quiet
```

API smoke test:

```powershell
.\scripts\api-smoke.ps1
```

## 결제성 테스트에서 중요한 Assertion

일반 CRUD 테스트:

```text
요청 성공 여부
응답 status code
응답 body shape
```

PayFlow에서 추가로 보는 것:

```text
최종 wallet balance
wallet_transactions 중복 여부
idempotency key 재요청 결과
transfer status
outbox event status
ledger entry 중복 여부
compensation retry count
failure reason 저장 여부
```

## 미검증 리스크

현재 문서 기준으로 명확히 분리해야 할 리스크입니다.

- 전체 Docker Compose runtime smoke test는 로컬 Docker Desktop 상태에 의존합니다.
- Kafka/Redis/MySQL을 실제 컨테이너로 묶은 Testcontainers 통합 테스트는 추가하면 좋습니다.
- 부하 테스트 결과 수치는 실제 k6 실행 후 채워야 합니다.
- 운영 수준의 chaos test, network timeout test는 별도 확장 과제입니다.
