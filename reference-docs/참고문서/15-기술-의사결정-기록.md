# 기술 의사결정 기록

> 도메인 전환 안내: 현재 PayFlow는 **청년 정책 참여 미션 및 지원금 지급 플랫폼**으로 설명한다. 내부 구현 호환성을 위해 `PARENT`/`CHILD`, `/api/families`, `/api/missions`, `/api/cashbook`, `reward-service` 같은 명칭은 유지하지만, 문서와 발표에서는 각각 **기관 담당자**, **청년 참여자**, **참여자 연결**, **정책 미션**, **지원금 사용 내역**, **정책 미션/지원금 서비스**로 해석한다.

이 문서는 PayFlow 구현 중 주요 기술 선택과 trade-off를 기록합니다.

## ADR-001. wallet-service를 잔액의 단일 진실 공급원으로 둔다

### Context

banking-service, transfer-service, reward-service 모두 돈의 이동과 관련이 있습니다. 각 서비스가 직접 잔액을 수정하면 정합성 책임이 분산됩니다.

### Decision

지갑 잔액 변경은 wallet-service만 수행합니다.

### Consequence

장점:

- 잔액 정합성 책임이 명확함
- wallet transaction 기록을 한 곳에서 관리
- 중복 반영 방어를 공통화 가능

단점:

- 다른 서비스가 wallet-service에 의존
- 송금 latency가 내부 HTTP 호출에 영향받음

## ADR-002. 서비스별 DB를 분리한다

### Context

MSA 포트폴리오에서 서비스 경계와 데이터 소유권을 보여주고 싶었습니다.

### Decision

서비스별 DB를 분리하고 cross-service DB join을 사용하지 않습니다.

### Consequence

장점:

- 서비스 책임과 소유권이 명확함
- 특정 서비스 schema 변경 영향이 제한됨

단점:

- 조회 조합이 복잡해짐
- API 호출 또는 이벤트 기반 데이터 동기화가 필요함

## ADR-003. 분산 트랜잭션 대신 보상 트랜잭션을 사용한다

### Context

송금은 sender 출금과 receiver 입금이 모두 성공해야 합니다. 하지만 MSA에서 여러 서비스/DB를 하나의 ACID transaction으로 묶는 것은 비용이 큽니다.

### Decision

2PC를 사용하지 않고 상태 전이와 보상 트랜잭션으로 처리합니다.

### Consequence

장점:

- 구현과 운영 복잡도가 낮음
- 실패 상태를 명시적으로 추적 가능
- 현실적인 결제 장애 복구 모델을 보여줄 수 있음

단점:

- 일시적 불일치를 허용해야 함
- 보상 API와 운영 절차가 필요함

## ADR-004. 송금 이벤트 발행에 transactional outbox를 사용한다

### Context

송금 성공 DB commit 후 Kafka publish가 실패하면 ledger-service 원장 기록이 누락될 수 있습니다.

### Decision

transfer 상태 변경과 outbox event 저장을 같은 DB transaction에 묶습니다.

### Consequence

장점:

- 이벤트 발행 의도가 DB에 남음
- Kafka 장애 시 retry 가능
- 운영 API로 outbox 적체를 관찰 가능

단점:

- relay 구현 필요
- outbox table 관리 필요
- exactly-once가 아니라 consumer idempotency가 여전히 필요

## ADR-005. 송금 실행은 동기 HTTP, 원장 기록은 비동기 Kafka로 처리한다

### Context

사용자 송금 응답은 비교적 즉시 결과를 반환해야 합니다. 반면 원장 기록은 송금 성공 후 event-driven으로 처리해도 됩니다.

### Decision

wallet debit/credit은 동기 HTTP로 처리하고, ledger 기록은 Kafka 이벤트로 처리합니다.

### Consequence

장점:

- 사용자는 송금 결과를 바로 알 수 있음
- ledger-service 장애가 송금 API를 직접 막지 않음
- 이벤트 기반 확장 가능

단점:

- 송금과 원장 사이 eventual consistency 발생
- Kafka consumer 지연 관찰 필요

## ADR-006. API 멱등성과 지갑 반영 멱등성을 분리한다

### Context

같은 API 요청 반복과 같은 원천 거래의 지갑 중복 반영은 비슷하지만 다른 문제입니다.

### Decision

API 단계는 `Idempotency-Key` + `requestHash`, wallet 단계는 `referenceType` + `referenceId`로 방어합니다.

### Consequence

장점:

- 각 계층에서 독립적으로 중복 방어 가능
- retry, timeout, 내부 재호출에 강함

단점:

- 키 설계 규칙을 문서화해야 함
- referenceType 관리가 필요함

## ADR-007. Open Banking HTTP 성공과 금융 성공을 분리한다

### Context

외부 금융 API는 HTTP 응답이 성공이어도 실제 금융 거래가 최종 성공이 아닐 수 있습니다.

### Decision

외부 응답을 banking transfer 상태로 저장하고, 모호한 결과는 result-check로 최종화합니다.

### Consequence

장점:

- timeout/processing을 실패로 오판하지 않음
- 금융 도메인의 불확실성을 상태로 표현

단점:

- 상태 모델이 복잡해짐
- scheduler와 수동 result-check API가 필요함

## ADR-008. settlement-service는 확장 영역으로 둔다

### Context

정산은 결제 시스템에서 중요한 영역이지만 MVP 범위가 커질 수 있습니다.

### Decision

settlement-service는 구조에 포함하되, 핵심 MVP는 지갑/송금/보상/원장 중심으로 완성합니다.

### Consequence

장점:

- 향후 정산/배치 확장 경로를 보여줄 수 있음
- 현재 구현 범위를 통제 가능

단점:

- 포트폴리오 설명에서 구현 완료 범위와 확장 예정 범위를 명확히 구분해야 함

## ADR-009. Docker Compose로 로컬 재현성을 확보한다

### Context

MSA 프로젝트는 여러 서비스와 인프라를 함께 실행해야 합니다.

### Decision

Nginx, API Gateway, MySQL, Redis, Kafka, 각 서비스를 Docker Compose로 구성합니다.

### Consequence

장점:

- 로컬 실행 경로가 명확함
- 포트폴리오 리뷰어가 구조를 이해하기 쉬움

단점:

- 로컬 머신 리소스 사용량 증가
- Docker Desktop 상태에 따라 전체 smoke test가 제한될 수 있음
