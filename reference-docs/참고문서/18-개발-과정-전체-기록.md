# 개발 과정 전체 기록

이 문서는 PayFlow가 어떤 순서로 설계되고 구현되었는지 블로그용으로 복원한 개발 기록입니다.

## 0. 출발점

목표는 단순 CRUD가 아니라 결제성 시스템의 핵심 문제를 작게 구현하는 것이었습니다.

초기 목표:

- 사용자별 지갑 생성
- 은행 충전
- 사용자 간 송금
- 부모-자녀 미션 보상
- 송금 결과 원장 기록
- 멱등성, 실패 복구, 테스트까지 설명 가능한 MVP

초기 설계 원칙:

- 하나의 기능은 끝까지 동작하게 만든다.
- 테이블은 실제 API와 서비스 흐름에서 쓰는 것만 둔다.
- 잔액 변경은 wallet-service만 수행한다.
- 서비스 간 DB join은 하지 않는다.
- 실패는 상태값과 복구 API로 남긴다.

## 1. 프로젝트 골격 만들기

구성한 것:

- MSA 디렉터리 구조
- 각 Spring Boot 서비스
- Docker Compose
- MySQL, Redis, Kafka, Nginx
- `.env`, `.env.example`
- 공통 에러 응답과 보안 규칙

블로그 포인트:

- 처음부터 모든 기능을 만들지 않고 실행 가능한 뼈대를 먼저 만들었다.
- 환경 변수와 compose를 먼저 잡아야 서비스 간 연결 문제를 빨리 발견할 수 있다.

## 2. user-service

구현한 것:

- 회원가입
- 로그인
- 사용자 조회
- role: `PARENT`, `CHILD`
- status: `ACTIVE`, `LOCKED`, `WITHDRAWN`
- JWT 발급
- 회원가입 후 wallet-service 지갑 생성 연계

중요한 결정:

- 인증 이후 사용자 식별은 client body가 아니라 Gateway가 주입한 user header를 기준으로 한다.

블로그 포인트:

- user-service는 작지만 전체 시스템의 사용자 식별 기준을 만든다.

## 3. wallet-service

구현한 것:

- 사용자별 지갑 생성
- 지갑 조회
- 입금/출금
- wallet transaction 저장
- 잔액 부족 처리
- reference 기반 중복 반영 방어
- DB row lock 또는 transaction 기반 정합성 제어

중요한 결정:

- PayFlow에서 실제 잔액은 wallet-service만 변경한다.
- 다른 서비스는 wallet-service API를 호출할 뿐 직접 balance를 수정하지 않는다.

블로그 포인트:

- 결제성 시스템에서는 "잔액"과 "잔액 변경 근거"가 항상 함께 움직여야 한다.

## 4. banking-service

구현한 것:

- 계좌 등록
- 계좌 목록 조회
- 충전 요청
- 출금 요청
- Open Banking authorization/callback/sync 모델
- token 암호화 저장 방향
- result-check
- compensation API

중요한 결정:

- HTTP 성공과 금융 거래 성공을 분리한다.
- 권한 없는 Open Banking API는 business success로 연결하지 않는다.
- 모호한 응답은 실패로 단정하지 않고 상태로 남긴다.

블로그 포인트:

- 외부 금융 API는 CRUD API처럼 다루면 안 된다.

## 5. transfer-service

구현한 것:

- 송금 요청
- 송금 조회
- 송금 목록 조회
- idempotency key와 request hash
- sender wallet Redis lock
- sender withdraw, receiver deposit
- `COMPENSATION_REQUIRED`
- compensation lookup/refund API
- outbox event 저장

중요한 결정:

- 출금 전 실패와 출금 후 실패를 다르게 처리한다.
- 출금 후 입금 실패는 `FAILED`가 아니라 `COMPENSATION_REQUIRED`다.

블로그 포인트:

- 분산 환경에서 실패는 "없애는 것"보다 "복구 가능한 상태로 남기는 것"이 중요하다.

## 6. reward-service

구현한 것:

- 부모-자녀 연결
- 부모의 미션 생성
- 자녀의 미션 제출
- 부모 승인/반려
- 보상 지급
- 캐시북 요약/내역

중요한 결정:

- reward-service는 지갑 잔액을 직접 바꾸지 않는다.
- 보상 지급은 transfer-service에 송금으로 위임한다.
- 보상 지급 멱등키는 `reward-payment-{missionId}`를 사용한다.

블로그 포인트:

- 도메인 기능이 돈을 움직여야 할 때도 잔액 책임을 가져오지 않는 것이 중요하다.

## 7. ledger-service

구현한 것:

- `transfer.completed` Kafka event 소비
- 원장 기록
- `transfer.failed` event 소비
- 실패 추적 저장
- 실패 조회 API
- transferId 기반 중복 소비 방어

중요한 결정:

- 송금 실행은 동기 HTTP로 끝내고, 원장 기록은 Kafka 이벤트로 분리한다.

블로그 포인트:

- 이벤트 기반 구조에서는 producer뿐 아니라 consumer 멱등성이 중요하다.

## 8. Kafka와 transactional outbox

구현한 것:

- Kafka container
- topic 설계
- transfer-service outbox table
- relay
- PROCESSING claim
- retry limit
- stale PROCESSING recovery
- outbox summary API

중요한 결정:

- transfer 상태 변경과 Kafka 발행 의도를 같은 DB transaction에 저장한다.

블로그 포인트:

- Kafka 발행 실패는 네트워크 문제가 아니라 데이터 정합성 문제로 이어질 수 있다.

## 9. api-gateway와 보안

구현한 것:

- route 설정
- JWT 검증 필터
- 인증 제외 경로
- 외부 `X-User-*`, `X-Internal-*` 제거
- 내부 서비스로 검증된 사용자 헤더 주입

중요한 결정:

- 하위 서비스는 클라이언트가 보낸 사용자 헤더를 신뢰하지 않는다.

블로그 포인트:

- Gateway는 단순 reverse proxy가 아니라 인증 신뢰 경계다.

## 10. sample-react와 화면 연결

구현한 것:

- 로그인/회원가입
- 부모 홈
- 자녀 홈
- 가족 연결
- 미션 생성/제출/승인/반려
- 충전/계좌 등록
- 출금 화면
- API adapter 정렬
- Playwright 검증

중요한 결정:

- 화면은 백엔드 상태 모델을 숨기지 않고 사용자 흐름으로 보여준다.

블로그 포인트:

- 프론트엔드 흐름을 맞추면서 API shape의 불일치를 발견하고 정리했다.

## 11. 테스트와 검증

검증한 것:

- backend service tests
- `docker compose config --quiet`
- `sample-react npm run check`
- frontend dummy/e2e/API mode Playwright
- API smoke script 준비

미검증 또는 남은 것:

- Docker Desktop 미실행으로 전체 runtime smoke test는 별도 확인 필요
- Testcontainers 기반 Kafka/Redis/MySQL 통합 테스트 추가 필요
- k6 실제 결과 수치 입력 필요

블로그 포인트:

- 검증한 것과 검증하지 못한 것을 분리해서 기록하는 것이 신뢰를 만든다.
