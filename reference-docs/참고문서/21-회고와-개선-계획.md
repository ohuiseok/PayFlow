# 회고와 개선 계획

> 도메인 전환 안내: 현재 PayFlow는 **청년 정책 참여 미션 및 지원금 지급 플랫폼**으로 설명한다. 내부 구현 호환성을 위해 `PARENT`/`CHILD`, `/api/families`, `/api/missions`, `/api/cashbook`, `reward-service` 같은 명칭은 유지하지만, 문서와 발표에서는 각각 **기관 담당자**, **청년 참여자**, **참여자 연결**, **정책 미션**, **지원금 사용 내역**, **정책 미션/지원금 서비스**로 해석한다.

이 문서는 PayFlow를 블로그 시리즈 마지막 글로 정리할 때 사용할 회고입니다.

## 잘한 점

### 1. 돈의 책임을 wallet-service로 모았다

여러 서비스가 잔액을 직접 수정하지 않게 만든 점이 가장 중요했습니다. banking, transfer, reward는 모두 돈을 움직이는 흐름을 갖지만, 실제 balance 변경은 wallet-service가 담당합니다.

### 2. 멱등성을 여러 계층으로 나눴다

API request 멱등성과 wallet reflection 멱등성을 분리했습니다.

```text
API: Idempotency-Key + requestHash
Wallet: referenceType + referenceId
Kafka consumer: transferId/source key
```

이 구분 덕분에 클라이언트 재시도, 내부 재호출, 이벤트 중복 소비를 각각 다룰 수 있습니다.

### 3. 실패를 상태로 표현했다

송금 실패를 단순 exception으로 끝내지 않고 `FAILED`, `COMPENSATION_REQUIRED`, `COMPENSATED` 같은 상태로 남겼습니다.

### 4. outbox로 이벤트 발행 유실을 줄였다

송금 상태 변경과 이벤트 발행 의도를 같은 DB transaction에 저장했습니다. Kafka 장애가 있어도 outbox row가 남아 재시도할 수 있습니다.

### 5. 문서와 구현을 함께 정리했다

API, ERD, Architecture, 테스트 전략, 운영 런북, 의사결정 기록을 함께 남겼습니다. 블로그와 면접에서 구현 근거를 설명하기 쉬워졌습니다.

### 6. 실시간 처리와 정산을 분리했다

Toss 승인·취소는 banking-service에서 즉시 확정하고 outbox로 이벤트를 남깁니다. settlement-service는 이를 멱등 수집해 다음 날 Spring Batch로 원장과 대사하므로, 실시간 결제 응답과 사후 검증의 책임이 분리됐습니다.

## 아쉬운 점

### 1. migration 도구를 처음부터 강제하지 못했다

개발 편의를 위해 `ddl-auto=update`에 기대는 구간이 있었습니다. 운영 관점에서는 Flyway 또는 Liquibase로 schema 변경 이력을 관리해야 합니다.

### 2. 전체 runtime smoke test가 환경에 의존했다

Docker Desktop이 실행되지 않은 환경에서는 전체 compose runtime 검증이 제한되었습니다. 검증한 것과 검증하지 못한 것을 문서에 분리했지만, CI에서 자동화하면 더 좋습니다.

### 3. Testcontainers 통합 테스트가 부족하다

Kafka, Redis, MySQL이 함께 움직이는 흐름은 Testcontainers로 검증하는 것이 좋습니다.

### 4. 운영 지표가 더 필요하다

현재 outbox summary API는 있지만, 운영 수준에서는 metrics/alert가 필요합니다.

필요한 지표:

- outbox pending count
- oldest pending age
- publish failure count
- compensation required count
- compensation retry count
- wallet duplicate reference count
- Kafka consumer lag

### 5. DLQ 전략이 아직 약하다

outbox retry가 반복 실패할 때 별도의 DLQ나 운영 알림으로 넘기는 전략이 필요합니다.

ledger consumer에는 DLT가 적용됐지만 settlement consumer에는 아직 없습니다. 또한 banking settlement outbox는 원자적 claim과 적체 모니터링 API가 없어 다중 인스턴스 운영에 보강이 필요합니다.

## 다음 개선 계획

## 1단계. 검증 강화

- Full Docker Compose runtime smoke test 완료
- k6 실제 부하 테스트 실행
- Testcontainers로 MySQL/Redis/Kafka 통합 테스트 추가
- idempotency conflict contract test 보강
- 정산 자정 경계, 원장 누락/금액 불일치, 실제 Kafka 통합 테스트 보강

## 2단계. 운영 관찰성

- actuator metrics 추가
- outbox lag metric 추가
- compensation metric 추가
- Kafka consumer lag 확인
- alert rule 문서화
- settlement discrepancy와 payment settlement outbox lag metric 추가

## 3단계. 데이터 관리

- Flyway/Liquibase 도입
- migration naming rule 정리
- seed/test data 분리
- 운영용 `ddl-auto=validate` 전환
- settlement의 Hibernate update와 batch-schema 초기화를 Flyway로 단일화

## 4단계. 장애 복구 고도화

- compensation 자동 retry workflow
- retry exhausted 이벤트 분리
- DLQ topic 설계
- 관리자용 복구 화면 또는 API

## 5단계. 오픈뱅킹 고도화

- token refresh flow
- 외부 HTTP timeout/retry/circuit breaker
- sandbox end-to-end test
- API log masking rule 강화

## 6단계. 배포 전략

- staging/prod profile 분리
- secret manager 적용
- blue-green 또는 rolling deployment 문서화
- DB migration 배포 순서 정리

## 블로그 마지막 글 메시지

PayFlow는 완성된 운영 결제 시스템이라기보다, 결제 시스템에서 중요한 문제를 포트폴리오 범위로 압축해 구현한 프로젝트입니다. 핵심은 "돈이 두 번 움직이지 않게 하기", "실패를 복구 가능한 상태로 남기기", "서비스가 나뉘어도 정합성 책임을 명확히 하기"였습니다.

## 면접용 마무리

이 프로젝트를 통해 단순 기능 구현보다 상태 모델, 멱등성, 보상 트랜잭션, outbox, consumer idempotency가 결제 시스템에서 왜 중요한지 배웠습니다. 다음 단계에서는 Testcontainers, metrics/alert, DLQ, migration 전략을 보강해 운영에 더 가까운 구조로 발전시키고 싶습니다.
