# 멱등성과 데이터 정합성

> 도메인 전환 안내: 현재 PayFlow는 **청년 정책 참여 미션 및 지원금 지급 플랫폼**으로 설명한다. 내부 구현 호환성을 위해 `PARENT`/`CHILD`, `/api/families`, `/api/missions`, `/api/cashbook`, `reward-service` 같은 명칭은 유지하지만, 문서와 발표에서는 각각 **기관 담당자**, **청년 참여자**, **참여자 연결**, **정책 미션**, **지원금 사용 내역**, **정책 미션/지원금 서비스**로 해석한다.

PayFlow에서 멱등성은 "같은 API를 다시 호출해도 같은 응답을 받는다"만 뜻하지 않습니다. 더 중요한 목표는 같은 요청이 반복되어도 지갑 잔액이 한 번만 움직이게 하는 것입니다.

## 문제 상황

결제성 API에서는 아래 상황이 흔합니다.

- 사용자가 버튼을 여러 번 누름
- 클라이언트 timeout 후 재시도
- API Gateway 또는 네트워크 구간 재전송
- 외부 은행 API 응답 지연
- Kafka 이벤트 중복 전달

이때 서버가 단순히 요청을 한 번 더 처리하면 중복 충전, 중복 송금, 중복 원장 기록이 발생할 수 있습니다.

## 멱등성 계층

PayFlow는 멱등성을 여러 계층에서 나눠 처리합니다.

| 계층 | 키 | 방어 대상 |
| --- | --- | --- |
| API request | `Idempotency-Key` + `requestHash` | 같은 API 요청 반복 |
| Wallet reflection | `referenceType` + `referenceId` | 같은 원천 거래의 잔액 중복 반영 |
| Open Banking | `bank_tran_id` | 은행 거래 식별과 결과 조회 |
| Kafka consumer | `transferId` 또는 source key | 이벤트 중복 소비 |
| Mission reward | `reward-payment-{missionId}` | 같은 정책 미션 지원금 중복 지급 |

## API Request Idempotency

송금, 충전처럼 돈을 움직이는 API는 `Idempotency-Key`를 받습니다.

```http
Idempotency-Key: transfer-parent-child-20260620-001
```

요청 body는 hash로 저장합니다.

```text
same idempotency key + same request hash
-> 기존 결과 반환

same idempotency key + different request hash
-> 409 Conflict
```

이 규칙이 필요한 이유는 같은 key로 금액만 바꿔 재요청하는 위험을 막기 위해서입니다.

## Wallet Reflection Idempotency

API 요청이 멱등해도 wallet-service 입장에서는 다른 서비스가 같은 원천 거래를 다시 반영하려 할 수 있습니다.

그래서 wallet-service는 잔액 변경 요청에 원천 거래 식별자를 받습니다.

```json
{
  "amount": 10000,
  "referenceType": "TRANSFER_CREDIT",
  "referenceId": "5000"
}
```

같은 `referenceType` + `referenceId`는 한 번만 반영됩니다.

예시:

| 원천 | referenceType | referenceId |
| --- | --- | --- |
| 은행 충전 | `OPEN_BANKING_CHARGE` | `bank_tran_id` |
| 송금 출금 | `TRANSFER_DEBIT` | `transferId` |
| 송금 입금 | `TRANSFER_CREDIT` | `transferId` |
| 정책 미션 지원금 | `REWARD_PAYMENT` | `missionId` 또는 transferId |
| 보상 환불 | `TRANSFER_COMPENSATION` | `transferId` |

## 데이터 정합성 원칙

### 1. 잔액의 단일 진실 공급원

지갑 잔액은 wallet-service만 직접 변경합니다.

다른 서비스는 아래처럼 요청만 보냅니다.

```text
banking-service -> wallet-service deposit/withdraw
transfer-service -> wallet-service withdraw/deposit
reward-service -> transfer-service transfer request
```

### 2. 잔액과 거래 이력은 같은 트랜잭션

wallet balance 변경과 wallet transaction 저장은 분리되면 안 됩니다.

```text
wallet balance update
wallet_transactions insert
commit
```

둘 중 하나만 성공하면 잔액 추적이 불가능해집니다.

### 3. 서비스 간 DB join 금지

MSA 구조에서는 다른 서비스 DB를 직접 join하지 않습니다.

대신:

```text
ID 참조
HTTP API
Kafka event
```

를 사용합니다.

### 4. 이벤트는 중복될 수 있다고 가정

Kafka 이벤트는 at-least-once로 처리될 수 있습니다. 따라서 ledger-service는 같은 transferId 이벤트를 여러 번 받아도 원장을 한 번만 저장해야 합니다.

## 송금 정합성 흐름

```text
POST /api/transfers
-> Idempotency-Key 확인
-> requestHash 확인
-> sender wallet lock
-> sender wallet withdraw(referenceType=TRANSFER_DEBIT, referenceId=transferId)
-> receiver wallet deposit(referenceType=TRANSFER_CREDIT, referenceId=transferId)
-> transfer SUCCEEDED
-> outbox event 저장
```

중복 요청이 들어오면:

```text
same key + same body
-> 기존 transfer 반환

same key + different body
-> conflict
```

wallet-service가 같은 reference를 다시 받으면:

```text
이미 처리된 wallet transaction 반환 또는 중복 반영 방지
```

## 보상 지급 정합성 흐름

정책 미션 지원금 지급은 reward-service가 직접 잔액을 바꾸지 않습니다.

```text
reward-service
-> transfer-service POST /transfers
   Idempotency-Key = reward-payment-{missionId}
-> transfer-service가 지갑 송금 처리
-> reward_tasks.paid_transfer_id 저장
```

이 구조의 장점:

- 미션 지급도 일반 송금과 같은 멱등성/보상 정책을 사용
- reward-service에 잔액 변경 책임이 생기지 않음
- 같은 missionId로 보상 지급을 반복해도 중복 지급 방어 가능

## 정합성 체크리스트

- 같은 API 요청 재시도 시 잔액이 한 번만 변경되는가?
- 같은 idempotency key에 다른 body가 들어오면 거절되는가?
- wallet transaction에 원천 거래 reference가 남는가?
- 송금 실패 상태와 실패 사유가 저장되는가?
- outbox event가 transfer 상태 변경과 함께 저장되는가?
- ledger는 같은 이벤트를 두 번 받아도 한 번만 기록하는가?

## 면접용 요약

PayFlow는 API 멱등성과 잔액 반영 멱등성을 분리했습니다. API 단계에서는 `Idempotency-Key`와 `requestHash`로 중복 요청을 방어하고, wallet-service에서는 `referenceType`과 `referenceId`로 같은 원천 거래가 잔액에 두 번 반영되지 않도록 했습니다. Kafka 이벤트도 중복될 수 있다고 보고 ledger-service에서 consumer idempotency를 적용했습니다.
