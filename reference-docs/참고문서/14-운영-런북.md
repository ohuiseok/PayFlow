# 운영 런북

> 도메인 전환 안내: 현재 PayFlow는 **청년 정책 참여 미션 및 지원금 지급 플랫폼**으로 설명한다. 내부 구현 호환성을 위해 `PARENT`/`CHILD`, `/api/families`, `/api/missions`, `/api/cashbook`, `reward-service` 같은 명칭은 유지하지만, 문서와 발표에서는 각각 **기관 담당자**, **청년 참여자**, **참여자 연결**, **정책 미션**, **지원금 사용 내역**, **정책 미션/지원금 서비스**로 해석한다.

이 문서는 PayFlow를 로컬 또는 포트폴리오 시연 환경에서 실행하고 문제를 확인할 때 사용하는 절차입니다.

## 1. 기본 실행

### 환경 변수 준비

`.env.example`을 기준으로 `.env`를 준비합니다.

```powershell
Copy-Item .env.example .env
```

```powershell
docker compose up --build
```


## 6. Outbox 적체 확인

API:

```http
GET /api/transfers/outbox/summary
```

확인할 값:

```text
PENDING count
FAILED count
retryableFailureCount
retryExhaustedCount
oldestPendingEventAgeSeconds
```

판단:

| 증상 | 가능 원인 |
| --- | --- |
| PENDING 증가 | relay 미동작 또는 처리 지연 |
| FAILED 증가 | Kafka 연결 실패, serialization 실패 |
| PROCESSING 오래 유지 | relay 중단, publish 중 장애 |
| retry exhausted | 반복 발행 실패 |

## 7. 보상 필요 송금 확인

목록:

```http
GET /api/transfers/compensations
```

단건:

```http
GET /api/transfers/compensations/{transferId}
```

환불 실행:

```http
POST /api/transfers/compensations/{transferId}/refund
```

확인할 값:

```text
status = COMPENSATED
compensatedAt
compensationRetryCount
compensationFailureReason
wallet transaction referenceType=TRANSFER_COMPENSATION
```

## 8. Open Banking 모호 상태 확인

결과 조회:

```http
POST /api/bank/transfers/{bankingTransferId}/result-check
```

보상:

```http
POST /api/bank/transfers/{bankingTransferId}/compensate
```

상태 판단:

| 상태 | 의미 |
| --- | --- |
| `BANK_PROCESSING` | 은행 처리 중, 결과 재조회 필요 |
| `UNKNOWN` | timeout 등으로 최종 결과 모호 |
| `COMPENSATION_REQUIRED` | 지갑 복구 필요 |
| `COMPENSATED` | 보상 완료 |

## 9. Ledger 확인

송금 실패 추적:

```http
GET /api/ledgers/transfer-failures
GET /api/ledgers/transfer-failures/{transferId}
```

확인할 것:

- transferId가 중복 저장되지 않았는가?
- failureReason이 남았는가?
- Kafka consumer log에 에러가 없는가?

## 10. 자주 보는 문제

### 정산 실행과 대사 확인

```http
POST /api/settlements/daily/{yyyy-MM-dd}
GET  /api/settlements/daily/{yyyy-MM-dd}
```

확인할 값:

```text
status
transactionCount
discrepancyCount
grossAmount / cancelAmount / feeAmount / expectedNetAmount
```

`WITH_DISCREPANCY`면 `settlement_items`의 `MISSING_LEDGER`, `AMOUNT_MISMATCH`와 사유를 확인합니다. `payment_settlement_outbox`의 `PENDING/FAILED`, retry count도 함께 확인해야 합니다. 현재 전용 모니터링 API는 없어 DB와 로그 확인이 필요합니다.

### API Gateway 401

확인:

- access token 만료
- Authorization header 형식
- JWT secret 불일치
- 인증 제외 경로 설정

### 하위 서비스 403 또는 내부 secret 오류

확인:

- `X-Gateway-Secret`
- 서비스 환경 변수
- Gateway route filter

### wallet-service 잔액 불일치 의심

확인:

- wallet balance
- wallet_transactions 합계
- reference 중복 여부
- 송금/충전 status

### Kafka 이벤트 누락 의심

확인:

- outbox summary
- transfer-service relay log
- Kafka container status
- ledger-service consumer log

## 11. 정리 명령

컨테이너 중지:

```powershell
docker compose down
```

볼륨까지 삭제:

```powershell
docker compose down -v
```

주의:

볼륨 삭제는 MySQL/Kafka 데이터를 모두 지웁니다. 시연 데이터가 필요한 경우 사용하지 않습니다.
