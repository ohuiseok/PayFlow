# 실패 복구와 보상 트랜잭션

PayFlow는 실패를 단순히 exception으로 끝내지 않고, 상태와 복구 경로로 모델링합니다. 결제성 시스템에서는 "실패하지 않게 만들기"보다 "실패했을 때 추적하고 복구할 수 있게 만들기"가 중요합니다.

## 실패 분류

| 실패 위치 | 예시 | 처리 |
| --- | --- | --- |
| 요청 검증 전 | 잘못된 금액, 권한 없음 | 즉시 실패 응답 |
| 출금 전 | sender wallet 조회 실패 | `FAILED` |
| 출금 후 입금 전 | receiver deposit 실패 | `COMPENSATION_REQUIRED` |
| 이벤트 발행 | Kafka publish 실패 | outbox `FAILED`, retry |
| 이벤트 소비 | ledger 저장 실패 | consumer retry, dedup 필요 |
| 외부 은행 API | timeout, processing, no permission | `UNKNOWN`, `BANK_PROCESSING`, `COMPENSATION_REQUIRED` |

## Transfer State Model

```text
REQUESTED
-> PROCESSING
-> SUCCEEDED

REQUESTED
-> PROCESSING
-> FAILED

REQUESTED
-> PROCESSING
-> COMPENSATION_REQUIRED
-> COMPENSATED
```

## 출금 전 실패

출금 전에는 아직 돈이 움직이지 않았기 때문에 `FAILED`로 종료할 수 있습니다.

```text
transfer REQUESTED
-> sender wallet lookup 실패
-> FAILED
```

저장할 정보:

- transfer status
- failure reason
- created/updated time

## 출금 후 입금 실패

가장 위험한 구간입니다.

```text
sender withdraw 성공
receiver deposit 실패
```

이 경우 단순히 `FAILED`로 끝내면 이미 sender 지갑에서 돈이 빠져나간 상태가 됩니다.

PayFlow는 이 상태를 격리합니다.

```text
PROCESSING
-> COMPENSATION_REQUIRED
```

이후 refund API를 통해 sender wallet에 환불합니다.

```text
POST /api/transfers/compensations/{transferId}/refund
-> wallet-service deposit
   referenceType = TRANSFER_COMPENSATION
   referenceId = {transferId}
-> transfer COMPENSATED
```

## 보상 실패

보상 요청도 실패할 수 있습니다. 이때는 상태를 유지하고 retry metadata를 남깁니다.

```text
COMPENSATION_REQUIRED
-> refund attempt
-> failed
-> COMPENSATION_REQUIRED 유지
-> compensationRetryCount 증가
-> compensationFailureReason 저장
```

이렇게 하면 운영자가 같은 API를 나중에 다시 호출할 수 있습니다.

## Banking Failure

Open Banking 연동에서는 외부 API 응답이 모호할 수 있습니다.

### 충전

```text
POST /api/bank/deposits
-> Open Banking withdraw transfer
-> bank success confirmed
-> wallet deposit
-> COMPLETED
```

은행 성공이 확인되기 전에는 지갑에 돈을 넣지 않습니다.

### 결과 모호

```text
timeout
processing
unknown response
```

이 경우:

```text
BANK_PROCESSING 또는 UNKNOWN
-> result-check 대상
```

### 권한 없는 API

Open Banking reference에서 권한 없는 API는 business success로 연결하지 않습니다.

```text
no permission deposit transfer
-> 지갑 출금 성공분 격리
-> COMPENSATION_REQUIRED
```

## Outbox Failure Recovery

송금 성공 후 이벤트 발행 실패는 원장 누락으로 이어질 수 있습니다.

PayFlow는 outbox를 사용합니다.

```text
transfer SUCCEEDED
outbox PENDING 저장
commit
```

relay 흐름:

```text
PENDING
-> PROCESSING
-> PUBLISHED
```

발행 실패:

```text
PROCESSING
-> FAILED
-> retry 가능
```

오래 stuck 된 processing:

```text
PROCESSING timeout
-> FAILED로 복구
-> retry 가능
```

## 복구 시나리오

### Scenario 1. 송금 API timeout 후 클라이언트 재시도

```text
client timeout
-> same Idempotency-Key retry
-> existing transfer result 반환
```

### Scenario 2. 출금 후 입금 실패

```text
transfer COMPENSATION_REQUIRED
-> 운영자/관리 API로 refund
-> wallet deposit with TRANSFER_COMPENSATION
-> transfer COMPENSATED
```

### Scenario 3. Kafka 발행 실패

```text
outbox FAILED
-> relay retry
-> Kafka publish success
-> PUBLISHED
```

### Scenario 4. ledger consumer 중복 수신

```text
same transfer.completed twice
-> ledger-service dedup
-> ledger entry one time only
```

## 운영 확인 포인트

| 확인 대상 | 방법 |
| --- | --- |
| 보상 필요 송금 | `GET /api/transfers/compensations` |
| 단건 보상 상태 | `GET /api/transfers/compensations/{transferId}` |
| 보상 실행 | `POST /api/transfers/compensations/{transferId}/refund` |
| outbox 적체 | `GET /api/transfers/outbox/summary` |
| 송금 실패 이벤트 | `GET /api/ledgers/transfer-failures` |

## 면접용 요약

PayFlow는 출금 전 실패와 출금 후 실패를 다르게 처리합니다. 출금 전에는 `FAILED`로 종료하지만, 출금 이후 입금 실패는 이미 돈이 움직였기 때문에 `COMPENSATION_REQUIRED`로 격리하고 별도 refund API로 복구합니다. Kafka 발행 실패는 transactional outbox로 복구 가능하게 만들고, Open Banking의 모호한 응답은 result-check로 최종화합니다.
