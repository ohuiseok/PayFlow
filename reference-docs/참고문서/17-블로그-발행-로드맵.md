# 블로그 발행 로드맵

이 문서는 PayFlow 개발 과정을 블로그 시리즈로 공개할 때의 권장 발행 순서입니다. 독자가 "왜 만들었는지 -> 어떻게 설계했는지 -> 어떤 순서로 구현했는지 -> 어떻게 검증했는지"를 자연스럽게 따라오도록 구성했습니다.

## 전체 구성

| Part | 주제 | 목표 |
| ---: | --- | --- |
| 1 | 프로젝트 시작 | 문제 정의와 MVP 범위 |
| 2 | 사용자 흐름 | 부모/자녀 화면 흐름 |
| 3 | 아키텍처 | MSA와 서비스 경계 |
| 4 | 데이터 모델 | 서비스별 DB와 ERD |
| 5 | API 계약 | Gateway route와 API 설계 |
| 6 | user/wallet | 인증과 지갑 잔액 |
| 7 | 멱등성 | 중복 요청 방어 |
| 8 | banking | 계좌/충전/Open Banking |
| 9 | transfer | 송금 상태 머신 |
| 10 | compensation | 실패 복구와 보상 |
| 11 | reward | 미션 보상 지급 |
| 12 | outbox/kafka | 이벤트 발행과 원장 |
| 13 | security | 보안과 민감정보 |
| 14 | frontend | 화면과 API 연결 |
| 15 | testing | 테스트 전략 |
| 16 | load test | 부하 테스트 |
| 17 | troubleshooting | 문제 해결 기록 |
| 18 | retrospective | 회고와 개선 계획 |

## 발행 팁

- 각 글 첫머리에는 "이번 글에서 해결할 질문"을 3개 정도 둡니다.
- 글 끝에는 "다음 글에서 이어질 문제"를 남깁니다.
- 코드 전체를 붙이기보다 핵심 클래스, 핵심 상태값, 핵심 제약 조건을 중심으로 설명합니다.
- 검증하지 못한 것은 숨기지 말고 "미검증 리스크"로 분리합니다.
