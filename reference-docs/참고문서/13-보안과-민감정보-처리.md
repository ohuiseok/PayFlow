# 보안과 민감정보 처리

> 도메인 전환 안내: 현재 PayFlow는 **청년 정책 참여 미션 및 지원금 지급 플랫폼**으로 설명한다. 내부 구현 호환성을 위해 `PARENT`/`CHILD`, `/api/families`, `/api/missions`, `/api/cashbook`, `reward-service` 같은 명칭은 유지하지만, 문서와 발표에서는 각각 **기관 담당자**, **청년 참여자**, **참여자 연결**, **정책 미션**, **지원금 사용 내역**, **정책 미션/지원금 서비스**로 해석한다.

PayFlow는 포트폴리오 MVP이지만 결제/금융 도메인을 다루기 때문에 인증, 내부 헤더, 계좌 정보, Open Banking token을 별도로 고려했습니다.

## 인증 구조

```text
Client
-> API Gateway
   -> JWT 검증
   -> 사용자 정보 추출
   -> 내부 헤더 주입
-> Target Service
```

외부 사용자는 직접 내부 서비스에 접근하지 않는 것을 기본 전제로 합니다.

## Gateway 책임

API Gateway는 다음 일을 담당합니다.

- 인증 제외 경로 관리
- JWT 검증
- 만료/위조 token 거절
- 외부 요청의 `X-User-*`, `X-Internal-*` 헤더 제거
- 검증된 사용자 정보만 내부 헤더로 주입
- 서비스별 라우팅

## Header Trust Boundary

하위 서비스는 클라이언트가 보낸 userId를 신뢰하지 않습니다.

신뢰하는 값:

```text
X-User-Id
X-User-Role
```

단, 이 값은 Gateway가 검증 후 주입한 경우에만 신뢰합니다.

위험한 패턴:

```text
Client가 직접 X-User-Id: 1 전송
```

방어:

```text
Gateway에서 외부 X-User-* 제거
검증된 token claim으로 다시 생성
```

## Internal Secret

내부 서비스 호출은 `X-Gateway-Secret` 또는 `X-Internal-Secret` 같은 shared secret으로 보호할 수 있습니다.

목적:

- 외부에서 내부 API를 직접 호출하는 것 방지
- Gateway 또는 허용된 내부 서비스만 접근 가능하게 제한

운영 환경에서는 secret을 코드나 Git에 저장하지 않고 환경 변수 또는 secret manager로 주입해야 합니다.

## Password Handling

사용자 password는 plain text로 저장하지 않습니다.

기본 원칙:

- password hash 저장
- login 시 hash 비교
- 응답 DTO에 password/hash 포함 금지

## Bank Account Handling

계좌번호는 민감 정보입니다.

PayFlow의 설계 방향:

- 화면/응답에는 masked account number만 노출
- 중복 검출이 필요하면 hash 저장
- raw account number는 저장하지 않는 방향

예시:

```text
accountNumber: 1234567890
accountNumberMasked: 123-****-7890
accountNumberHash: SHA-256(...)
```

## Open Banking Token Handling

Open Banking 사용자 token은 계좌 접근 권한을 의미하므로 암호화 저장이 필요합니다.

저장 대상:

```text
accessTokenEncrypted
refreshTokenEncrypted
scope
expiresAt
userSeqNo
```

저장하지 않거나 로그에 남기지 않을 대상:

```text
raw access token
raw refresh token
client secret
authorization code
raw account number
```

## API Log Policy

금융 API 요청/응답 전체 payload를 그대로 저장하면 민감 정보가 로그에 남을 수 있습니다.

PayFlow에서는 운영 추적에 필요한 metadata 중심으로 남기는 방향을 사용합니다.

저장 가능:

```text
apiName
requestId
bankingTransferId
apiResponseCode
bankResponseCode
request/response key list
error message
```

저장 지양:

```text
raw request body
raw response body
token
account number
client secret
```

## Authorization Rules

서비스별 권한 예시:

| API | 권한 |
| --- | --- |
| 기관 정책 미션 생성 | `PARENT` |
| 청년 정책 미션 제출 | 연결된 `CHILD` |
| 기관 정책 미션 승인/반려 | 미션 소유 `PARENT` |
| 지원금 지급 | 승인된 정책 미션의 기관 담당자 |
| 지갑 조회 | 본인 또는 내부 서비스 |
| 송금 요청 | sender는 현재 사용자 |

## 보안 체크리스트

- JWT 만료 시간을 설정했는가?
- Gateway가 외부 사용자 헤더를 제거하는가?
- 하위 서비스가 Gateway secret을 검증하는가?
- password/hash가 응답에 포함되지 않는가?
- 계좌번호가 raw 형태로 저장되지 않는가?
- Open Banking token이 암호화 저장되는가?
- 로그에 token, 계좌번호, secret이 남지 않는가?
- `.env`가 Git에 올라가지 않는가?

## 운영 환경 개선 과제

- refresh token rotation
- key rotation
- secret manager 도입
- TLS termination과 내부 통신 정책
- rate limiting
- audit log와 access log 분리
- 개인정보 파기 정책
- 관리자 API 권한 분리

## 면접용 요약

PayFlow는 Gateway를 인증 신뢰 경계로 두고, 외부에서 들어온 사용자 식별 헤더를 제거한 뒤 JWT claim 기반으로 내부 헤더를 다시 주입합니다. 계좌번호는 masking/hash 중심으로 다루고, Open Banking token은 암호화 저장하는 방향으로 설계했습니다. 또한 raw 금융 API payload를 로그에 저장하지 않고 운영 추적용 metadata만 남기는 정책을 잡았습니다.
