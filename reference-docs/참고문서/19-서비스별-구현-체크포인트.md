# 서비스별 구현 체크포인트

> 도메인 전환 안내: 현재 PayFlow는 **청년 정책 참여 미션 및 지원금 지급 플랫폼**으로 설명한다. 내부 구현 호환성을 위해 `PARENT`/`CHILD`, `/api/families`, `/api/missions`, `/api/cashbook`, `reward-service` 같은 명칭은 유지하지만, 문서와 발표에서는 각각 **기관 담당자**, **청년 참여자**, **참여자 연결**, **정책 미션**, **지원금 사용 내역**, **정책 미션/지원금 서비스**로 해석한다.

이 문서는 블로그 작성 시 각 서비스에서 어떤 코드를 보면 되는지 정리한 가이드입니다.

## api-gateway

읽을 코드:

```text
api-gateway/src/main/java/com/payflow/gateway/filter/JwtAuthenticationFilter.java
api-gateway/src/main/java/com/payflow/gateway/config/SecurityConfig.java
api-gateway/src/main/resources/application.yml
```

설명할 내용:

- JWT 검증
- 인증 제외 경로
- 외부 사용자 헤더 제거
- 내부 사용자 헤더 주입
- 서비스 route

검증 포인트:

- token 없는 요청 거절
- 외부 `X-User-*` 조작 방어
- 정상 token 요청 라우팅

## banking-service

### 오픈뱅킹 (BankingController / BankingService)

읽을 코드:

```text
banking-service/src/main/java/com/payflow/banking/controller/BankingController.java
banking-service/src/main/java/com/payflow/banking/service/BankingService.java
banking-service/src/main/java/com/payflow/banking/openbanking/RealOpenBankingClient.java
banking-service/src/main/java/com/payflow/banking/openbanking/MockOpenBankingClient.java
banking-service/src/main/java/com/payflow/banking/service/TokenCryptoService.java
banking-service/src/main/java/com/payflow/banking/service/OpenBankingResultCheckScheduler.java
banking-service/src/main/java/com/payflow/banking/entity/BankingTransfer.java
banking-service/src/main/java/com/payflow/banking/entity/OpenBankingAuthorization.java
banking-service/src/main/java/com/payflow/banking/entity/OpenBankingToken.java
banking-service/src/main/java/com/payflow/banking/entity/BankingApiLog.java
```

설명할 내용:

- 오픈뱅킹 OAuth 인가 URL 발급 및 callback 처리
- 토큰 암호화 저장 (`TokenCryptoService`)
- 계좌 동기화 (`fintech_use_num`, `user_seq_no` 저장)
- 충전/출금 상태 모델 (`BANK_PROCESSING` → 지수 백오프 재조회)
- `OpenBankingResultCheckScheduler`: 결과 불명 거래 자동 확정
- 보상 (`COMPENSATION_REQUIRED` → `/transfers/{id}/compensate`)

검증 포인트:

- 같은 idempotency key 처리
- 모호한 외부 응답을 실패로 단정하지 않음
- 은행 성공 확정 이후에만 지갑 반영 수행
- 오픈뱅킹 API 호출 내역이 `banking_api_logs`에 기록됨

### Toss PG (TossPaymentController / TossPaymentService)

읽을 코드:

```text
banking-service/src/main/java/com/payflow/banking/controller/TossPaymentController.java
banking-service/src/main/java/com/payflow/banking/service/TossPaymentService.java
banking-service/src/main/java/com/payflow/banking/toss/RealTossPaymentsClient.java
banking-service/src/main/java/com/payflow/banking/toss/MockTossPaymentsClient.java
banking-service/src/main/java/com/payflow/banking/entity/PaymentCharge.java
banking-service/src/main/java/com/payflow/banking/entity/TossPaymentOrder.java
banking-service/src/main/java/com/payflow/banking/entity/TossPaymentEvent.java
banking-service/src/main/java/com/payflow/banking/client/LedgerClient.java
```

설명할 내용:

- Toss 위젯 충전 흐름 (`createCharge` → `confirm`)
- 결제 승인 후 지갑 입금 → 원장 기록까지 단계별 상태 관리
- `COMPENSATION_REQUIRED`: 승인 후 지갑 입금 실패 → 재입금 API
- `ledger_recorded=false`: 지갑 입금 후 원장 기록 실패 → 원장 재기록 API
- 웹훅 수신과 `event_idempotency_key` 중복 방지
- 취소 흐름 (Toss 취소 → 지갑 차감 → 원장 기록)
- 운영 모니터링 (`/operations/summary`, `/operations/compensations`, `/operations/ledger-compensations`)

검증 포인트:

- 같은 `Idempotency-Key` 재요청 처리
- Toss 승인 성공 후 지갑 입금 실패 시 `COMPENSATION_REQUIRED` 전환
- 지갑 입금 성공 후 원장 기록 실패 시 `ledger_recorded=false`
- 웹훅 동일 이벤트 중복 수신 시 중복 처리 없음
- `ledger-service` 원장 기록 후 `payment_charges.ledger_recorded=true`

## 블로그 작성 팁

각 서비스 글은 아래 구조를 반복하면 읽기 좋습니다.

```text
1. 이 서비스가 맡은 책임
2. 이 서비스가 소유한 데이터
3. 핵심 API
4. 가장 중요한 도메인 규칙
5. 실패 처리
6. 테스트한 것
7. 다음 서비스와 이어지는 지점
```
