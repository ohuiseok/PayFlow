# 16. Agent Checklist

구현 전에 이 체크리스트로 범위가 커지지 않았는지 확인한다.

## Scope

- [ ] 현재 API에서 쓰지 않는 테이블을 추가하지 않는다.
- [ ] 서비스 간 호출은 동기 HTTP로 구현한다.
- [ ] 충전, 송금, 보상 지급은 멱등하게 구현한다.
- [ ] 가족 관계는 `parent_child_links`로 표현한다.
- [ ] 미션 제출과 승인은 `reward_tasks` 상태 전이로 표현한다.
- [ ] 캐시북은 지갑 거래와 지급 완료 미션을 기반으로 조회한다.

## Database

- [ ] 각 서비스는 자기 DB만 직접 접근한다.
- [ ] 금액은 정수 원화 단위로 저장한다.
- [ ] 거래 테이블에는 unique 제약이 있다.
- [ ] 원장 전표는 source 기준으로 중복 저장되지 않는다.

## API

- [ ] 외부 API는 api-gateway를 통과한다.
- [ ] 내부 API는 `/internal/**`로 분리한다.
- [ ] 공통 에러 응답 형식을 지킨다.
- [ ] traceId를 응답과 로그에 남긴다.

## Tests

- [ ] 회원 가입 후 지갑 생성 테스트가 있다.
- [ ] 충전 성공/실패 테스트가 있다.
- [ ] 송금 성공/실패/동시성 테스트가 있다.
- [ ] 보상 지급 재시도 테스트가 있다.
- [ ] 원장 차대 일치 테스트가 있다.
- [ ] E2E 시나리오가 README 흐름과 맞다.
