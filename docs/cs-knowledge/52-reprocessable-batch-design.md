# Reprocessable Batch Design

## 핵심 개념

재처리 가능한 배치 설계는 배치가 실패하거나 일부 결과가 잘못되었을 때 같은 기준으로 다시 실행해도 안전하게 처리되도록 만드는 것이다.

배치는 긴 시간 동안 많은 데이터를 처리하므로 실패 가능성을 기본 전제로 둬야 한다.

## PayFlow 연결

정산 배치가 중간에 실패하면 이미 처리된 거래와 아직 처리되지 않은 거래가 섞일 수 있다. 재실행 시 이미 정산된 거래가 다시 반영되면 중복 정산이 발생한다.

따라서 정산 대상 거래별 처리 상태, 정산 배치 ID, 유니크 제약조건이 필요하다.

## 실무 포인트

- 배치 실행 ID를 남긴다.
- 처리 대상과 처리 결과를 분리한다.
- 같은 거래가 같은 정산에 두 번 들어가지 않게 한다.
- 실패 지점부터 재시작할 수 있게 한다.
- 재실행 결과가 예측 가능해야 한다.

## 체크 질문

- 배치 재처리가 필요한 대표 상황은 무엇인가
- 중복 정산을 막으려면 어떤 제약이 필요한가
- 배치 실행 ID는 왜 유용한가

## 실무 설계 보강

### 대표 장애 시나리오

정산 재실행 시 이미 정산된 거래가 다시 포함된다.

### 잘못된 구현 예시

~~~text
실패하면 처음부터 다시 돌리면 된다고 생각한다.
~~~

### 좋은 구현 예시

~~~text
배치 실행 ID, 처리 대상 스냅샷, 결과 unique constraint, skip 정책을 둔다.
~~~

### 대안과 선택 이유

현재 잔액이나 최종 집계값만 저장하는 단순한 방식도 있지만, 결제 시스템에서는 나중에 왜 그런 결과가 나왔는지 설명할 수 있어야 한다. PayFlow는 금액 정책, 원장, 이력, 정산 결과, 대사 리포트를 남기는 방식이 분쟁 대응과 복구에 더 강하다.

### PayFlow에서 확인할 위치

settlement batch execution table, settlement target/result table

### 면접에서 설명하기

재처리 가능한 배치는 같은 입력에 대해 같은 결과를 내야 한다. 이것도 멱등성의 한 형태다.

### 관련 문서

40, 48, 50

## 심화 보강

### 재처리 가능한 배치의 핵심 조건

재처리 가능한 배치는 단순히 "다시 실행할 수 있다"가 아니다. 같은 기준으로 다시 실행했을 때 중복 반영 없이 같은 결론에 도달해야 한다.

- 입력 고정: 어떤 데이터를 대상으로 처리했는지 재구성할 수 있어야 한다.
- 결과 멱등성: 같은 대상이 두 번 처리되어도 결과가 중복 생성되지 않아야 한다.
- 진행 상태 기록: 어디까지 성공했고 어디서 실패했는지 알아야 한다.
- 실패 격리: 일부 데이터 실패가 전체 배치의 상태를 애매하게 만들지 않아야 한다.
- 검증 가능성: 재처리 후 합계, 건수, 실패 사유를 비교할 수 있어야 한다.

### 테이블 설계 예시

```text
settlement_batch
- batch_id
- business_date
- status              // CREATED, RUNNING, COMPLETED, FAILED, CANCELED
- started_at
- finished_at
- requested_by

settlement_target
- batch_id
- transfer_id
- amount
- fee_amount
- status              // PENDING, SUCCESS, FAILED, SKIPPED
- failure_reason
- unique(batch_id, transfer_id)

settlement_result
- result_id
- batch_id
- merchant_id
- total_amount
- total_fee
- transfer_count
- unique(batch_id, merchant_id)
```

`settlement_target`을 먼저 확정하면 배치 실행 중 신규 거래가 들어와도 이번 배치의 입력 집합이 흔들리지 않는다. 이후 재실행은 같은 `batch_id`의 `FAILED` 또는 `PENDING` 대상만 다시 처리할 수 있다.

### 재실행 정책

- 전체 재실행: 대상 스냅샷은 유지하고 결과 테이블의 유니크 제약으로 중복 반영을 막는다.
- 실패 건만 재실행: 실패 사유가 일시 장애일 때 유용하다.
- 새 배치 생성: 기준일이나 수수료 정책이 바뀌어 입력 자체를 다시 산정해야 할 때 사용한다.
- 수동 보정: 자동 재처리가 위험한 데이터는 별도 승인 흐름을 둔다.

재실행 버튼 하나로 모든 것을 해결하려고 하면 위험하다. "같은 입력의 재처리"와 "새 기준의 재산정"은 운영 화면과 API에서 명확히 구분해야 한다.

### PayFlow 적용 예시

```text
1. 2026-05-30 정산 배치 생성
2. 대상 거래를 settlement_target에 스냅샷으로 저장
3. 거래별 정산 결과를 settlement_result에 반영
4. 일부 거래 실패 시 target 상태를 FAILED로 기록
5. 재실행 시 FAILED 대상만 다시 처리
6. batch_id + transfer_id unique 제약으로 중복 정산 차단
```

### 테스트 포인트

- 배치가 중간에 죽은 뒤 재실행해도 이미 성공한 거래가 중복 반영되지 않는지 확인한다.
- 같은 기준일에 신규 거래가 뒤늦게 들어와도 기존 배치 대상이 바뀌지 않는지 확인한다.
- 같은 `batch_id`, `transfer_id` 조합이 두 번 저장되지 않는지 확인한다.
- 실패 건만 재처리했을 때 합계와 건수가 기대값과 일치하는지 확인한다.

## 질문의 질문으로 더 깊게 보기

### 이 문서에서 잡아야 할 CS 감각

이 구간의 핵심은 결제 도메인의 불변식이다. 금액은 정확해야 하고, 잔액은 설명 가능해야 하며, 원장과 정산은 나중에 검증 가능해야 한다. 좋은 결제 시스템은 성공 처리보다 설명 가능한 기록을 더 중요하게 본다.

### 질문의 질문

아래 질문들은 답을 외우기 위한 질문이 아니다. 하나씩 답하면서 "왜 그런 설계가 필요한지"를 설명하는 연습을 하기 위한 질문이다.

- Reprocessable Batch Design 개념이 없다면 PayFlow에서 가장 먼저 어떤 장애가 생기는가?
- 이 개념은 정확성, 성능, 보안, 운영성 중 무엇을 가장 크게 개선하는가?
- 반대로 이 개념을 잘못 적용하면 어떤 복잡도가 추가되는가?
- 이 개념을 적용하면 어떤 문제가 해결되지만, 어떤 비용이나 복잡도가 새로 생기는가?
- 장애가 발생했을 때 이 개념은 문제를 예방하는가, 감지하는가, 복구하는가?
- PayFlow 코드에서 이 개념을 확인하려면 어떤 서비스, 테이블, 로그, 테스트를 봐야 하는가?

#### 핵심 답변

Reprocessable Batch Design 개념은 PayFlow에서 다음 이유로 중요하다.

- 결제 도메인 개념은 PayFlow에서 돈의 이동을 정확하고 설명 가능하며 나중에 검증 가능하게 만들기 위해 필요하다.
- 지갑 잔액, 잔액 변경 이력, 원장 엔트리, 정산 결과, 대사 리포트가 진실을 검증하는 기준이다.
- 금액 반올림 오류, 중복 정산, 원장 누락, 잔액과 원장 불일치가 발생하면 분쟁과 금전 손실로 이어진다.
- 정수 금액 모델, append-only 원장, 수수료 정책 버전, 재처리 가능한 배치, 대사로 방어한다.
- 운영에서는 잔액-원장 차이, 정산 검증 실패 수, 수수료 합계 차이, 중복 정산 차단 건수, 대사 실패 리포트를 본다.

#### 질문에 답하는 방식

좋은 답변은 용어 정의에서 멈추지 않고 다음 순서로 이어져야 한다.

1. 어떤 데이터나 요청을 보호하려는 개념인지 말한다.
2. 그 데이터의 진실의 원천이 어느 서비스에 있는지 말한다.
3. 장애, 중복, 동시성, 지연 중 어떤 상황에서 문제가 생기는지 설명한다.
4. 코드 레벨의 방어 수단과 운영 레벨의 확인 수단을 함께 말한다.

#### PayFlow 예시 답변

```text
Reprocessable Batch Design 개념은 PayFlow에서 정산 배치가 실패해도 같은 기준으로 다시 실행해 안전하게 복구하기 위해 필요하다.
이 개념이 없으면 이미 처리된 거래가 재실행에서 다시 포함되어 중복 정산이 발생할 수 있다.
그래서 코드에서는 배치 실행 ID, 거래별 처리 상태, 결과 유니크 제약, 실패 지점 재시작을 적용하고,
운영에서는 재처리 횟수, skip/duplicate 건수, 배치 재실행 후 합계를 확인해야 한다.
```

#### 더 생각해볼 점

이 답안은 하나의 예시다. 실제 설계에서는 성능, 복잡도, 장애 복구 난이도 사이의 trade-off를 함께 봐야 한다. 특히 PayFlow처럼 돈을 다루는 시스템에서는 빠른 성공보다 명확한 실패, 추적 가능한 상태, 재처리 가능한 구조가 더 중요하다.

### PayFlow에 대입해보기

1. 이 개념이 가장 직접적으로 연결되는 PayFlow 서비스 하나를 고른다.
2. 그 서비스가 소유한 데이터의 "진실의 원천"이 무엇인지 적어본다.
3. 동시에 두 요청이 들어오거나, 네트워크가 끊기거나, 프로세스가 죽는 상황을 가정한다.
4. 그 상황에서 데이터가 깨지지 않으려면 어떤 제약조건, 트랜잭션, 락, 이벤트, 재처리 장치가 필요한지 설명한다.
5. 마지막으로 운영자가 문제를 발견할 수 있는 로그나 지표가 무엇인지 적어본다.
