# OWASP API Security

## 핵심 개념

OWASP API Security는 API에서 자주 발생하는 보안 취약점을 정리한 기준이다. 대표적으로 객체 수준 권한 검증 실패, 인증 실패, 과도한 데이터 노출, 리소스 제한 부재, 잘못된 보안 설정 등이 있다.

API 기반 MSA에서는 UI보다 API가 실제 보안 경계가 된다. 따라서 모든 API는 입력, 권한, 응답, 호출 빈도를 신중하게 다뤄야 한다.

### OWASP API Security Top 10 상세 설명

**API1: Broken Object Level Authorization (BOLA)**

가장 흔하고 위험한 취약점이다. 객체에 접근할 때 그 객체가 요청자의 것인지 확인하지 않는다.

```text
취약한 예:
  GET /wallets/999  <- wallet 999가 다른 사람 것이어도 반환

안전한 예:
  GET /wallets/999
  서버: wallet.ownerId == requestUserId 확인 후 반환
```

PayFlow에서 BOLA는 지갑 조회, 송금 이력 조회, 지갑 ID를 받는 모든 API에서 발생할 수 있다.

**API2: Broken Authentication**

인증 메커니즘 자체의 결함이다.

```text
- 약한 JWT 서명 알고리즘 사용 (alg: none 허용)
- 만료된 토큰 수락
- JWT secret이 너무 짧거나 예측 가능
- 토큰 갱신 로직에 검증 미흡
```

특히 `alg: none` 공격은 JWT 라이브러리가 서명 알고리즘을 none으로 설정한 토큰을 수락하는 취약점이다. 서버는 허용 알고리즘 목록을 명시적으로 지정해야 한다.

**API3: Broken Object Property Level Authorization**

객체 접근 자체는 허용되지만, 특정 필드에 대한 접근 권한을 검증하지 않는다.

```text
취약한 예:
  PATCH /users/me
  요청 바디: {"role": "ADMIN"}  <- 일반 사용자가 역할을 바꿀 수 있음

안전한 예:
  PATCH /users/me
  서버: 허용된 필드(name, phone)만 업데이트 처리
  role 필드는 관리자 전용 API에서만 변경 가능
```

**API4: Unrestricted Resource Consumption**

Rate Limit이 없어 과도한 요청으로 서비스가 마비된다.

```text
공격 시나리오:
  - 무차별 대입 공격 (Brute Force): 로그인 시도 무한 반복
  - 대량 조회 공격: 거래 이력 전체를 반복 조회
  - 대량 송금 시도: 작은 금액으로 계속 송금해 수수료 소진

방어:
  - IP/사용자별 요청 제한 (Rate Limit)
  - 페이지 크기 제한
  - 계정별 일일 한도
```

**API5: Broken Function Level Authorization**

관리자 기능에 일반 사용자가 접근할 수 있다.

```text
취약한 예:
  DELETE /admin/users/{id}  <- 인증만 하고 관리자 역할 확인 안 함

안전한 예:
  엔드포인트 분리: /admin/** 는 ADMIN 역할만
  내부 API /internal/** 는 외부 접근 불가
```

**API6: Unrestricted Access to Sensitive Business Flows**

비즈니스 로직을 과도하게 자동화할 수 있는 경우다.

```text
결제 시스템 예:
  - 자동화 스크립트로 이벤트 쿠폰을 대량 발급
  - 반복 송금/취소로 수수료 이익 추출
  - 환율 차이를 이용한 자동 차익 거래

방어:
  - 캡차, 행동 분석
  - 요청 패턴 이상 감지
  - 동일 사용자의 반복 행동 제한
```

**API7: Server-Side Request Forgery (SSRF)**

서버가 클라이언트 제공 URL을 그대로 호출해 내부 시스템이 노출된다.

```text
취약한 예:
  POST /webhooks/test
  {"url": "http://internal-db:3306"}  <- 내부 DB 접근

안전한 예:
  허용 URL 화이트리스트 관리
  내부 IP 대역 차단 (169.254.x.x, 10.x.x.x 등)
```

**API8: Security Misconfiguration**

보안 설정 실수다.

```text
- CORS가 모든 오리진을 허용 (Access-Control-Allow-Origin: *)
- HTTP에서 민감 정보 전송
- 디버그 모드로 운영
- 기본 비밀번호 미변경
- 불필요한 HTTP 메서드 허용
- 보안 헤더 누락 (HSTS, X-Content-Type-Options 등)
```

**API9: Improper Inventory Management**

사용하지 않거나 구버전 API가 방치된다.

```text
- 테스트용 API 엔드포인트가 운영 환경에 남아 있음
- 구버전 /api/v1/** 가 보안 패치 없이 운영 중
- 문서화되지 않은 내부 API가 외부에 노출
```

**API10: Unsafe Consumption of APIs**

자신이 호출하는 외부 API의 응답을 그대로 신뢰하는 경우다.

```text
취약한 예:
  외부 결제 게이트웨이 응답의 금액을 검증 없이 사용
  -> 금액이 조작된 응답을 받으면 잘못된 처리 발생

안전한 예:
  외부 API 응답도 서버에서 재검증
  금액, 상태, 서명을 모두 확인
```

### 결제 시스템에서 가장 자주 발생하는 취약점

결제 시스템에서는 BOLA와 API4(Rate Limit 부재)가 가장 위험하다.

BOLA는 지갑이나 송금 내역에 대해 소유권 검증을 빠트릴 때 발생한다. 서비스 규모가 커질수록 새로운 API를 추가할 때 소유권 검증을 일관되게 적용하기 어려워진다. 이를 방지하려면 소유권 검증 로직을 공통 컴포넌트로 추출하거나, 도메인 서비스 내부에서 항상 소유권을 확인하는 구조를 만들어야 한다.

Rate Limit 부재는 악의적 사용자뿐만 아니라 실수로 과도한 재시도를 보내는 클라이언트로부터도 서비스를 보호한다. 결제 API에서는 사용자별, IP별, 엔드포인트별 Rate Limit을 조합해 설정한다.

### 흔한 오해와 함정

OWASP 목록을 외우면 보안이 된다는 생각은 위험하다. OWASP는 출발점이지 완전한 보안 체크리스트가 아니다. 실제 위협은 시스템마다 다르므로 자신의 시스템에 맞는 위협 모델링이 필요하다.

또한 보안 취약점은 새 기능을 추가할 때마다 다시 검토해야 한다. 기존 코드에 보안 검증이 있어도 새 API에서 빠질 수 있다. 코드 리뷰 체크리스트에 OWASP 항목을 포함시켜야 한다.

## PayFlow 연결

PayFlow에서 중요한 취약점은 BOLA, 즉 Broken Object Level Authorization이다. 예를 들어 사용자가 다른 사람의 `walletId`를 넣어 송금하면, 서버가 지갑 소유권을 확인하지 않는 한 권한 우회가 발생한다.

또한 과도한 응답 데이터 노출도 위험하다. 사용자 조회 API가 비밀번호 해시, 내부 식별자, 토큰 관련 정보를 반환하면 안 된다.

## 실무 포인트

- 모든 리소스 접근에 소유권 검증을 넣는다.
- 요청 바디의 사용자 ID를 그대로 믿지 않는다.
- 응답 DTO를 명확히 분리한다.
- Rate Limit으로 무차별 요청을 줄인다.
- 에러 메시지에 내부 구조를 노출하지 않는다.

## 체크 질문

- BOLA는 무엇이고 PayFlow에서는 어디서 발생할 수 있는가
- 요청자가 보낸 `userId`를 그대로 믿으면 왜 위험한가
- 결제 API에서 Rate Limit이 필요한 이유는 무엇인가

## 실무 설계 보강

### 대표 장애 시나리오

사용자가 URL의 walletId만 바꿔 다른 사용자의 지갑 거래 내역을 조회한다.

### 잘못된 구현 예시

~~~text
인증된 사용자라면 모든 리소스 조회가 가능하다고 가정한다.
~~~

### 좋은 구현 예시

~~~text
모든 객체 접근마다 사용자와 리소스의 소유 관계를 확인하고 응답 DTO를 최소화한다.
~~~

### 대안과 선택 이유

Gateway에서 한 번만 보안을 처리하는 방식도 있지만, 결제 시스템에서는 단일 방어선이 뚫렸을 때 피해가 크다. PayFlow는 Gateway 인증, 서비스 내부 인가, 입력 검증, 로그 마스킹, Secret 분리를 겹겹이 두는 방식이 더 안전하다.

### PayFlow에서 확인할 위치

gateway, wallet-service 조회 API, transfer-service 상태 조회, user-service 응답 DTO

### 면접에서 설명하기

OWASP API Security에서 결제 시스템의 핵심은 객체 수준 권한 검증이다. ID를 아는 것과 접근 권한은 다르다.

### 관련 문서

07, 08, 11, 33

## 질문의 질문으로 더 깊게 보기

### 이 문서에서 잡아야 할 CS 감각

이 구간의 핵심은 API가 단순한 입구가 아니라 시스템의 계약이자 보안 경계라는 점이다. 누가 호출할 수 있는지, 어떤 입력을 믿을 수 있는지, 실패를 어떤 형태로 돌려주는지가 서비스 전체의 안정성을 결정한다.

### 질문의 질문

아래 질문들은 답을 외우기 위한 질문이 아니다. 하나씩 답하면서 "왜 그런 설계가 필요한지"를 설명하는 연습을 하기 위한 질문이다.

- 이 요청에서 서버가 믿어도 되는 값과 믿으면 안 되는 값은 무엇인가?
- 인증된 사용자라는 사실만으로 이 리소스에 접근해도 되는가?
- 로그, 에러 응답, 헤더 중 어디로 민감 정보가 새어나갈 수 있는가?
- 이 개념을 적용하면 어떤 문제가 해결되지만, 어떤 비용이나 복잡도가 새로 생기는가?
- 장애가 발생했을 때 이 개념은 문제를 예방하는가, 감지하는가, 복구하는가?
- PayFlow 코드에서 이 개념을 확인하려면 어떤 서비스, 테이블, 로그, 테스트를 봐야 하는가?

#### 핵심 답변

OWASP API Security 개념은 PayFlow에서 다음 이유로 중요하다.

- API와 보안 개념은 PayFlow에서 외부 요청이 내부 금전 명령으로 바뀌는 경계를 안전하게 만들기 위해 필요하다.
- Gateway가 인증 정보를 검증하지만, 실제 리소스 소유권의 진실은 각 도메인 서비스의 DB에 있다.
- 사용자가 보낸 userId나 walletId를 그대로 믿으면 다른 사람의 지갑에 접근하거나 금액을 조작할 수 있다.
- JWT 검증, 소유권 확인, 입력 검증, 내부 헤더 재생성, 민감 정보 마스킹, Secret 분리로 방어한다.
- 운영에서는 인증 실패율, 인가 실패율, 4xx 비율, 비정상 요청 패턴, 민감 정보 로그 노출 여부를 확인한다.

#### 질문에 답하는 방식

좋은 답변은 용어 정의에서 멈추지 않고 다음 순서로 이어져야 한다.

1. 어떤 데이터나 요청을 보호하려는 개념인지 말한다.
2. 그 데이터의 진실의 원천이 어느 서비스에 있는지 말한다.
3. 장애, 중복, 동시성, 지연 중 어떤 상황에서 문제가 생기는지 설명한다.
4. 코드 레벨의 방어 수단과 운영 레벨의 확인 수단을 함께 말한다.

#### PayFlow 예시 답변

```text
OWASP API Security 개념은 PayFlow에서 API 보안 취약점이 지갑 소유권 우회나 개인정보 노출로 이어지지 않게 하기 위해 필요하다.
이 개념이 없으면 BOLA 취약점으로 다른 사용자의 지갑, 송금 내역, 정산 정보에 접근할 수 있다.
그래서 코드에서는 리소스 단위 인가, 응답 DTO 분리, Rate Limit, 에러 메시지 최소화를 적용하고,
운영에서는 403 발생 패턴, 과도한 요청, 민감 필드 응답 여부를 확인해야 한다.
```

#### 더 생각해볼 점

이 답안은 하나의 예시다. 실제 설계에서는 성능, 복잡도, 장애 복구 난이도 사이의 trade-off를 함께 봐야 한다. 특히 PayFlow처럼 돈을 다루는 시스템에서는 빠른 성공보다 명확한 실패, 추적 가능한 상태, 재처리 가능한 구조가 더 중요하다.

### PayFlow에 대입해보기

1. 이 개념이 가장 직접적으로 연결되는 PayFlow 서비스 하나를 고른다.
2. 그 서비스가 소유한 데이터의 "진실의 원천"이 무엇인지 적어본다.
3. 동시에 두 요청이 들어오거나, 네트워크가 끊기거나, 프로세스가 죽는 상황을 가정한다.
4. 그 상황에서 데이터가 깨지지 않으려면 어떤 제약조건, 트랜잭션, 락, 이벤트, 재처리 장치가 필요한지 설명한다.
5. 마지막으로 운영자가 문제를 발견할 수 있는 로그나 지표가 무엇인지 적어본다.

OWASP API Security 개념은 PayFlow에서 ______ 문제를 막기 위해 필요하다.
