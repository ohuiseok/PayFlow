# Deadlock

## 핵심 개념

### 데드락의 발생 조건

Deadlock은 두 개 이상의 트랜잭션이 서로가 가진 락을 기다리며 영원히 진행하지 못하는 상태다. 데드락이 발생하려면 네 가지 조건이 동시에 충족되어야 한다(Coffman 조건).

1. **상호 배제(Mutual Exclusion)**: 자원을 한 번에 하나의 프로세스만 사용할 수 있다.
2. **점유 대기(Hold and Wait)**: 자원을 점유한 채 다른 자원을 기다린다.
3. **비선점(No Preemption)**: 다른 트랜잭션의 락을 강제로 빼앗을 수 없다.
4. **순환 대기(Circular Wait)**: T1 -> T2 -> T1 형태로 순환 대기가 형성된다.

데이터베이스에서는 이 중 **순환 대기**를 제거하는 것이 가장 현실적인 예방법이다.

### DB 데드락의 대표 패턴

**패턴 1: 역방향 송금**

```text
T1: A->B 송금 - wallet_A 락 획득 -> wallet_B 락 대기
T2: B->A 송금 - wallet_B 락 획득 -> wallet_A 락 대기
-> 순환 대기 = 데드락
```

**패턴 2: 같은 행의 인덱스 Gap Lock**

```sql
-- T1: id > 10 범위 Gap Lock 획득
SELECT * FROM transfer WHERE id > 10 FOR UPDATE;

-- T2: id = 15 행 삽입 시도 -> Gap Lock으로 대기
INSERT INTO transfer (id, ...) VALUES (15, ...);

-- T1: 같은 범위에 다른 조건으로 다시 락 시도 -> 데드락
```

**패턴 3: 인덱스 순서 불일치**

한 트랜잭션이 Primary Key 인덱스 락을 잡고 Secondary Index 락을 기다리고, 다른 트랜잭션이 반대 순서로 락을 잡으면 데드락이 생긴다.

### 데드락 감지와 해결

InnoDB는 내장 데드락 감지 알고리즘을 가진다. 락 대기 그래프에서 순환을 발견하면 피해가 가장 적은 트랜잭션(Undo 비용이 낮은 쪽)을 선택해 롤백한다. 롤백된 트랜잭션은 `ER_LOCK_DEADLOCK` 오류를 받는다.

`innodb_deadlock_detect` 설정으로 데드락 감지를 활성화/비활성화할 수 있다. 비활성화하면 `innodb_lock_wait_timeout`(기본 50초) 후 타임아웃으로 해결한다.

### 데드락 예방: 락 획득 순서 통일

가장 효과적인 예방법은 여러 자원을 잠글 때 항상 같은 순서로 잠그는 것이다.

```java
// 잘못된 구현: 순서가 보장되지 않음
// T1은 fromWallet=A, toWallet=B 순서로 잠금
// T2는 fromWallet=B, toWallet=A 순서로 잠금 -> 데드락 가능
walletRepository.findByIdForUpdate(fromWalletId);
walletRepository.findByIdForUpdate(toWalletId);

// 올바른 구현: walletId 기준으로 정렬해 항상 같은 순서
List<Long> sortedIds = Stream.of(fromWalletId, toWalletId)
    .sorted()
    .collect(Collectors.toList());

Wallet first = walletRepository.findByIdForUpdate(sortedIds.get(0));
Wallet second = walletRepository.findByIdForUpdate(sortedIds.get(1));
```

이 방법으로 순환 대기 조건을 제거한다.

### 데드락 처리: 재시도 정책

데드락은 완전히 없애기 어렵다. 감지 후 재시도 정책을 설계하는 것이 중요하다.

```java
@Retryable(
    value = {DeadlockLoserDataAccessException.class, CannotAcquireLockException.class},
    maxAttempts = 3,
    backoff = @Backoff(delay = 50, multiplier = 2, random = true)
)
@Transactional
public void transfer(TransferRequest request) {
    // 송금 로직
}
```

재시도 간격에 랜덤성(Jitter)을 추가하면 여러 트랜잭션이 동시에 재시도해 다시 데드락이 생기는 것을 방지한다.

### MySQL 데드락 로그 분석

데드락이 발생하면 MySQL 에러 로그에 기록된다. `SHOW ENGINE INNODB STATUS`로 마지막 데드락 정보를 볼 수 있다.

```text
LATEST DETECTED DEADLOCK
------------------------
...
TRANSACTION 1, ACTIVE ... starting index read
...
LOCK WAIT ... lock struct(s), heap size ...
...
HOLDS THE LOCK(S):
RECORD LOCKS space id ... page no ... n bits ... index PRIMARY of table `payflow`.`wallet`
...
WAITING FOR THIS LOCK TO BE GRANTED:
...
TRANSACTION 2, ACTIVE ...
...
```

분석 포인트:
- 어떤 테이블, 어떤 인덱스에서 발생했는가
- 어떤 트랜잭션이 희생양(Victim)으로 롤백됐는가
- 락 대기 순서는 어떠했는가

이를 보고 락 획득 순서 문제인지, 인덱스 설계 문제인지, 트랜잭션 범위 문제인지 파악한다.

### 분산 락에서의 데드락

Redis 분산 락에서도 유사한 문제가 생긴다.

```text
Service1: lock:wallet:A 획득 -> lock:wallet:B 대기
Service2: lock:wallet:B 획득 -> lock:wallet:A 대기
```

DB 데드락은 DB가 자동으로 감지하고 롤백하지만, 분산 락 데드락은 DB처럼 자동 감지가 안 된다. TTL 만료 또는 타임아웃으로만 해결된다. 따라서 분산 락에서도 획득 순서를 통일하거나 타임아웃을 짧게 설정해야 한다.

### 데드락과 락 타임아웃의 차이

**데드락**: 순환 대기로 영원히 진행 못하는 상태. DB가 감지해 빠르게 해결.
**락 타임아웃**: 지정된 시간 동안 락을 획득하지 못하면 오류 반환. `innodb_lock_wait_timeout`으로 설정(기본 50초).

결제 시스템에서는 락 타임아웃을 짧게 설정해(5~10초) 사용자가 너무 오래 기다리지 않도록 하고, 타임아웃 오류도 적절히 처리해야 한다.

### 흔한 오해와 함정

**"데드락은 코드 버그다"**: 락 순서가 잘못 설계된 경우에는 버그지만, 트랜잭션 격리와 동시성 특성상 정상적인 설계에서도 발생할 수 있다. 감지와 재시도까지 설계의 일부로 봐야 한다.

**"트랜잭션을 짧게 유지하면 데드락이 없다"**: 짧은 트랜잭션은 데드락 가능성을 낮추지만 완전히 없애지는 못한다. 순환 대기는 트랜잭션 길이와 무관하게 발생할 수 있다.

**"데드락 로그가 없으면 데드락이 없다"**: 락 타임아웃 오류나 롤백 오류는 데드락이 아닌 단순 타임아웃으로 처리될 수도 있어 로그만으로 판단하기 어렵다.

**"인덱스가 데드락과 관계없다"**: 인덱스 없는 조회는 더 넓은 Gap Lock을 만들어 데드락 가능성을 높인다. 인덱스 설계가 동시성에도 영향을 준다.


## PayFlow 연결

송금에서는 보내는 지갑과 받는 지갑을 함께 변경할 수 있다. 여러 송금이 반대 방향으로 동시에 발생하면 락 획득 순서가 엇갈려 데드락이 생길 수 있다.

```text
T1: wallet A lock -> wallet B lock 대기
T2: wallet B lock -> wallet A lock 대기
```

## 실무 포인트

- 여러 자원을 잠글 때는 항상 같은 순서로 잠근다.
- 트랜잭션을 짧게 유지한다.
- 데드락은 완전히 제거하기보다 감지 후 재시도까지 고려한다.
- 인덱스 부족은 불필요하게 넓은 락을 만들 수 있다.
- 데드락 로그를 분석해 원인을 찾는다.

## 체크 질문

- 데드락이 발생하는 조건은 무엇인가
- 송금 시스템에서 락 획득 순서가 중요한 이유는 무엇인가
- 데드락이 발생했을 때 애플리케이션은 어떻게 대응해야 하는가

## 실무 설계 보강

### 대표 장애 시나리오

A에서 B로, B에서 A로 송금이 동시에 발생해 서로의 지갑 락을 기다린다.

### 잘못된 구현 예시

~~~text
락이 필요하다는 것만 보고 락 획득 순서는 신경 쓰지 않는다.
~~~

### 좋은 구현 예시

~~~text
여러 자원을 잠글 때 walletId 정렬처럼 항상 같은 순서로 잠근다.
~~~

### 대안과 선택 이유

애플리케이션 코드에서만 검증하는 방식도 있지만, 동시 요청과 장애 상황에서는 코드 검증만으로 부족하다. PayFlow는 DB 트랜잭션, 락, 유니크 제약, 인덱스, 실행 계획을 함께 사용해 데이터 계층에서도 불변식을 지키는 쪽이 더 안전하다.

### PayFlow에서 확인할 위치

wallet-service transfer/withdraw/deposit flow, DB deadlock log

### 면접에서 설명하기

데드락은 버그가 없어도 락 순서만 엇갈리면 발생한다. 감지와 재시도까지 설계해야 한다.

### 관련 문서

#### 핵심 답변

Deadlock 개념은 PayFlow에서 다음 이유로 중요하다.

- DB 트랜잭션과 동시성 개념은 PayFlow에서 잔액, 송금 상태, 멱등성 키 같은 핵심 데이터를 깨지지 않게 지키기 위해 필요하다.

### PayFlow에 대입해보기

1. 이 개념이 가장 직접적으로 연결되는 PayFlow 서비스 하나를 고른다.
2. 그 서비스가 소유한 데이터의 "진실의 원천"이 무엇인지 적어본다.
3. 동시에 두 요청이 들어오거나, 네트워크가 끊기거나, 프로세스가 죽는 상황을 가정한다.
4. 그 상황에서 데이터가 깨지지 않으려면 어떤 제약조건, 트랜잭션, 락, 이벤트, 재처리 장치가 필요한지 설명한다.
5. 마지막으로 운영자가 문제를 발견할 수 있는 로그나 지표가 무엇인지 적어본다.
