# Load Testing And Performance

## 핵심 개념

부하 테스트(Load Testing)는 시스템이 예상 트래픽에서 얼마나 안정적으로 동작하는지 검증하는 테스트다. 성능은 응답 시간, 처리량, 에러율, 자원 사용률을 함께 봐야 한다.

### 부하 테스트의 종류

목적에 따라 다른 종류의 테스트를 선택해야 한다.

**Load Test (부하 테스트)**: 예상 최대 트래픽에서의 성능을 측정한다. "평상시 + 피크 트래픽을 처리할 수 있는가"를 확인한다.

**Stress Test (스트레스 테스트)**: 시스템의 한계를 찾는다. 트래픽을 계속 높여가면서 언제 어떻게 실패하는지 파악한다. 한계점을 알아야 장애를 미리 대비할 수 있다.

**Soak Test (장기 안정성 테스트)**: 낮은 부하를 수 시간~수일 유지하면서 메모리 누수, 커넥션 누수, 성능 저하를 감지한다. 단기 테스트에서는 나타나지 않는 문제를 발견한다.

**Spike Test (급격한 부하 테스트)**: 트래픽이 갑자기 몇 배로 증가했을 때의 동작을 검증한다. 이벤트나 마케팅 이후 급격한 트래픽 증가 상황을 시뮬레이션한다.

**Volume Test**: 대용량 데이터에서의 성능을 측정한다. DB에 수억 건의 데이터가 있을 때 쿼리 성능을 확인한다.

### 핵심 성능 지표

**처리량 (Throughput / RPS, TPS)**

초당 처리할 수 있는 요청 수(RPS) 또는 트랜잭션 수(TPS)다.

```text
목표: 송금 API 1000 TPS 처리 가능
현재: 최대 600 TPS에서 에러 발생
→ 병목 원인 분석 필요
```

**응답 시간 (Response Time / Latency)**

단순 평균(Mean)은 아웃라이어에 취약하다. 백분위수(Percentile)가 더 의미 있다.

```text
p50 (Median): 절반의 요청이 이 시간 이하로 처리됨
p90: 90%의 요청이 이 시간 이하로 처리됨
p99: 99%의 요청이 이 시간 이하로 처리됨
p99.9: 0.1% 요청 = 1000 RPS면 초당 1건이 이 이상

SLO 설정 예: p99 < 500ms
```

**에러율 (Error Rate)**

전체 요청 중 실패한 비율이다. 부하가 증가할수록 에러율이 증가하는 시점을 파악해야 한다.

**자원 사용률 (Resource Utilization)**

CPU, 메모리, 디스크 I/O, 네트워크 I/O를 함께 확인한다. 어느 자원이 먼저 포화되는지 파악해야 병목을 찾을 수 있다.

### k6를 이용한 부하 테스트

k6는 JavaScript로 부하 테스트 시나리오를 작성하는 현대적 도구다.

```javascript
// k6 부하 테스트 스크립트
import http from 'k6/http';
import { sleep, check } from 'k6';

export const options = {
  stages: [
    { duration: '1m', target: 100 },  // 1분간 100 VU로 램프업
    { duration: '3m', target: 100 },  // 3분간 100 VU 유지
    { duration: '1m', target: 500 },  // 1분간 500 VU로 증가
    { duration: '3m', target: 500 },  // 3분간 500 VU 유지
    { duration: '1m', target: 0 },    // 1분간 램프다운
  ],
  thresholds: {
    http_req_duration: ['p(99)<500'],  // p99 < 500ms
    http_req_failed: ['rate<0.01'],    // 에러율 < 1%
  },
};

export default function () {
  const payload = JSON.stringify({
    fromWalletId: 'wallet-001',
    toWalletId: 'wallet-002',
    amount: 10000,
    idempotencyKey: `key-${__VU}-${__ITER}`,
  });

  const res = http.post('http://api.payflow/transfers', payload, {
    headers: { 'Content-Type': 'application/json' },
  });

  check(res, {
    'status is 200': (r) => r.status === 200,
    'response time < 500ms': (r) => r.timings.duration < 500,
  });

  sleep(1);
}
```

**Gatling**은 Scala/Java 기반으로 더 복잡한 시나리오 작성에 적합하다. **Apache JMeter**는 GUI 기반으로 진입 장벽이 낮다.

### 부하 테스트 환경 설계

부하 테스트는 격리된 환경에서 수행해야 한다. 운영 환경에서 부하 테스트를 하면 실제 사용자에게 영향을 주고, 개발 환경은 운영과 스펙이 달라 의미 있는 결과를 얻기 어렵다. 별도의 스테이징/성능 테스트 환경이 필요하다.

테스트 데이터도 현실적이어야 한다. DB에 데이터가 거의 없으면 쿼리 성능이 실제와 다르다. 운영 규모와 비슷한 데이터 볼륨을 준비해야 한다.

부하 발생기(Load Generator)도 병목이 될 수 있다. 단일 서버에서 너무 많은 가상 사용자를 생성하면 부하 발생기가 먼저 CPU/네트워크 한계에 달한다. 분산 부하 발생을 지원하는 도구(k6 cloud, Gatling Enterprise)를 사용하거나 여러 서버에서 분산 실행해야 한다.

### 결과 분석

부하 테스트 결과는 단순히 "통과/실패"가 아니라 패턴을 분석해야 한다.

```text
정상 패턴:
  트래픽 증가 → 응답 시간 완만히 증가 → 에러 없음

비정상 패턴 1 - 병목:
  특정 트래픽 이상에서 응답 시간 급등
  → DB, 외부 API, 락 등 병목 지점 확인

비정상 패턴 2 - 메모리 누수:
  Soak Test 중 응답 시간이 서서히 증가
  → GC 로그, 힙 덤프 분석

비정상 패턴 3 - 커넥션 풀 고갈:
  특정 트래픽에서 갑자기 에러율 급등
  → DB/Redis 커넥션 풀 설정 확인
```

### 지속적 성능 테스트

성능 테스트는 배포 파이프라인에 통합해야 한다. 매 배포마다 기준 성능 지표와 비교해 회귀(Regression)를 감지한다.

```yaml
# CI/CD 파이프라인에 k6 통합
- name: Performance Test
  run: |
    k6 run --out json=results.json load-test.js
    # 결과 분석하여 기준치 초과 시 파이프라인 실패
```

### 흔한 오해와 함정

**오해: 부하 테스트는 배포 전에 한 번만 하면 된다**

코드 변경, DB 스키마 변경, 인프라 변경 모두 성능에 영향을 준다. 지속적으로 성능을 측정해야 회귀를 빨리 발견할 수 있다.

**함정: VU(가상 사용자)와 RPS를 혼동**

100 VU가 각각 1초에 1번 요청하면 100 RPS다. sleep 시간에 따라 같은 VU 수에서도 RPS가 크게 달라진다. 목표 RPS를 기준으로 VU 수를 계산해야 한다.

**함정: 콜드 스타트 영향**

테스트 시작 직후에는 JVM JIT 컴파일, 캐시 미충전, DB 커넥션 풀 생성 등의 영향으로 응답이 느리다. 워밍업 기간의 데이터는 분석에서 제외하거나 별도로 분석해야 한다.

### Trade-off

부하 테스트 환경을 운영과 동일하게 유지하는 것은 비용이 크다. 축소된 환경에서 테스트하면 비용은 절감되지만 결과의 정확도가 떨어진다. 운영 환경 1/10 규모로 테스트한 결과를 10배로 추정하는 것은 항상 정확하지 않다. 분산 시스템에서는 스케일에 따라 병목이 다른 곳에서 나타날 수 있다.

## PayFlow 연결

PayFlow에서는 회원가입보다 송금 요청, 지갑 조회, 송금 상태 조회, Outbox 발행, Kafka Consumer 처리량이 중요하다.

k6 같은 도구로 동시 송금 요청을 만들어 동시성 문제와 병목을 찾을 수 있다.

## 실무 포인트

- 정상 시나리오와 피크 시나리오를 나눈다.
- 평균보다 p95, p99 응답 시간을 본다.
- DB, Redis, Kafka 지표를 함께 본다.
- 부하 테스트 데이터는 정리 가능해야 한다.
- 금액 관련 테스트는 중복과 정합성을 검증한다.

## 체크 질문

- 부하 테스트에서 평균 응답 시간만 보면 안 되는 이유는 무엇인가
- 송금 부하 테스트에서 정합성 검증이 필요한 이유는 무엇인가
- 처리량 병목은 어디에서 발생할 수 있는가

## 실무 설계 참고

### 대표 장애 시나리오

동시 송금 부하에서 락 대기와 timeout이 급증하지만 개발 환경에서는 발견하지 못한다.

### 잘못된 구현 예시

~~~text
기능 테스트가 통과했으니 성능도 충분하다고 생각한다.
~~~

### 좋은 구현 예시

~~~text
k6로 실제 흐름에 가까운 부하를 만들고 latency, error, DB/Redis/Kafka 지표를 함께 본다.
~~~

### 대안과 선택 이유

문제가 생기면 로그를 직접 뒤져 해결하는 방식도 있지만, MSA 결제 흐름에서는 원인 추적이 너무 늦어진다. PayFlow는 테스트, 계약 검증, 관측성, SLO, 알림, 장애 주입을 통해 배포 전후에 시스템을 증명하는 방식이 더 전문적이다.

### PayFlow에서 확인할 위치

k6 scripts, service metrics, database metrics, Kafka dashboard

### 면접에서 설명하기

부하 테스트는 빠른지 보는 것이 아니라 어느 지점부터 어떤 방식으로 무너지는지 확인하는 작업이다.

## 질문의 질문으로 더 깊게 보기

### 이 문서에서 잡아야 할 CS 감각

이 구간의 핵심은 운영에서 시스템을 증명하는 방법이다. 테스트는 배포 전의 증명이고, 관측성은 배포 후의 증명이다. 결제 시스템은 "잘 될 것이다"가 아니라 "틀어졌을 때 발견하고 복구할 수 있다"를 보여줘야 한다.

### 질문의 질문

아래 질문들은 답을 외우기 위한 질문이 아니다. 하나씩 답하면서 "왜 그런 설계가 필요한지"를 설명하는 연습을 하기 위한 질문이다.

- 이 개념이 제대로 동작한다는 것을 어떤 지표나 테스트로 증명할 수 있는가?
- 정상 케이스보다 실패 케이스에서 어떤 상태가 남아야 하는가?
- 운영자가 새벽에 알림을 받았을 때 이 문서만 보고 다음 행동을 결정할 수 있는가?
- 이 개념을 적용하면 어떤 문제가 해결되지만, 어떤 비용이나 복잡도가 새로 생기는가?
- 장애가 발생했을 때 이 개념은 문제를 예방하는가, 감지하는가, 복구하는가?
- PayFlow 코드에서 이 개념을 확인하려면 어떤 서비스, 테이블, 로그, 테스트를 봐야 하는가?

#### 핵심 답변

Load Testing And Performance 개념은 PayFlow에서 다음 이유로 중요하다.

- 운영과 테스트 개념은 PayFlow가 실제 장애 상황에서도 기대한 대로 동작하는지 증명하기 위해 필요하다.
- 테스트 결과, 로그, 메트릭, 트레이스, 알림, 장애 대응 기록이 시스템 신뢰성의 근거다.
- 테스트하지 않은 실패 경로는 운영에서 처음 발견되고, 관측성이 없으면 어디서 깨졌는지 알 수 없다.
- 단위/통합/E2E/계약/장애 주입 테스트와 로그, 메트릭, 트레이싱, SLO, 알림 체계로 방어한다.
- 운영에서는 p95/p99 latency, 에러율, 테스트 실패율, 알림 발생 수, Trace별 실패 지점, SLO 위반 시간을 본다.

#### 질문에 답하는 방식

좋은 답변은 용어 정의에서 멈추지 않고 다음 순서로 이어져야 한다.

1. 어떤 데이터나 요청을 보호하려는 개념인지 말한다.
2. 그 데이터의 진실의 원천이 어느 서비스에 있는지 말한다.
3. 장애, 중복, 동시성, 지연 중 어떤 상황에서 문제가 생기는지 설명한다.
4. 코드 레벨의 방어 수단과 운영 레벨의 확인 수단을 함께 말한다.

### PayFlow에 대입해보기

1. 이 개념이 가장 직접적으로 연결되는 PayFlow 서비스 하나를 고른다.
2. 그 서비스가 소유한 데이터의 "진실의 원천"이 무엇인지 적어본다.
3. 동시에 두 요청이 들어오거나, 네트워크가 끊기거나, 프로세스가 죽는 상황을 가정한다.
4. 그 상황에서 데이터가 깨지지 않으려면 어떤 제약조건, 트랜잭션, 락, 이벤트, 재처리 장치가 필요한지 설명한다.
5. 마지막으로 운영자가 문제를 발견할 수 있는 로그나 지표가 무엇인지 적어본다.
