# Wallet Balance Integrity

## 핵심 개념

### 지갑 잔액 정합성이란 무엇인가

지갑 잔액 정합성(Wallet Balance Integrity)은 시스템에 저장된 잔액 값이 실제 거래 이력의 누적 합계와 항상 일치하도록 보장하는 속성이다. 결제 시스템에서 잔액은 단순한 숫자 하나가 아니라, 계정 생성 이후 발생한 모든 입금·출금·충전·환불 이벤트의 합산 결과다. 따라서 잔액이 맞다는 것은 그 숫자 자체가 맞는다는 뜻이 아니라, 해당 숫자를 설명할 수 있는 거래 이력이 모두 존재한다는 것을 의미한다.

### 잔액 정합성이 깨지는 원인

잔액 정합성이 깨지는 경로는 크게 세 가지다.

**첫째, 동시성 문제(Race Condition)**: 두 요청이 동시에 같은 지갑 잔액을 읽고, 각각 잔액을 계산한 후 덮어쓰는 상황이다. 잔액 10,000원인 지갑에 동시에 8,000원과 9,000원 출금 요청이 들어오면, 두 요청이 각각 10,000원을 읽고 자신의 출금만 반영해 한쪽 차감이 무시되거나 잔액이 음수가 될 수 있다.

```text
T1: SELECT balance = 10000
T2: SELECT balance = 10000
T1: UPDATE balance = 10000 - 8000 = 2000  (성공)
T2: UPDATE balance = 10000 - 9000 = 1000  (잔액 오류: T1 차감이 무시됨)
```

이를 방어하는 방법은 낙관적 락(version 컬럼 비교)이나 비관적 락(SELECT FOR UPDATE)이다.

**둘째, 중복 처리**: 같은 출금 요청이 네트워크 재시도, 메시지 재전달, 배치 재처리 등 여러 이유로 두 번 이상 반영되는 상황이다. 결제 시스템에서 중복 차감은 사용자의 돈이 실제보다 적게 유지되는 심각한 오류다.

**셋째, 트랜잭션 경계 문제**: 잔액 테이블 업데이트와 잔액 변경 이력(wallet_history) 저장이 별개 트랜잭션으로 처리되면, 하나가 성공하고 하나가 실패하는 부분 성공 상태가 발생한다. 잔액은 바뀌었지만 이력이 없거나, 이력은 있지만 잔액이 반영되지 않은 상태가 생긴다.

### 잔액 정합성 보장을 위한 핵심 메커니즘

**단일 진실 원천(Single Source of Truth)**: 지갑 잔액의 진실을 오직 wallet-service의 wallet 테이블 하나에만 두어야 한다. 다른 서비스가 직접 wallet 테이블을 수정하거나, Redis 캐시에 별도 잔액을 관리하면서 동기화를 맞추는 구조는 불일치가 발생할 수 있는 지점을 만든다.

**원자적 잔액 변경과 이력 저장**: 잔액 변경(wallet 테이블 UPDATE)과 이력 기록(wallet_history INSERT)을 반드시 하나의 DB 트랜잭션으로 묶어야 한다. 둘이 같은 트랜잭션에 있어야만 중간 상태 없이 성공 또는 실패가 보장된다.

```sql
BEGIN;
  UPDATE wallet SET balance = balance - 10000
   WHERE id = :walletId AND balance >= 10000;
  -- affected rows = 0이면 잔액 부족 -> ROLLBACK
  INSERT INTO wallet_history (wallet_id, type, amount, reference_type, reference_id)
    VALUES (:walletId, 'WITHDRAW', 10000, 'TRANSFER', :transferId);
COMMIT;
```

**음수 잔액 방어**: SQL UPDATE WHERE 절에 `balance >= amount` 조건을 포함시키면, 잔액이 부족한 경우 UPDATE가 0 rows affected를 반환한다. DB CHECK constraint(`CHECK (balance >= 0)`)를 추가하면 이중 방어가 된다.

**참조 중복 차단(Idempotency Key)**: `reference_type`과 `reference_id`의 조합에 UNIQUE constraint를 걸어두면, 같은 거래 ID로 출금을 두 번 시도할 때 두 번째 요청이 DB 제약조건 위반으로 실패한다. 이것이 잔액 정합성을 지키는 가장 강력한 방어선이다.

```sql
ALTER TABLE wallet_history
  ADD CONSTRAINT uq_wallet_history_ref
  UNIQUE (wallet_id, reference_type, reference_id);
```

### 잔액 정합성 검증 (대사)

실시간 트랜잭션으로 정합성을 유지하는 것과 별도로, 주기적으로 잔액이 이력과 일치하는지 검증하는 대사(Reconciliation) 잡을 운영해야 한다.

```text
wallet.balance == SUM(amount) WHERE type IN ('DEPOSIT', 'CHARGE')
               - SUM(amount) WHERE type IN ('WITHDRAW')
```

이 값이 다르면 중복 처리, 이력 누락, 직접 DB 수정 같은 문제가 발생했다는 신호다.

### 흔한 오해와 함정

- "트랜잭션을 쓰니까 안전하다"는 오해: 트랜잭션은 ACID를 보장하지만, 동시에 실행되는 두 트랜잭션 사이의 경쟁 상태는 격리 수준에 따라 다르게 처리된다. READ COMMITTED 수준에서는 동시 출금 경쟁 상태를 완전히 막을 수 없다. 비관적 락 또는 낙관적 락이 별도로 필요하다.
- "잔액만 맞으면 된다"는 오해: 잔액 숫자가 맞더라도 이력이 없으면 감사, 분쟁 대응, 정산 검증에서 문제가 생긴다. 잔액과 이력 모두 정합성 대상이다.
- "캐시에서 잔액을 관리하면 빠르다"는 함정: Redis에 잔액을 캐시하고 DB와 동기화하는 구조는 캐시-DB 불일치를 만들 수 있다. 잔액 정합성이 중요한 시스템에서는 캐시를 진실 원천으로 사용하지 않는 것이 안전하다.

### 성능과 정합성의 트레이드오프

비관적 락(SELECT FOR UPDATE)은 정합성이 높지만 같은 지갑에 대한 요청이 순차 처리되어 처리량이 제한된다. 낙관적 락은 동시성이 높지만 충돌 시 재시도 로직이 필요하고, 충돌이 잦은 상황에서는 오히려 처리량이 떨어질 수 있다. 결제 시스템에서는 정합성을 우선시하므로 비관적 락 또는 DB 레벨 constraint 조합을 주로 선택한다.


## PayFlow 연결

PayFlow에서 `wallet-service`는 지갑 잔액의 진실을 가진다. 출금, 입금, 충전은 모두 `wallet-service`를 통해 처리되어야 하며, 다른 서비스가 DB를 직접 수정하면 안 된다.

잔액 변경 시에는 잔액 테이블과 잔액 변경 이력을 함께 남겨야 한다. 문제가 생겼을 때 왜 잔액이 바뀌었는지 추적할 수 있어야 하기 때문이다.

## 실무 포인트

- 잔액 변경은 트랜잭션으로 묶는다.
- 음수 잔액을 허용하지 않는다.
- 변경 이력을 append-only로 남긴다.
- 중복 reference를 막는다.
- 동시 출금 테스트를 반드시 수행한다.

## 체크 질문

- 지갑 잔액을 다른 서비스가 직접 수정하면 왜 위험한가
- 잔액 변경 이력이 필요한 이유는 무엇인가
- 같은 출금 요청이 두 번 처리되면 어떤 문제가 생기는가

## 실무 설계 보강

### 대표 장애 시나리오

같은 referenceId 출금이 두 번 반영되어 지갑 잔액이 실제 거래보다 적어진다.

### 잘못된 구현 예시

~~~text
잔액 숫자만 맞으면 된다고 보고 변경 이력과 중복 근거를 남기지 않는다.
~~~

### 좋은 구현 예시

~~~text
잔액과 변경 이력을 같은 트랜잭션으로 저장하고 referenceType/referenceId를 unique하게 관리한다.
~~~

### 대안과 선택 이유

현재 잔액이나 최종 집계값만 저장하는 단순한 방식도 있지만, 결제 시스템에서는 나중에 왜 그런 결과가 나왔는지 설명할 수 있어야 한다. PayFlow는 금액 정책, 원장, 이력, 정산 결과, 대사 리포트를 남기는 방식이 분쟁 대응과 복구에 더 강하다.

### PayFlow에서 확인할 위치

wallet-service balance table, wallet history, withdraw/deposit API

이 구간의 핵심은 결제 도메인의 불변식이다. 금액은 정확해야 하고, 잔액은 설명 가능해야 하며, 원장과 정산은 나중에 검증 가능해야 한다. 좋은 결제 시스템은 성공 처리보다 설명 가능한 기록을 더 중요하게 본다.

### 질문의 질문

아래 질문들은 답을 외우기 위한 질문이 아니다. 하나씩 답하면서 "왜 그런 설계가 필요한지"를 설명하는 연습을 하기 위한 질문이다.

### PayFlow에 대입해보기

1. 이 개념이 가장 직접적으로 연결되는 PayFlow 서비스 하나를 고른다.
2. 그 서비스가 소유한 데이터의 "진실의 원천"이 무엇인지 적어본다.
3. 동시에 두 요청이 들어오거나, 네트워크가 끊기거나, 프로세스가 죽는 상황을 가정한다.

5. 마지막으로 운영자가 문제를 발견할 수 있는 로그나 지표가 무엇인지 적어본다.
