# Timezone And Date Boundary

> 도메인 전환 안내: 현재 PayFlow는 **청년 정책 참여 미션 및 지원금 지급 플랫폼**으로 설명한다. 내부 구현 호환성을 위해 `PARENT`/`CHILD`, `/api/families`, `/api/missions`, `/api/cashbook`, `reward-service` 같은 명칭은 유지하지만, 문서와 발표에서는 각각 **기관 담당자**, **청년 참여자**, **참여자 연결**, **정책 미션**, **지원금 사용 내역**, **정책 미션/지원금 서비스**로 해석한다.

## 핵심 개념

시간대와 날짜 경계는 시스템이 어느 기준 시각으로 날짜를 나누고 저장할지 정하는 문제다. 글로벌 시스템에서는 UTC 저장과 지역 시간 표시를 분리하는 것이 일반적이다.

## PayFlow 연결

정산은 보통 일별로 수행된다. 이때 "하루"가 한국 시간 기준인지 UTC 기준인지 명확해야 한다.

예를 들어 2026-05-30 00:30 KST 거래는 UTC로는 2026-05-29 15:30이다. 기준이 다르면 정산일이 달라진다.

## 실무 포인트

- 저장 시각은 UTC 또는 명확한 기준으로 통일한다.
- 정산 기준일은 비즈니스 시간대로 정의한다.
- 날짜 조건은 시작 포함, 끝 미포함 범위가 안전하다.
- 서버 시간대 설정에 의존하지 않는다.
- 배치 재실행 시 같은 기준일이 같은 결과를 내야 한다.

## 체크 질문

- UTC 저장과 지역 시간 표시를 분리하는 이유는 무엇인가
- 정산 기준일이 바뀌면 어떤 문제가 생기는가
- 날짜 조회에서 `start <= time < end` 방식이 유리한 이유는 무엇인가

## 실무 설계 참고

### 대표 장애 시나리오

KST 자정 근처 거래가 UTC 기준 전날로 분류되어 정산에서 빠진다.

### 잘못된 구현 예시

~~~text
서버 로컬 시간대를 기준으로 날짜를 계산한다.
~~~

### 좋은 구현 예시

~~~text
저장은 UTC, 비즈니스 기준일은 KST처럼 명시하고 start inclusive/end exclusive 범위를 쓴다.
~~~

### 대안과 선택 이유

현재 구현은 `SETTLEMENT_ZONE=Asia/Seoul`로 스케줄 실행일을 계산하고 이벤트의 `LocalDateTime occurredAt`을 `[00:00, 다음 날 00:00)` 범위로 조회한다. 즉, 아래의 UTC 저장 모델은 권장 설계이며 현재 코드가 완전히 적용한 상태는 아니다.

### PayFlow에서 확인할 위치

`DailySettlementScheduler`, `DailySettlementBatchConfig`, `PaymentSettlementEventPublisher`, `settlement.zone`

### 면접에서 설명하기

시간은 전역 값이 아니다. 정산에서는 날짜 경계가 곧 돈의 경계다.

### 관련 문서

48, 50, 52

## 심화 참고

### 시간 모델을 나누어 생각하기

결제 시스템에서 시간은 하나의 값처럼 보이지만 실제로는 목적에 따라 다르게 다뤄야 한다.

- 발생 시각: 거래가 실제로 생성되거나 승인된 시각이다. 보통 UTC `Instant`로 저장한다.
- 비즈니스 기준일: 정산, 수수료, 리포트에서 사용하는 날짜다. 예를 들어 KST 기준 `2026-05-30`이다.
- 표시 시각: 사용자 화면이나 관리자 화면에서 지역 시간대로 변환해 보여주는 시각이다.
- 처리 시각: 배치나 컨슈머가 해당 이벤트를 처리한 시각이다. 발생 시각과 다를 수 있다.

이 값들을 섞으면 장애가 난다. 예를 들어 정산은 발생 시각 기준이어야 하는데 처리 시각 기준으로 집계하면 Kafka 지연이나 재처리 때문에 정산일이 바뀔 수 있다.

### 설계 체크리스트

- DB에는 `created_at`, `occurred_at`, `processed_at`, `settlement_date`의 의미를 분리해서 둔다.
- 시각 컬럼은 UTC 기준으로 저장하고, 날짜 컬럼은 비즈니스 기준일임을 이름으로 드러낸다.
- 날짜 범위 조회는 `start <= occurred_at AND occurred_at < end` 형태로 작성한다.
- `BETWEEN`처럼 양 끝을 모두 포함하는 조건은 다음 구간과 겹칠 수 있으므로 조심한다.
- 배치 실행 시점의 서버 로컬 시간 대신 명시적인 `businessDate` 파라미터를 받는다.
- 정산 기준 시간대가 바뀌는 경우 기존 데이터와 신규 데이터의 해석 정책을 문서화한다.

### 권장 적용 예시

```text
정산일: 2026-05-30 KST
조회 범위:
  start = 2026-05-29T15:00:00Z
  end   = 2026-05-30T15:00:00Z

조건:
  start <= transfer.occurred_at < end
```

이 모델은 현재 `LocalDateTime` 기반 구현을 `Instant`/UTC 저장으로 개선할 때의 목표다. 현재 구현에서도 명시적인 `businessDate`와 start-inclusive/end-exclusive 범위를 사용하지만, 이벤트 생산자와 DB/server timezone 설정이 동일하다는 전제가 남는다.

### 테스트 포인트

- KST 자정 직전과 직후 거래가 서로 다른 정산일로 들어가는지 확인한다.
- UTC 날짜와 KST 날짜가 다른 거래를 넣고 정산 대상 포함 여부를 검증한다.
- 서버 timezone을 바꿔도 같은 `businessDate`에 대해 같은 결과가 나오는지 확인한다.
- 배치 재실행 시 처리 시각이 달라져도 정산 대상 거래 집합은 바뀌지 않는지 확인한다.

## 질문의 질문으로 더 깊게 보기

### 이 문서에서 잡아야 할 CS 감각

이 구간의 핵심은 결제 도메인의 불변식이다. 금액은 정확해야 하고, 잔액은 설명 가능해야 하며, 원장과 정산은 나중에 검증 가능해야 한다. 좋은 결제 시스템은 성공 처리보다 설명 가능한 기록을 더 중요하게 본다.

### 질문의 질문

아래 질문들은 답을 외우기 위한 질문이 아니다. 하나씩 답하면서 "왜 그런 설계가 필요한지"를 설명하는 연습을 하기 위한 질문이다.

- Timezone And Date Boundary 개념이 없다면 PayFlow에서 가장 먼저 어떤 장애가 생기는가?
- 이 개념은 정확성, 성능, 보안, 운영성 중 무엇을 가장 크게 개선하는가?
- 반대로 이 개념을 잘못 적용하면 어떤 복잡도가 추가되는가?
- 이 개념을 적용하면 어떤 문제가 해결되지만, 어떤 비용이나 복잡도가 새로 생기는가?
- 장애가 발생했을 때 이 개념은 문제를 예방하는가, 감지하는가, 복구하는가?
- PayFlow 코드에서 이 개념을 확인하려면 어떤 서비스, 테이블, 로그, 테스트를 봐야 하는가?

#### 핵심 답변

Timezone And Date Boundary 개념은 PayFlow에서 다음 이유로 중요하다.

- 결제 도메인 개념은 PayFlow에서 돈의 이동을 정확하고 설명 가능하며 나중에 검증 가능하게 만들기 위해 필요하다.
- 지갑 잔액, 잔액 변경 이력, 원장 엔트리, 정산 결과, 대사 리포트가 진실을 검증하는 기준이다.
- 금액 반올림 오류, 중복 정산, 원장 누락, 잔액과 원장 불일치가 발생하면 분쟁과 금전 손실로 이어진다.

- 운영에서는 잔액-원장 차이, 정산 검증 실패 수, 수수료 합계 차이, 중복 정산 차단 건수, 대사 실패 리포트를 본다.

#### 질문에 답하는 방식

좋은 답변은 용어 정의에서 멈추지 않고 다음 순서로 이어져야 한다.

1. 어떤 데이터나 요청을 보호하려는 개념인지 말한다.
2. 그 데이터의 진실의 원천이 어느 서비스에 있는지 말한다.
3. 장애, 중복, 동시성, 지연 중 어떤 상황에서 문제가 생기는지 설명한다.
4. 코드 레벨의 방어 수단과 운영 레벨의 확인 수단을 함께 말한다.

### PayFlow에 대입해보기

1. 이 개념이 가장 직접적으로 연결되는 PayFlow 서비스 하나를 고른다.
2. 그 서비스가 소유한 데이터의 "진실의 원천"이 무엇인지 적어본다.
3. 동시에 두 요청이 들어오거나, 네트워크가 끊기거나, 프로세스가 죽는 상황을 가정한다.
4. 그 상황에서 데이터가 깨지지 않으려면 어떤 제약조건, 트랜잭션, 락, 이벤트, 재처리 장치가 필요한지 설명한다.
5. 마지막으로 운영자가 문제를 발견할 수 있는 로그나 지표가 무엇인지 적어본다.
