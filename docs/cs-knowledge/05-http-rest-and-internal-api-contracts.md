# HTTP REST And Internal API Contracts

## 핵심 개념

HTTP는 클라이언트와 서버, 또는 서비스와 서비스가 요청과 응답을 주고받는 가장 일반적인 프로토콜이다. REST는 HTTP의 메서드, URI, 상태 코드, 표현 방식을 일관되게 사용해 리소스를 다루는 설계 스타일이다.

PayFlow에서는 외부 클라이언트가 API Gateway로 요청하고, Gateway가 내부 서비스로 라우팅한다. 또한 `transfer-service`가 `wallet-service`를 호출하는 것처럼 서비스 간 동기 통신에도 HTTP API가 사용될 수 있다.

### HTTP 메서드와 멱등성

REST에서 HTTP 메서드 선택은 단순한 컨벤션이 아니라 멱등성과 안전성이라는 의미론적 계약을 포함한다.

| 메서드 | 멱등성 | 안전성 | 주요 용도 |
|--------|--------|--------|-----------|
| GET | O | O | 리소스 조회 |
| POST | X | X | 리소스 생성, 명령 실행 |
| PUT | O | X | 리소스 전체 교체 |
| PATCH | X | X | 리소스 부분 수정 |
| DELETE | O | X | 리소스 삭제 |

멱등성이 없는 POST 요청은 재시도 시 중복 생성이 발생할 수 있다. 결제 시스템에서 송금 POST 요청을 재시도하면 같은 금액이 두 번 빠져나갈 수 있다. 이 때문에 `Idempotency-Key` 헤더로 서버 측에서 중복을 감지하는 패턴이 필요하다.

안전성은 요청이 서버 상태를 변경하지 않는다는 보장이다. GET은 여러 번 호출해도 같은 결과를 반환해야 하며, 조회 과정에서 DB를 수정하면 안 된다. 반대로 POST, PUT, PATCH, DELETE는 서버 상태를 바꾸므로 재시도 설계가 더 복잡하다.

### HTTP 상태 코드의 의미 구분

상태 코드는 클라이언트가 재시도 여부, 오류 처리 방식을 결정하는 기준이다. 결제 API에서 상태 코드를 잘못 쓰면 클라이언트가 재시도해서는 안 되는 요청을 재시도하거나, 반대로 재시도해야 할 요청을 포기하게 된다.

```text
2xx: 성공
  200 OK            - 조회 성공, 명령 처리 성공
  201 Created       - 리소스 생성 완료 (Location 헤더 포함 권장)
  202 Accepted      - 비동기 처리 수락 (결과는 나중에)
  204 No Content    - 성공했지만 반환 바디 없음

4xx: 클라이언트 오류 (재시도해도 같은 결과)
  400 Bad Request   - 입력 형식 오류, 유효성 검증 실패
  401 Unauthorized  - 인증 정보 없음 또는 유효하지 않음
  403 Forbidden     - 인증은 됐지만 권한 없음
  404 Not Found     - 리소스 없음
  409 Conflict      - 이미 존재하거나 상태 충돌 (잔액 부족, 중복 요청)
  422 Unprocessable Entity - 형식은 맞지만 비즈니스 규칙 위반
  429 Too Many Requests    - Rate Limit 초과

5xx: 서버 오류 (재시도 가능할 수 있음)
  500 Internal Server Error - 서버 내부 오류
  502 Bad Gateway           - 업스트림 서비스 오류
  503 Service Unavailable   - 서비스 일시 중단
  504 Gateway Timeout       - 업스트림 타임아웃
```

결제 시스템에서 `409 Conflict`는 특히 중요하다. 잔액 부족, 이미 처리된 멱등성 키, 지갑 상태 이상 같은 비즈니스 충돌을 나타낼 때 사용한다. 클라이언트는 `409`를 받으면 재시도하지 않고 사용자에게 원인을 안내해야 한다. 반면 `503`은 재시도 가능한 서버 오류이므로 멱등성 키와 함께 재시도 로직을 설계한다.

`500`과 `503`의 차이도 중요하다. `500`은 서버 로직 버그처럼 재시도해도 해결되지 않을 수 있는 오류고, `503`은 서비스가 일시적으로 과부하 상태임을 나타내어 재시도가 도움이 된다.

### 내부 API 계약의 구성 요소

외부 API와 달리 내부 API 계약은 종종 문서화되지 않는다. 하지만 MSA에서는 각 서비스가 독립적으로 배포되므로 내부 API가 더 엄격한 계약 관리를 필요로 한다.

내부 API 계약이 포함해야 할 항목은 다음과 같다.

```text
1. 요청 스키마
   - 필수 필드, 선택 필드
   - 각 필드의 타입, 형식, 유효 범위
   - 헤더 요구사항 (X-User-Id, Idempotency-Key 등)

2. 응답 스키마
   - 성공 응답 구조
   - 에러 응답 구조 (에러 코드, 메시지 형식)
   - 각 상태 코드의 의미와 클라이언트 대응 방법

3. 동작 특성
   - 멱등성 여부
   - 재시도 안전 여부
   - 예상 응답 시간 (SLO)
   - 타임아웃 권장값

4. 부작용
   - 어떤 상태 변경이 발생하는가
   - 어떤 이벤트나 Outbox 레코드가 생성되는가
   - 다른 서비스 호출이 포함되는가
```

### 에러 응답 형식 통일

서비스마다 에러 응답 형식이 다르면 호출하는 서비스가 각각 다른 파싱 로직을 만들어야 한다. 통일된 에러 응답 형식의 예시는 다음과 같다.

```json
{
  "code": "INSUFFICIENT_BALANCE",
  "message": "잔액이 부족합니다",
  "detail": {
    "required": 50000,
    "available": 30000
  },
  "traceId": "abc-123-xyz"
}
```

`code` 필드로 비즈니스 에러를 구분하면 클라이언트는 HTTP 상태 코드와 에러 코드를 조합해 정확한 처리를 할 수 있다. 특히 결제 시스템에서는 에러 코드에 따라 재시도, 사용자 안내, 운영팀 알림 중 어떤 경로를 택할지 결정한다.

에러 응답에 내부 스택 트레이스나 SQL 쿼리를 노출하면 보안 위험이 생긴다. 에러 메시지는 클라이언트가 이해할 수 있는 수준으로만 제공하고, 내부 원인은 `traceId`를 통해 서버 로그에서만 확인할 수 있도록 한다.

### 흔한 오해와 함정

REST를 쓰면 자동으로 계약이 명확해진다고 생각하는 것은 오해다. HTTP 메서드와 URI를 쓴다고 REST가 되는 것이 아니라, 일관된 의미론과 계약이 있어야 한다.

내부 API를 "내부니까 느슨하게 관리해도 된다"고 보는 것은 위험하다. MSA에서 각 서비스의 배포 시점은 다르다. `wallet-service v2`가 배포되는 순간 아직 구버전 응답을 기대하는 `transfer-service`가 장애를 일으킬 수 있다.

또한 POST를 멱등 연산에 사용하면 안 된다는 강제가 있는 것은 아니다. 실제로 많은 결제 시스템이 POST + Idempotency-Key 조합으로 멱등한 명령 API를 만든다. 중요한 것은 메서드 이름이 아니라 실제 계약이다.

### 성능과 trade-off 관점

동기 HTTP 호출은 호출하는 서비스가 응답을 기다리는 동안 스레드를 점유한다. 응답이 늦어지면 스레드 풀이 고갈될 수 있다.

- 타임아웃 없이 외부 서비스를 호출하면 하나의 느린 서비스가 전체 서비스 스레드를 점유할 수 있다.
- 재시도 없이 한 번 실패하면 데이터 처리가 누락될 수 있다.
- 재시도를 너무 많이 하면 장애 중인 서비스에 추가 부하를 가중시킨다.

결제 API에서는 멱등성 키 + 제한적 재시도 + 명확한 타임아웃이 함께 설계되어야 한다. 타임아웃은 너무 짧으면 일시적 지연을 실패로 오판하고, 너무 길면 스레드 풀 고갈을 유발한다. 통상적으로 내부 서비스 간 호출은 2~5초, 외부 서비스(카드사, 은행 등) 호출은 5~15초 범위에서 설정하고 운영 지표를 보며 조정한다.

## PayFlow 연결

내부 API 계약은 단순한 URL 목록이 아니다. 어떤 요청 헤더가 필요한지, 어떤 상태 코드가 반환되는지, 실패 응답 형식이 무엇인지, 같은 요청을 재시도해도 안전한지까지 포함한다.

송금 요청 API라면 `Idempotency-Key`, 사용자 식별 정보, 금액, 송신 지갑, 수신 지갑, 실패 응답 코드가 명확해야 한다. 내부 서비스가 이 계약을 다르게 해석하면 중복 차감, 권한 우회, 상태 불일치가 생길 수 있다.

## 실무 포인트

- 명령 API와 조회 API를 구분한다.
- POST 요청은 멱등하지 않을 수 있으므로 멱등성 키를 명시한다.
- 내부 API도 외부 API처럼 계약을 문서화한다.
- 에러 응답 형식을 서비스마다 통일한다.
- 타임아웃과 재시도 가능 여부를 API 계약에 포함한다.

## 체크 질문

- HTTP 상태 코드 `400`, `401`, `403`, `409`, `500`은 각각 어떤 상황에 적합한가
- 송금 API에 `Idempotency-Key`가 필요한 이유는 무엇인가
- 내부 API 계약이 깨지면 PayFlow에서 어떤 문제가 생길 수 있는가

## 실무 설계 보강

### 대표 장애 시나리오

wallet-service가 잔액 부족 응답 형식을 바꾸자 transfer-service가 이를 일반 장애로 처리한다.

### 잘못된 구현 예시

~~~text
내부 API니까 문서화가 필요 없다고 보고 DTO와 에러 코드를 임의로 바꾼다.
~~~

### 좋은 구현 예시

~~~text
요청/응답 스키마, 에러 코드, 재시도 가능 여부, 멱등성 요구사항을 계약으로 관리한다.
~~~

### 대안과 선택 이유

문서 없이 코드만 맞춰가는 방식도 가능하지만, 서비스가 나뉘면 배포 시점이 달라져 작은 응답 변경도 장애가 된다. PayFlow에서는 API 계약과 버전 전략을 명확히 두는 편이 내부 서비스 간 변경 비용을 줄인다.

### PayFlow에서 확인할 위치

api-gateway 라우팅, transfer-service의 wallet 호출부, 각 service의 application.yml

### 면접에서 설명하기

REST API는 URL이 아니라 계약이다. 특히 내부 API도 배포 시점이 다르므로 호환성과 실패 의미를 설명해야 한다.

이 구간의 핵심은 API가 단순한 입구가 아니라 시스템의 계약이자 보안 경계라는 점이다. 누가 호출할 수 있는지, 어떤 입력을 믿을 수 있는지, 실패를 어떤 형태로 돌려주는지가 서비스 전체의 안정성을 결정한다.

### 질문의 질문

아래 질문들은 답을 외우기 위한 질문이 아니다. 하나씩 답하면서 "왜 그런 설계가 필요한지"를 설명하는 연습을 하기 위한 질문이다.

- HTTP REST And Internal API Contracts 개념이 없다면 PayFlow에서 가장 먼저 어떤 장애가 생기는가?

- 반대로 이 개념을 잘못 적용하면 어떤 복잡도가 추가되는가?
- 이 개념을 적용하면 어떤 문제가 해결되지만, 어떤 비용이나 복잡도가 새로 생기는가?
- 장애가 발생했을 때 이 개념은 문제를 예방하는가, 감지하는가, 복구하는가?
- PayFlow 코드에서 이 개념을 확인하려면 어떤 서비스, 테이블, 로그, 테스트를 봐야 하는가?

#### 핵심 답변

HTTP REST And Internal API Contracts 개념은 PayFlow에서 다음 이유로 중요하다.

- API와 보안 개념은 PayFlow에서 외부 요청이 내부 금전 명령으로 바뀌는 경계를 안전하게 만들기 위해 필요하다.
- Gateway가 인증 정보를 검증하지만, 실제 리소스 소유권의 진실은 각 도메인 서비스의 DB에 있다.
- 사용자가 보낸 userId나 walletId를 그대로 믿으면 다른 사람의 지갑에 접근하거나 금액을 조작할 수 있다.
- JWT 검증, 소유권 확인, 입력 검증, 내부 헤더 재생성, 민감 정보 마스킹, Secret 분리로 방어한다.
- 운영에서는 인증 실패율, 인가 실패율, 4xx 비율, 비정상 요청 패턴, 민감 정보 로그 노출 여부를 확인한다.

#### 질문에 답하는 방식

좋은 답변은 용어 정의에서 멈추지 않고 다음 순서로 이어져야 한다.

1. 어떤 데이터나 요청을 보호하려는 개념인지 말한다.
2. 그 데이터의 진실의 원천이 어느 서비스에 있는지 말한다.
3. 장애, 중복, 동시성, 지연 중 어떤 상황에서 문제가 생기는지 설명한다.
4. 코드 레벨의 방어 수단과 운영 레벨의 확인 수단을 함께 말한다.

1. 이 개념이 가장 직접적으로 연결되는 PayFlow 서비스 하나를 고른다.
2. 그 서비스가 소유한 데이터의 "진실의 원천"이 무엇인지 적어본다.
3. 동시에 두 요청이 들어오거나, 네트워크가 끊기거나, 프로세스가 죽는 상황을 가정한다.
4. 그 상황에서 데이터가 깨지지 않으려면 어떤 제약조건, 트랜잭션, 락, 이벤트, 재처리 장치가 필요한지 설명한다.
5. 마지막으로 운영자가 문제를 발견할 수 있는 로그나 지표가 무엇인지 적어본다.
