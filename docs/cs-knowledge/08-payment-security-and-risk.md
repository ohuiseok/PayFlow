# Payment Security And Risk

## 핵심 개념

결제 보안은 단순히 로그인 기능을 붙이는 것이 아니라, 돈이 잘못 이동하거나 악의적으로 조작되지 않도록 전체 흐름을 보호하는 것이다.

결제 시스템은 중복 요청, 위조 요청, 권한 우회, 금액 변조, 재시도 악용, 내부 API 노출, 로그를 통한 개인정보 유출 같은 위험을 고려해야 한다.

### 결제 보안의 위협 모델

결제 시스템에서 보안 위협은 외부 공격자뿐만 아니라 내부 실수, 로직 버그, 구성 오류에서도 온다. 위협을 체계적으로 생각하는 방법이 STRIDE 모델이다.

```text
Spoofing (사칭)
  - 다른 사용자인 척하는 요청
  - 방어: JWT 검증, X-User-Id 헤더 Gateway 재생성

Tampering (변조)
  - 요청 바디의 금액, 지갑 ID 변조
  - 방어: HTTPS, 서버 측 검증, DB 제약 조건

Repudiation (부인)
  - "나는 그 송금을 요청한 적 없다"
  - 방어: 감사 로그, 서명된 요청 기록

Information Disclosure (정보 유출)
  - 에러 응답에 내부 정보 포함, 로그 유출
  - 방어: 에러 메시지 최소화, 로그 마스킹

Denial of Service (서비스 거부)
  - 과도한 요청으로 서비스 마비
  - 방어: Rate Limit, Circuit Breaker

Elevation of Privilege (권한 상승)
  - 일반 사용자가 관리자 API 접근
  - 방어: RBAC, 소유권 검증, 엔드포인트 분리
```

### 금액 검증의 계층 구조

금액 검증은 여러 계층에서 중복으로 이루어져야 한다. 한 계층이 뚫렸을 때 다른 계층이 막아주는 Defense in Depth 원칙이다.

```text
계층 1: 입력 검증 (Controller/DTO 레벨)
  - 금액 > 0
  - 최소 금액 이상 (예: 1원 이상)
  - 최대 금액 이하 (예: 1회 1억원 이하)
  - 소수점 자릿수 제한
  - null, 빈 문자열 불허

계층 2: 비즈니스 검증 (Service 레벨)
  - 잔액 >= 요청 금액
  - 지갑 상태 ACTIVE 확인
  - 일일/월별 한도 확인
  - 수신 지갑 존재 및 상태 확인

계층 3: DB 제약 조건 (Database 레벨)
  - balance CHECK (balance >= 0)
  - amount NOT NULL
  - amount 타입 DECIMAL(19, 4)
```

특히 잔액 음수 방지는 DB 제약 조건(`CHECK balance >= 0`)으로 최종 방어선을 반드시 구성해야 한다. 서비스 코드에서 버그가 생겨도 DB가 막아준다.

### 멱등성과 보안의 연결

멱등성은 단순히 편의 기능이 아니라 보안과 직결된다.

```text
멱등성이 없는 경우:
  공격자가 같은 송금 요청을 10번 재시도하면
  -> 10번 모두 처리될 수 있음
  -> 잔액에서 금액이 10배 빠져나감

멱등성이 있는 경우:
  같은 Idempotency-Key로 10번 요청해도
  -> 서버가 첫 번째 처리 결과를 캐시
  -> 이후 9번은 같은 결과를 반환하고 중복 처리 없음
```

Idempotency-Key는 클라이언트가 생성하는 UUID여야 하며, 서버는 이 키와 처리 결과를 Redis나 DB에 저장한다. 키의 만료 시간은 클라이언트의 재시도 기간보다 충분히 길어야 한다.

### 내부 API 보호

내부 출금/입금 API가 외부에서 직접 호출 가능하면 매우 위험하다. 내부 API 보호 방법은 다음과 같다.

```text
방법 1: 네트워크 레벨 격리
  내부 API는 내부 Docker 네트워크에서만 접근 가능
  외부에서는 API Gateway를 통해서만 접근 가능
  -> gateway -> wallet-service (내부 네트워크)
  -> 외부 -> wallet-service (직접 접근 불가)

방법 2: 서비스 식별자 헤더
  내부 서비스 호출 시 X-Internal-Service: transfer-service 헤더 포함
  wallet-service가 내부 API에서 이 헤더 검증
  Gateway는 외부 요청에서 이 헤더를 제거

방법 3: 엔드포인트 분리
  /internal/wallets/{id}/debit  <- 내부 서비스용
  /api/wallets/{id}             <- 외부 사용자용
  Gateway는 /internal/** 경로를 라우팅하지 않음
```

### 감사 로그와 추적 가능성

결제 시스템에서 모든 금전 변경은 추적 가능해야 한다. 감사 로그는 장애 분석뿐만 아니라 분쟁 해결과 규정 준수(컴플라이언스)를 위해서도 필요하다.

```text
감사 로그에 포함해야 할 정보:
  - 요청 시각 (밀리초 단위)
  - 요청자 ID (userId)
  - 요청 IP (프록시 경유 시 X-Forwarded-For)
  - 요청 ID (traceId)
  - 작업 종류 (TRANSFER, DEBIT, CREDIT)
  - 대상 리소스 (walletId, transferId)
  - 변경 전/후 값
  - 처리 결과 (SUCCESS, FAILED)
  - 실패 사유 (실패한 경우)

감사 로그에 포함하면 안 되는 정보:
  - 비밀번호
  - JWT 토큰 전체
  - 카드 번호 전체
  - 기타 민감 개인정보
```

### 흔한 오해와 함정

프론트엔드에서 금액 검증을 했으니 서버에서 다시 할 필요 없다는 생각은 위험하다. 클라이언트 검증은 언제든 우회될 수 있다. 서버에서 모든 검증을 독립적으로 수행해야 한다.

HTTPS가 있으면 모든 통신이 안전하다는 오해도 있다. HTTPS는 전송 구간을 암호화할 뿐이다. 서버에 도달한 요청의 내용을 검증하고, 권한을 확인하고, 중복을 거르는 것은 별도의 로직이 필요하다.

### trade-off 관점

보안을 강화할수록 개발 복잡도와 처리 비용이 올라간다. 금액 검증, 소유권 확인, 멱등성 처리, 감사 로그, Rate Limit이 모두 추가될수록 API 응답 시간도 늘어날 수 있다. 하지만 결제 시스템에서는 이 비용이 정당하다. 돈이 잘못 이동하는 것을 막는 비용이, 잘못 이동한 후 복구하는 비용보다 훨씬 작기 때문이다.

## PayFlow 연결

PayFlow에서 가장 위험한 지점은 송금 요청과 지갑 잔액 변경이다. 같은 요청이 두 번 처리되거나, 다른 사용자의 지갑을 지정하거나, 금액이 음수로 들어오거나, 내부 출금 API가 외부에 노출되면 치명적인 문제가 된다.

따라서 인증, 인가, 입력 검증, 멱등성, 감사 로그, 내부 API 보호가 함께 필요하다.

## 실무 포인트

- 금액은 음수, 0, 최대 한도를 검증한다.
- 송신 지갑 소유자를 반드시 검증한다.
- 같은 요청이 여러 번 들어와도 한 번만 처리한다.
- 내부 API는 외부에서 직접 호출할 수 없어야 한다.
- 민감 정보는 로그에 남기지 않는다.
- 실패한 결제 흐름은 추적 가능해야 한다.

## 체크 질문

- 결제 시스템에서 멱등성이 보안과도 연결되는 이유는 무엇인가
- 내부 출금 API가 외부에 노출되면 어떤 문제가 생기는가
- 금액 검증에서 반드시 막아야 할 입력은 무엇인가

## 실무 설계 참고

### 대표 장애 시나리오

공격자가 음수 금액이나 매우 큰 금액을 보내 잔액 검증 로직을 우회하려 한다.

### 잘못된 구현 예시

~~~text
프론트엔드 입력 검증만 믿고 서버에서 금액과 소유권을 검증하지 않는다.
~~~

### 좋은 구현 예시

~~~text
서버에서 금액 범위, 지갑 소유권, 멱등성 키, 내부 API 접근을 모두 검증한다.
~~~

### 대안과 선택 이유

Gateway에서 한 번만 보안을 처리하는 방식도 있지만, 결제 시스템에서는 단일 방어선이 뚫렸을 때 피해가 크다. PayFlow는 Gateway 인증, 서비스 내부 인가, 입력 검증, 로그 마스킹, Secret 분리를 겹겹이 두는 방식이 더 안전하다.

### PayFlow에서 확인할 위치

transfer-service 송금 요청 검증, wallet-service 출금/입금 API, audit log

### 면접에서 설명하기

결제 보안은 로그인 기능이 아니라 돈이 잘못 움직일 수 있는 모든 입력과 경계를 막는 것이다.

### 관련 문서

07, 09, 25, 40, 45

## 질문의 질문으로 더 깊게 보기

### 이 문서에서 잡아야 할 CS 감각

이 구간의 핵심은 API가 단순한 입구가 아니라 시스템의 계약이자 보안 경계라는 점이다. 누가 호출할 수 있는지, 어떤 입력을 믿을 수 있는지, 실패를 어떤 형태로 돌려주는지가 서비스 전체의 안정성을 결정한다.

### 질문의 질문

아래 질문들은 답을 외우기 위한 질문이 아니다. 하나씩 답하면서 "왜 그런 설계가 필요한지"를 설명하는 연습을 하기 위한 질문이다.

- 이 요청에서 서버가 믿어도 되는 값과 믿으면 안 되는 값은 무엇인가?
- 인증된 사용자라는 사실만으로 이 리소스에 접근해도 되는가?
- 로그, 에러 응답, 헤더 중 어디로 민감 정보가 새어나갈 수 있는가?
- 이 개념을 적용하면 어떤 문제가 해결되지만, 어떤 비용이나 복잡도가 새로 생기는가?
- 장애가 발생했을 때 이 개념은 문제를 예방하는가, 감지하는가, 복구하는가?
- PayFlow 코드에서 이 개념을 확인하려면 어떤 서비스, 테이블, 로그, 테스트를 봐야 하는가?

#### 핵심 답변

Payment Security And Risk 개념은 PayFlow에서 다음 이유로 중요하다.

- API와 보안 개념은 PayFlow에서 외부 요청이 내부 금전 명령으로 바뀌는 경계를 안전하게 만들기 위해 필요하다.
- Gateway가 인증 정보를 검증하지만, 실제 리소스 소유권의 진실은 각 도메인 서비스의 DB에 있다.
- 사용자가 보낸 userId나 walletId를 그대로 믿으면 다른 사람의 지갑에 접근하거나 금액을 조작할 수 있다.
- JWT 검증, 소유권 확인, 입력 검증, 내부 헤더 재생성, 민감 정보 마스킹, Secret 분리로 방어한다.
- 운영에서는 인증 실패율, 인가 실패율, 4xx 비율, 비정상 요청 패턴, 민감 정보 로그 노출 여부를 확인한다.

#### 질문에 답하는 방식

좋은 답변은 용어 정의에서 멈추지 않고 다음 순서로 이어져야 한다.

1. 어떤 데이터나 요청을 보호하려는 개념인지 말한다.
2. 그 데이터의 진실의 원천이 어느 서비스에 있는지 말한다.
3. 장애, 중복, 동시성, 지연 중 어떤 상황에서 문제가 생기는지 설명한다.
4. 코드 레벨의 방어 수단과 운영 레벨의 확인 수단을 함께 말한다.

Payment Security And Risk 개념은 PayFlow에서 금액 변조, 중복 송금, 내부 API 오남용 같은 결제 특유의 위험을 줄이기 위해 필요하다.
이 개념이 없으면 음수 금액, 과도한 금액, 다른 사용자 지갑 지정, 내부 출금 API 직접 호출이 금전 사고로 이어질 수 있다.

운영에서는 비정상 금액 요청, 중복 요청, 인가 실패, 내부 API 접근 시도를 확인해야 한다.

### PayFlow에 대입해보기

1. 이 개념이 가장 직접적으로 연결되는 PayFlow 서비스 하나를 고른다.
2. 그 서비스가 소유한 데이터의 "진실의 원천"이 무엇인지 적어본다.
3. 동시에 두 요청이 들어오거나, 네트워크가 끊기거나, 프로세스가 죽는 상황을 가정한다.
4. 그 상황에서 데이터가 깨지지 않으려면 어떤 제약조건, 트랜잭션, 락, 이벤트, 재처리 장치가 필요한지 설명한다.
5. 마지막으로 운영자가 문제를 발견할 수 있는 로그나 지표가 무엇인지 적어본다.
