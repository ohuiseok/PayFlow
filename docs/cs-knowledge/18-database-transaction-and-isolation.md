# Database Transaction And Isolation

> 도메인 전환 안내: 현재 PayFlow는 **청년 정책 참여 미션 및 지원금 지급 플랫폼**으로 설명한다. 내부 구현 호환성을 위해 `PARENT`/`CHILD`, `/api/families`, `/api/missions`, `/api/cashbook`, `reward-service` 같은 명칭은 유지하지만, 문서와 발표에서는 각각 **기관 담당자**, **청년 참여자**, **참여자 연결**, **정책 미션**, **지원금 사용 내역**, **정책 미션/지원금 서비스**로 해석한다.

## 핵심 개념

### 트랜잭션이란 무엇인가

DB 트랜잭션은 여러 작업을 하나의 논리적 단위로 묶는 기능이다. 트랜잭션은 ACID 특성을 가진다.

- **Atomicity(원자성)**: 트랜잭션 내의 모든 작업은 전부 성공하거나 전부 실패한다. 중간 상태는 없다. 커밋 전에 장애가 나면 Redo/Undo Log를 이용해 시작 시점으로 롤백된다.
- **Consistency(일관성)**: 트랜잭션이 완료되면 DB는 정의된 제약조건(NOT NULL, UNIQUE, FK, CHECK 등)을 모두 만족하는 상태가 된다. 트랜잭션 중에는 일시적으로 제약을 위반할 수 있지만 커밋 시점에 검증한다.
- **Isolation(격리성)**: 동시에 실행되는 트랜잭션은 서로의 중간 상태를 보지 않는다. 완전한 격리는 성능 비용이 크므로 격리 수준을 선택할 수 있다.
- **Durability(지속성)**: 커밋된 트랜잭션의 결과는 시스템 장애가 발생해도 보존된다. InnoDB에서는 Redo Log(WAL, Write-Ahead Logging)가 이를 보장한다.

### 격리 수준과 발생 가능한 문제

격리 수준은 동시에 실행되는 트랜잭션이 서로의 변경을 어느 정도 볼 수 있는지 정한다. 격리 수준이 낮을수록 성능은 좋아지지만 정합성 문제가 생길 수 있다.

**Dirty Read**: 아직 커밋되지 않은 다른 트랜잭션의 변경 값을 읽는다. T1이 잔액을 1,000으로 변경했지만 아직 커밋하지 않았는데, T2가 그 1,000을 읽고 처리한 뒤 T1이 롤백되면 T2는 존재하지 않는 잔액을 기준으로 처리한 셈이 된다.

**Non-Repeatable Read**: 같은 트랜잭션 안에서 같은 행을 두 번 읽었을 때 값이 달라지는 현상이다. T1이 잔액을 읽고, T2가 그 잔액을 변경 후 커밋하면, T1이 다시 읽으면 값이 바뀐다. 잔액 확인 후 출금 처리 사이에 이 문제가 생기면 초과 출금이 발생할 수 있다.

**Phantom Read**: 같은 트랜잭션 안에서 같은 범위 조건으로 두 번 조회했을 때 결과 행 수가 달라지는 현상이다. T1이 특정 조건 조회 후 T2가 해당 범위에 새 행을 삽입·커밋하면 T1의 두 번째 조회에서 새 행이 보인다.

| 격리 수준 | Dirty Read | Non-Repeatable Read | Phantom Read |
|---|---|---|---|
| READ UNCOMMITTED | 발생 | 발생 | 발생 |
| READ COMMITTED | 방지 | 발생 | 발생 |
| REPEATABLE READ | 방지 | 방지 | 발생(InnoDB는 Gap Lock으로 방지) |
| SERIALIZABLE | 방지 | 방지 | 방지 |

MySQL InnoDB의 기본 격리 수준은 `REPEATABLE READ`다. InnoDB는 MVCC(Multi-Version Concurrency Control)로 Non-Repeatable Read를 방지하고, Gap Lock과 Next-Key Lock으로 Phantom Read도 실질적으로 방지한다.

### MVCC: 읽기와 쓰기가 서로 막히지 않는 이유

InnoDB는 MVCC를 사용해 읽기 트랜잭션이 쓰기 트랜잭션을 막지 않도록 한다. 모든 변경은 Undo Log에 이전 버전을 남기고, 읽기 트랜잭션은 트랜잭션 시작 시점의 스냅샷을 본다.

```text
T1 시작 (snapshot: t=100)
  T2: balance 10000 -> 9000 변경, 커밋
T1: balance 조회 -> 여전히 10000 (스냅샷 기준)
T1 종료
```

이를 Consistent Read 또는 Snapshot Read라고 한다. 단, `SELECT ... FOR UPDATE`나 `SELECT ... LOCK IN SHARE MODE`는 현재 커밋된 최신 값을 읽는 Locking Read가 된다.

### 트랜잭션 격리 수준과 락의 관계

격리 수준만으로 모든 동시성 문제가 해결되지는 않는다.

REPEATABLE READ에서 `SELECT`는 스냅샷을 보지만, `SELECT ... FOR UPDATE`는 현재 행에 배타 락을 잡는다. 잔액 차감처럼 "읽고 조건 확인 후 변경"을 안전하게 하려면 스냅샷 읽기가 아닌 락 읽기가 필요하다.

```sql
-- 안전하지 않은 방식 (스냅샷 읽기)
SELECT balance FROM wallet WHERE id = 1;
-- 다른 트랜잭션이 이미 차감했을 수 있음
UPDATE wallet SET balance = balance - 5000 WHERE id = 1;

-- 안전한 방식 (락 읽기)
SELECT balance FROM wallet WHERE id = 1 FOR UPDATE;
-- 이 시점에 행 락이 잡혀 다른 트랜잭션의 변경을 막음
UPDATE wallet SET balance = balance - 5000 WHERE id = 1 AND balance >= 5000;
```

### 트랜잭션 범위 설계 시 주의점

트랜잭션 범위를 잘못 설계하면 여러 문제가 생긴다.

**트랜잭션이 너무 길 때**: 락 보유 시간이 길어져 다른 트랜잭션의 대기 시간이 늘어난다. Undo Log가 많이 쌓여 스토리지를 압박한다. 데드락 가능성이 높아진다.

**트랜잭션이 너무 짧을 때**: 원자적으로 묶어야 할 작업이 나뉘면 중간 실패 시 데이터 불일치가 생긴다. 잔액 차감과 이력 저장이 별도 트랜잭션이면 이력 없는 차감이 발생할 수 있다.

**외부 I/O를 트랜잭션 안에 두면**: HTTP 호출이나 파일 I/O를 트랜잭션 안에서 실행하면 트랜잭션이 길어지고 DB 커넥션 점유 시간도 늘어난다. 외부 호출 응답 대기 중 다른 트랜잭션들이 해당 행의 락을 기다린다.

### 흔한 오해와 함정

**"격리 수준이 높으면 무조건 안전하다"**: SERIALIZABLE은 가장 안전하지만 동시 처리량이 크게 줄어든다. 결제 시스템에서는 REPEATABLE READ + 명시적 락이 성능과 안전성의 균형점인 경우가 많다.

**"롤백하면 모든 것이 원상복구된다"**: DB 안의 데이터는 롤백되지만, 이미 발행한 이벤트, 외부 API 호출 결과, 캐시 상태는 롤백되지 않는다. Outbox Pattern이 필요한 이유가 여기 있다.

**"트랜잭션 어노테이션만 붙이면 된다"**: Spring `@Transactional`은 프록시 기반이라 같은 클래스 내 메서드 호출(self-invocation)에는 적용되지 않는다.


## PayFlow 연결

`wallet-service`에서 잔액 차감은 반드시 하나의 트랜잭션 안에서 처리되어야 한다.

```text
1. 지갑 조회
2. 잔액 부족 검증
3. 잔액 차감
4. 잔액 변경 이력 저장
5. 커밋
```

중간에 실패하면 잔액만 차감되고 이력이 없는 상태가 되면 안 된다.

## 실무 포인트

- 트랜잭션 범위를 너무 길게 잡지 않는다.
- 외부 HTTP 호출을 트랜잭션 안에서 오래 수행하지 않는다.
- 잔액 변경은 동시성까지 함께 고려한다.
- 격리 수준만으로 모든 동시성 문제가 해결되지는 않는다.
- DB 제약조건과 애플리케이션 검증을 함께 사용한다.

## 체크 질문

- 트랜잭션의 ACID는 각각 무엇인가
- 잔액 차감과 잔액 이력 저장을 같은 트랜잭션으로 묶어야 하는 이유는 무엇인가
- 트랜잭션 시간이 길어지면 어떤 문제가 생기는가

## 실무 설계 참고

### 대표 장애 시나리오

출금 트랜잭션 중간에 장애가 나서 잔액과 변경 이력이 서로 맞지 않는다.

### 잘못된 구현 예시

~~~text
잔액 update와 이력 insert를 따로 처리하거나 트랜잭션 밖에서 처리한다.
~~~

### 좋은 구현 예시

~~~text
잔액 검증, 차감, 변경 이력 저장을 하나의 트랜잭션으로 묶고 실패 시 전체 롤백한다.
~~~

### 대안과 선택 이유

애플리케이션 코드에서만 검증하는 방식도 있지만, 동시 요청과 장애 상황에서는 코드 검증만으로 부족하다. PayFlow는 DB 트랜잭션, 락, 유니크 제약, 인덱스, 실행 계획을 함께 사용해 데이터 계층에서도 불변식을 지키는 쪽이 더 안전하다.

### PayFlow에서 확인할 위치

wallet-service 잔액 변경 로직, wallet history table, transaction annotation

### 면접에서 설명하기

트랜잭션은 코드를 예쁘게 묶는 도구가 아니라 데이터 불변식을 지키는 경계다.

Database Transaction And Isolation 개념은 PayFlow에서 다음 이유로 중요하다.
