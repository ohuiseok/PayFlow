# Personal Data Protection And Masking

> 도메인 전환 안내: 현재 PayFlow는 **청년 정책 참여 미션 및 지원금 지급 플랫폼**으로 설명한다. 내부 구현 호환성을 위해 `PARENT`/`CHILD`, `/api/families`, `/api/missions`, `/api/cashbook`, `reward-service` 같은 명칭은 유지하지만, 문서와 발표에서는 각각 **기관 담당자**, **청년 참여자**, **참여자 연결**, **정책 미션**, **지원금 사용 내역**, **정책 미션/지원금 서비스**로 해석한다.

## 핵심 개념

개인정보 보호는 사용자 식별 정보, 연락처, 계정 정보, 결제 관련 민감 데이터를 안전하게 저장하고 노출을 최소화하는 것이다. 마스킹은 로그나 화면에 민감 정보를 그대로 보여주지 않도록 일부를 가리는 방식이다.

### 민감 정보의 종류와 분류

결제 시스템에서 민감 정보는 다음과 같이 분류할 수 있다.

```text
직접 식별 정보 (PII - Personally Identifiable Information)
  - 이름, 이메일, 전화번호
  - 주민등록번호, 여권번호
  - 주소

금융 정보
  - 계좌번호, 카드번호
  - CVC/CVV, 카드 만료일
  - 거래 금액, 잔액 (특정 맥락에서 민감)

인증 정보
  - 비밀번호, PIN
  - JWT 토큰
  - API 키, 세션 ID

간접 식별 정보
  - IP 주소 (개인과 연결 가능)
  - 기기 정보 (개인 특정 가능)
  - 행동 패턴 (거래 이력 조합)
```

GDPR(유럽), 개인정보보호법(한국) 등 규정마다 민감 정보의 범위와 처리 요건이 다르다. 결제 시스템은 금융 정보를 다루므로 일반 서비스보다 엄격한 요건이 적용된다.

### 저장 단계의 보호

**비밀번호 해싱**

비밀번호는 절대 평문으로 저장하면 안 된다. 해시 함수를 사용하되 단순한 MD5, SHA-1은 레인보우 테이블 공격에 취약하다. 비밀번호 해싱에는 Salt가 추가된 느린 해시 함수를 사용해야 한다.

```java
// 올바른 비밀번호 저장 (BCrypt 예시)
String hashed = BCrypt.hashpw(password, BCrypt.gensalt(12));

// 검증
boolean matches = BCrypt.checkpw(inputPassword, hashed);
```

BCrypt, Argon2, PBKDF2가 권장된다. 이들은 의도적으로 느리게 설계되어 대량 무차별 대입 공격을 어렵게 만든다.

**암호화와 마스킹의 차이**

```text
암호화 (Encryption):
  - 원본 값을 복호화할 수 있음
  - 카드번호처럼 나중에 조회가 필요한 경우
  - AES-256 같은 대칭키 암호화 사용

해싱 (Hashing):
  - 단방향, 복호화 불가
  - 비밀번호처럼 원본이 필요 없는 경우
  - 검증은 동일 입력 -> 동일 해시로 확인

마스킹 (Masking):
  - 화면/로그 표시 시 일부를 가림
  - 저장 값을 바꾸지 않음
  - 예: 카드번호 **** **** **** 1234
```

### 로그에서의 민감 정보 관리

로그는 개발과 운영에 필수적이지만 민감 정보 유출의 주요 경로다. 다음 정보는 로그에 원문으로 남기면 안 된다.

```text
절대 로깅하면 안 되는 정보:
  - 비밀번호, PIN
  - JWT 토큰 전체
  - 카드번호 전체
  - CVC/CVV

주의해서 로깅해야 하는 정보:
  - 이메일 (마스킹 처리: ab***@example.com)
  - 전화번호 (마스킹 처리: 010-****-5678)
  - 계좌번호 (마스킹 처리: **** **** 1234)
  - 금액 (거래 추적 목적으로는 필요하지만 접근 권한 관리)

로그에 포함해야 하는 정보:
  - traceId (요청 추적)
  - userId (실패 분석)
  - 요청 시각, 응답 코드
  - 처리 결과 (SUCCESS/FAILED)
  - 실패 원인 (에러 코드)
```

Spring에서 로그 마스킹을 구현하는 방법은 다음과 같다.

```java
// Logback PatternLayout 커스텀 마스킹
public class MaskingPatternLayout extends PatternLayout {
    private static final Pattern EMAIL_PATTERN =
        Pattern.compile("([a-zA-Z0-9._%+-]{2})[a-zA-Z0-9._%+-]+(@[a-zA-Z0-9.-]+)");

    @Override
    public String doLayout(ILoggingEvent event) {
        String message = super.doLayout(event);
        return EMAIL_PATTERN.matcher(message).replaceAll("$1***$2");
    }
}
```

또는 요청/응답 로깅 필터에서 민감 필드를 제거하는 방법도 있다.

### 응답 DTO에서의 정보 최소화

API 응답에 불필요한 내부 정보를 포함하면 보안 위험이 생긴다.

```java
// 나쁜 예: Entity를 직접 반환
@GetMapping("/users/{id}")
public User getUser(@PathVariable Long id) {
    return userRepository.findById(id); // 비밀번호 해시, 내부 ID, 모든 필드 포함
}

// 좋은 예: 응답 DTO 분리
@GetMapping("/users/{id}")
public UserResponse getUser(@PathVariable Long id) {
    User user = userRepository.findById(id);
    return UserResponse.of(user); // 공개 가능한 필드만 포함
}

// UserResponse
public record UserResponse(
    String name,
    String maskedEmail,  // ab***@example.com
    String maskedPhone   // 010-****-5678
) {
    public static UserResponse of(User user) {
        return new UserResponse(
            user.getName(),
            maskEmail(user.getEmail()),
            maskPhone(user.getPhone())
        );
    }
}
```

### 데이터 생명주기 관리

개인정보는 수집 목적이 완료되면 삭제하거나 익명화해야 한다. 결제 시스템에서는 규정(전자금융거래법 등)에 따라 거래 기록 보관 의무가 있지만, 개인 식별 정보는 의무 기간 후 삭제하거나 익명화해야 한다.

```text
익명화 예:
  실명 거래 이력: userId=123, amount=50000
  익명화 후:      userId=DELETED, amount=50000  <- 통계 목적으로 보관 가능

가명화 예:
  실명: userId=123 -> pseudoId=uuid-xyz (역추적 불가한 식별자)
  특정 조건에서 역추적 가능하지만 일반적으로는 개인 특정 불가
```

### 접근 제어와 감사 추적

민감 데이터에 대한 접근 자체를 최소화하고 감사해야 한다.

```text
원칙 of Least Privilege:
  - 개발자가 운영 DB에 직접 접근하지 못하도록
  - 쿼리 권한을 읽기 전용으로 제한
  - 민감 테이블 접근 시 별도 승인 프로세스

DB 접근 감사:
  - 어떤 사용자가 언제 어떤 쿼리를 실행했는지 기록
  - 대량 조회(SELECT *) 패턴 감지
  - 근무 시간 외 접근 알림
```

### 흔한 오해와 함정

"내부 시스템이니 암호화가 필요 없다"는 생각은 위험하다. 내부 시스템이 침해되거나 내부자 위협이 발생할 수 있다. 특히 민감 데이터는 저장 단계부터 암호화하는 것이 좋다.

로그를 개발팀만 볼 수 있으니 괜찮다는 생각도 위험하다. 로그 시스템이 침해되거나, 개발팀 계정이 탈취되거나, 로그가 외부 시스템으로 전송되는 경우 민감 정보가 노출될 수 있다. 로그에는 처음부터 민감 정보를 남기지 않는 것이 원칙이다.

## PayFlow 연결

PayFlow는 사용자, 지갑, 송금, 원장 데이터를 다룬다. 이 중 사용자 식별자, 이메일, 전화번호, 계좌와 유사한 정보, 거래 내역은 민감하게 다뤄야 한다.

특히 로그에 요청 바디 전체를 찍으면 JWT, 개인정보, 금액 정보가 그대로 남을 수 있다. 장애 분석을 위해 로그는 필요하지만, 필요한 정보만 남겨야 한다.

## 실무 포인트

- 비밀번호는 해시로 저장한다.
- 민감 정보는 로그에 원문으로 남기지 않는다.
- 응답 DTO에서 내부 필드를 제외한다.
- 운영 DB 접근 권한을 최소화한다.
- 감사 목적의 거래 로그와 개인정보 로그를 구분한다.

## 체크 질문

- 마스킹과 암호화의 차이는 무엇인가
- 결제 시스템 로그에 남기면 위험한 정보는 무엇인가
- 장애 분석과 개인정보 보호를 어떻게 함께 만족시킬 수 있는가

## 실무 설계 참고

### 대표 장애 시나리오

장애 분석을 위해 request body 전체를 로그로 남겼다가 개인정보와 거래 정보가 노출된다.

### 잘못된 구현 예시

~~~text
운영 편의를 위해 민감 데이터를 원문 로그에 남긴다.
~~~

### 좋은 구현 예시

~~~text
로그에는 식별 가능한 최소 정보만 남기고 이메일, 토큰, 개인정보, 거래 상세는 마스킹한다.
~~~

### 대안과 선택 이유

Gateway에서 한 번만 보안을 처리하는 방식도 있지만, 결제 시스템에서는 단일 방어선이 뚫렸을 때 피해가 크다. PayFlow는 Gateway 인증, 서비스 내부 인가, 입력 검증, 로그 마스킹, Secret 분리를 겹겹이 두는 방식이 더 안전하다.

### PayFlow에서 확인할 위치

GlobalExceptionHandler, logging 설정, user-service 응답 DTO, audit log

### 면접에서 설명하기

개인정보 보호는 저장 암호화만이 아니다. 로그, 에러 응답, 운영 도구까지 노출 경로로 봐야 한다.

## 질문의 질문으로 더 깊게 보기

### 이 문서에서 잡아야 할 CS 감각

이 구간의 핵심은 API가 단순한 입구가 아니라 시스템의 계약이자 보안 경계라는 점이다. 누가 호출할 수 있는지, 어떤 입력을 믿을 수 있는지, 실패를 어떤 형태로 돌려주는지가 서비스 전체의 안정성을 결정한다.

### 질문의 질문

아래 질문들은 답을 외우기 위한 질문이 아니다. 하나씩 답하면서 "왜 그런 설계가 필요한지"를 설명하는 연습을 하기 위한 질문이다.

- 이 요청에서 서버가 믿어도 되는 값과 믿으면 안 되는 값은 무엇인가?
- 인증된 사용자라는 사실만으로 이 리소스에 접근해도 되는가?
- 로그, 에러 응답, 헤더 중 어디로 민감 정보가 새어나갈 수 있는가?
- 이 개념을 적용하면 어떤 문제가 해결되지만, 어떤 비용이나 복잡도가 새로 생기는가?
- 장애가 발생했을 때 이 개념은 문제를 예방하는가, 감지하는가, 복구하는가?
- PayFlow 코드에서 이 개념을 확인하려면 어떤 서비스, 테이블, 로그, 테스트를 봐야 하는가?

#### 핵심 답변

Personal Data Protection And Masking 개념은 PayFlow에서 다음 이유로 중요하다.

- API와 보안 개념은 PayFlow에서 외부 요청이 내부 금전 명령으로 바뀌는 경계를 안전하게 만들기 위해 필요하다.
- Gateway가 인증 정보를 검증하지만, 실제 리소스 소유권의 진실은 각 도메인 서비스의 DB에 있다.
- 사용자가 보낸 userId나 walletId를 그대로 믿으면 다른 사람의 지갑에 접근하거나 금액을 조작할 수 있다.
- JWT 검증, 소유권 확인, 입력 검증, 내부 헤더 재생성, 민감 정보 마스킹, Secret 분리로 방어한다.
- 운영에서는 인증 실패율, 인가 실패율, 4xx 비율, 비정상 요청 패턴, 민감 정보 로그 노출 여부를 확인한다.

#### 질문에 답하는 방식

좋은 답변은 용어 정의에서 멈추지 않고 다음 순서로 이어져야 한다.

1. 어떤 데이터나 요청을 보호하려는 개념인지 말한다.
2. 그 데이터의 진실의 원천이 어느 서비스에 있는지 말한다.
3. 장애, 중복, 동시성, 지연 중 어떤 상황에서 문제가 생기는지 설명한다.
4. 코드 레벨의 방어 수단과 운영 레벨의 확인 수단을 함께 말한다.

이 개념이 없으면 장애 분석용 로그에 JWT, 이메일, 전화번호, 거래 상세가 그대로 남아 2차 유출이 발생할 수 있다.

### PayFlow에 대입해보기

1. 이 개념이 가장 직접적으로 연결되는 PayFlow 서비스 하나를 고른다.
2. 그 서비스가 소유한 데이터의 "진실의 원천"이 무엇인지 적어본다.
3. 동시에 두 요청이 들어오거나, 네트워크가 끊기거나, 프로세스가 죽는 상황을 가정한다.
4. 그 상황에서 데이터가 깨지지 않으려면 어떤 제약조건, 트랜잭션, 락, 이벤트, 재처리 장치가 필요한지 설명한다.
5. 마지막으로 운영자가 문제를 발견할 수 있는 로그나 지표가 무엇인지 적어본다.
