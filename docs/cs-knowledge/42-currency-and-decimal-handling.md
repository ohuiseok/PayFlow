# Currency And Decimal Handling

> 도메인 전환 안내: 현재 PayFlow는 **청년 정책 참여 미션 및 지원금 지급 플랫폼**으로 설명한다. 내부 구현 호환성을 위해 `PARENT`/`CHILD`, `/api/families`, `/api/missions`, `/api/cashbook`, `reward-service` 같은 명칭은 유지하지만, 문서와 발표에서는 각각 **기관 담당자**, **청년 참여자**, **참여자 연결**, **정책 미션**, **지원금 사용 내역**, **정책 미션/지원금 서비스**로 해석한다.

## 핵심 개념

통화마다 소수 자릿수와 최소 단위가 다르다. KRW는 보통 소수점 없이 원 단위로 다루지만, USD는 센트 단위가 있고, 일부 통화는 소수 자릿수가 다르다.

## PayFlow 연결

PayFlow가 원화 기반 간편송금이라면 금액을 원 단위 정수로 다루는 것이 단순하고 안전하다. 하지만 다중 통화를 지원하려면 `currency`, `amount`, `scale` 또는 최소 단위 기준을 명확히 해야 한다.

정산과 수수료 계산에서도 통화 단위가 섞이면 큰 문제가 생긴다.

## 실무 포인트

- 통화 코드를 명시적으로 저장한다.
- 서로 다른 통화를 단순 합산하지 않는다.
- 최소 단위를 기준으로 금액을 저장한다.
- 환율 적용 시 기준 시점과 반올림 정책을 저장한다.
- 화면 표시 금액과 저장 금액을 구분한다.

## 체크 질문

- KRW와 USD의 금액 처리 방식은 어떻게 다른가
- 통화 코드 없이 금액만 저장하면 어떤 문제가 생기는가
- 환율을 적용할 때 어떤 기준을 남겨야 하는가

## 실무 설계 참고

### 대표 장애 시나리오

KRW와 USD 금액을 통화 코드 없이 합산해 정산 리포트가 틀어진다.

### 잘못된 구현 예시

~~~text
금액 컬럼 하나만 있으면 모든 통화를 표현할 수 있다고 생각한다.
~~~

### 좋은 구현 예시

~~~text
amount와 currency를 함께 저장하고 통화별 소수 자릿수와 환율 기준을 기록한다.
~~~

### 대안과 선택 이유

PayFlow의 현재 구현 통화는 KRW이고 금액은 `DECIMAL(19,0)` 정수 단위다. 원장과 정산 결과는 남기지만 다통화 정책과 수수료 정책 버전은 아직 없다.

### PayFlow에서 확인할 위치

money value object, settlement aggregation, API DTO

### 면접에서 설명하기

통화는 금액의 단위다. 단위 없는 숫자는 결제 시스템에서 의미가 없다.

### 관련 문서

41, 43, 49

## 질문의 질문으로 더 깊게 보기

### 이 문서에서 잡아야 할 CS 감각

이 구간의 핵심은 결제 도메인의 불변식이다. 금액은 정확해야 하고, 잔액은 설명 가능해야 하며, 원장과 정산은 나중에 검증 가능해야 한다. 좋은 결제 시스템은 성공 처리보다 설명 가능한 기록을 더 중요하게 본다.

### 질문의 질문

아래 질문들은 답을 외우기 위한 질문이 아니다. 하나씩 답하면서 "왜 그런 설계가 필요한지"를 설명하는 연습을 하기 위한 질문이다.

- 이 데이터가 틀렸을 때 돈이 실제로 잘못 움직이는가, 아니면 후속 기록만 늦어지는가?
- 원천 데이터와 집계 데이터가 다를 때 무엇을 진실로 볼 것인가?
- 나중에 분쟁이 생겼을 때 이 기록만 보고 거래를 설명할 수 있는가?
- 이 개념을 적용하면 어떤 문제가 해결되지만, 어떤 비용이나 복잡도가 새로 생기는가?
- 장애가 발생했을 때 이 개념은 문제를 예방하는가, 감지하는가, 복구하는가?
- PayFlow 코드에서 이 개념을 확인하려면 어떤 서비스, 테이블, 로그, 테스트를 봐야 하는가?

#### 핵심 답변

Currency And Decimal Handling 개념은 PayFlow에서 다음 이유로 중요하다.

- 결제 도메인 개념은 PayFlow에서 돈의 이동을 정확하고 설명 가능하며 나중에 검증 가능하게 만들기 위해 필요하다.
- 지갑 잔액, 잔액 변경 이력, 원장 엔트리, 정산 결과, 대사 리포트가 진실을 검증하는 기준이다.
- 금액 반올림 오류, 중복 정산, 원장 누락, 잔액과 원장 불일치가 발생하면 분쟁과 금전 손실로 이어진다.
- 정수 금액 모델, append-only 원장, 기준일 배치와 거래별 대사로 방어하며 다통화/정책 버전은 개선 과제다.
- 운영에서는 잔액-원장 차이, 정산 검증 실패 수, 수수료 합계 차이, 중복 정산 차단 건수, 대사 실패 리포트를 본다.

2. 그 데이터의 진실의 원천이 어느 서비스에 있는지 말한다.
3. 장애, 중복, 동시성, 지연 중 어떤 상황에서 문제가 생기는지 설명한다.
4. 코드 레벨의 방어 수단과 운영 레벨의 확인 수단을 함께 말한다.

### PayFlow에 대입해보기

1. 이 개념이 가장 직접적으로 연결되는 PayFlow 서비스 하나를 고른다.
2. 그 서비스가 소유한 데이터의 "진실의 원천"이 무엇인지 적어본다.
3. 동시에 두 요청이 들어오거나, 네트워크가 끊기거나, 프로세스가 죽는 상황을 가정한다.
4. 그 상황에서 데이터가 깨지지 않으려면 어떤 제약조건, 트랜잭션, 락, 이벤트, 재처리 장치가 필요한지 설명한다.
5. 마지막으로 운영자가 문제를 발견할 수 있는 로그나 지표가 무엇인지 적어본다.
