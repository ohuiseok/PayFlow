# Blue Green Deployment

## 핵심 개념

Blue-Green Deployment는 현재 운영 중인 Blue 환경과 새 버전 Green 환경을 동시에 준비한 뒤, 트래픽을 한 번에 전환하는 배포 방식이다.

문제가 생기면 트래픽을 다시 Blue로 돌려 빠르게 복구할 수 있다.

### 동작 원리

Blue-Green은 두 개의 동일한 프로덕션 환경을 유지한다.

```text
[로드밸런서/API Gateway]
       |
       v
  [Blue - v1]  <-- 현재 트래픽 처리 중
  [Green - v2] <-- 새 버전 준비 중 (트래픽 없음)
```

배포 과정은 다음과 같다.

1. Green 환경에 새 버전을 배포한다.
2. Green 환경에서 스모크 테스트, 헬스체크를 수행한다.
3. 로드밸런서의 트래픽을 Blue에서 Green으로 한 번에 전환한다.
4. 일정 시간 Green을 모니터링한다.
5. 문제가 없으면 Blue 환경을 다음 배포를 위해 유지한다. 문제가 있으면 즉시 Blue로 롤백한다.

```text
배포 후:
[로드밸런서/API Gateway]
       |
       v
  [Blue - v1]  <-- 대기 (롤백용)
  [Green - v2] <-- 현재 트래픽 처리 중
```

### 트래픽 전환 방법

**DNS 전환**: Blue와 Green이 각각 다른 IP를 가지고, DNS 레코드를 변경해 트래픽을 전환한다. TTL 때문에 전환이 즉각적이지 않을 수 있다.

**로드밸런서 전환**: AWS ALB, Nginx 등에서 Target Group을 전환한다. 거의 즉각적으로 전환된다.

```nginx
# Nginx upstream 전환 예시
upstream app {
    server green-server:8080;  # Blue에서 Green으로 변경
}
```

**AWS CodeDeploy / ECS**: 자동으로 Blue-Green 전환을 관리한다. 트래픽을 점진적으로 이동하는 Canary 전환도 지원한다.

**Kubernetes**: Service의 selector를 변경해 트래픽을 전환한다.

```yaml
# Blue Deployment
apiVersion: apps/v1
kind: Deployment
metadata:
  name: payflow-blue
spec:
  selector:
    matchLabels:
      version: blue

# Service - 전환 전
apiVersion: v1
kind: Service
spec:
  selector:
    version: blue  # --> green으로 변경하면 전환 완료
```

### Warm-up과 커넥션 드레이닝

트래픽을 전환할 때 두 가지를 고려해야 한다.

**Warm-up**: 새 인스턴스는 JVM JIT 컴파일, 캐시 워밍, DB 커넥션 풀 생성 등이 완료되기 전까지 응답이 느리다. 트래픽 전환 전에 Warm-up 요청을 보내거나, 전환 후 초기에는 소수의 트래픽만 보내는 전략이 필요하다.

**커넥션 드레이닝**: Blue로 가던 기존 요청들이 처리 완료될 때까지 기다려야 한다. 로드밸런서의 "Deregistration Delay" 설정(AWS ALB 기본값 300초)으로 Blue 인스턴스를 즉시 종료하지 않고 기존 요청이 완료될 때까지 유지한다.

### DB 마이그레이션과의 관계

Blue-Green에서 가장 복잡한 문제는 DB 스키마 변경이다.

Blue와 Green이 같은 DB를 공유하면, Green이 적용한 스키마 변경이 Blue 코드와 호환되어야 한다. 롤백 시 Green에서 추가한 컬럼에 Blue 코드가 접근하지 못할 수 있다.

그래서 Blue-Green 배포 시 DB 변경은 하위 호환 방식(컬럼 추가, 기본값 설정)으로만 해야 하며, 컬럼 삭제나 타입 변경은 롤백 기간이 지난 후 별도 단계에서 수행해야 한다.

일부 팀은 Blue와 Green이 별도 DB를 가지고, 데이터를 실시간으로 동기화하는 방식을 선택하지만 이는 더 복잡하다.

### 비용과 자원 요구사항

Blue-Green은 두 배의 인프라가 필요하다. 10대의 서버가 필요한 서비스라면 20대를 준비해야 한다. 이는 비용이 크게 증가하는 단점이다.

클라우드 환경에서는 배포 시에만 Green 환경을 생성하고 전환 완료 후 Blue를 종료하는 방식으로 비용을 절감할 수 있다. 단, 이 경우 롤백 시 Blue를 다시 생성해야 하므로 시간이 걸린다.

### Canary 배포와의 차이

Blue-Green은 트래픽을 100% 전환한다. Canary 배포는 일부 트래픽(1%, 5%, 10% 등)을 새 버전으로 먼저 보내 검증한 후 점진적으로 늘린다.

Blue-Green은 롤백이 빠르고 단순하다. Canary는 실제 트래픽으로 새 버전을 검증할 수 있고 위험을 분산한다. 두 방식을 조합해 Canary로 검증하고 Blue-Green으로 완전 전환하는 방식도 사용된다.

### 흔한 오해와 함정

**오해: Blue-Green은 완전한 무중단이다**

트래픽 전환 순간에 처리 중이던 요청이 영향받을 수 있다. 특히 세션 상태가 Blue에만 있고 Green에 없다면 전환 후 사용자가 로그아웃되거나 오류가 발생한다. 세션을 외부 저장소(Redis)에 유지하는 Stateless 설계가 전제되어야 진정한 무중단이 된다.

**오해: 롤백은 항상 즉각적이다**

트래픽은 즉각 롤백되지만, Green이 DB에 기록한 데이터는 롤백되지 않는다. 신버전에서 생성된 데이터를 구버전이 처리할 수 없는 경우 데이터 정합성 문제가 생긴다.

### Trade-off

| 항목 | Blue-Green | Rolling |
|------|-----------|---------|
| 롤백 속도 | 즉각적 | 느림 |
| 배포 중 구/신 버전 공존 | 없음 | 있음 |
| 자원 비용 | 2배 | 동일 |
| DB 마이그레이션 복잡도 | 높음 | 낮음 |
| 트래픽 검증 | 전환 전 테스트만 | 배포 중 자동 |

## PayFlow 연결

결제 시스템에서는 배포 중 장애가 사용자 송금 실패로 이어질 수 있다. Blue-Green은 빠른 롤백에 유리하지만, DB 스키마 변경이 하위 호환되지 않으면 롤백이 어려울 수 있다.

## 실무 포인트

- 두 환경이 같은 DB를 쓸 때 스키마 호환성을 지킨다.
- 전환 전 Green 환경 헬스체크를 확인한다.
- 세션과 캐시 영향을 고려한다.
- Kafka Consumer 중복 실행 여부를 조심한다.
- 롤백 절차를 미리 정한다.

## 체크 질문

- Blue-Green 배포의 장점은 무엇인가
- DB 변경이 있으면 롤백이 어려워지는 이유는 무엇인가
- Kafka Consumer를 두 환경에서 동시에 실행할 때 주의할 점은 무엇인가

## 실무 설계 보강

### 대표 장애 시나리오

새 Green 환경에서 송금 API가 실패하지만 트래픽 전환 후 빠르게 되돌리지 못한다.

### 잘못된 구현 예시

~~~text
새 버전을 띄웠으니 바로 기존 환경을 내린다.
~~~

### 좋은 구현 예시

~~~text
Blue/Green 환경을 동시에 두고 health check 후 트래픽을 전환하며 롤백 경로를 유지한다.
~~~

### 대안과 선택 이유

장애가 나면 사람이 재시작하거나 배포를 되돌리는 방식도 가능하지만, 결제 시스템에서는 장애 전파 속도가 빠르고 수동 대응은 늦다. PayFlow는 Timeout, Retry 제한, Circuit Breaker, Bulkhead, Graceful Shutdown, 호환 배포를 통해 피해 범위를 자동으로 줄이는 편이 낫다.

### PayFlow에서 확인할 위치

deployment compose/profile, nginx upstream, health check

### 면접에서 설명하기

Blue-Green은 빠른 전환과 빠른 복구를 위한 방식이다. DB 호환성이 없으면 장점이 줄어든다.

## 질문의 질문으로 더 깊게 보기

### 이 문서에서 잡아야 할 CS 감각

이 구간의 핵심은 장애를 예외 상황이 아니라 설계 입력으로 보는 것이다. Timeout, Retry, Circuit Breaker, Graceful Shutdown, 배포 전략은 모두 장애가 났을 때 피해를 제한하고 복구 가능하게 만드는 장치다.

### 질문의 질문

아래 질문들은 답을 외우기 위한 질문이 아니다. 하나씩 답하면서 "왜 그런 설계가 필요한지"를 설명하는 연습을 하기 위한 질문이다.

- 배포 중 구버전과 신버전이 동시에 존재해도 계약이 깨지지 않는가?
- 종료 또는 재시작 중 처리하던 요청은 어디까지 완료되었다고 볼 수 있는가?
- 롤백했을 때 DB 스키마와 이벤트 스키마도 함께 되돌릴 수 있는가?
- 이 개념을 적용하면 어떤 문제가 해결되지만, 어떤 비용이나 복잡도가 새로 생기는가?
- 장애가 발생했을 때 이 개념은 문제를 예방하는가, 감지하는가, 복구하는가?
- PayFlow 코드에서 이 개념을 확인하려면 어떤 서비스, 테이블, 로그, 테스트를 봐야 하는가?

#### 핵심 답변

Blue Green Deployment 개념은 PayFlow에서 다음 이유로 중요하다.

- 장애 대응과 배포 개념은 PayFlow가 실패를 완전히 피하지 못하더라도 피해를 제한하고 안전하게 복구하기 위해 필요하다.
- 서비스 상태, 배포 버전, DB 마이그레이션 이력, health check, retry/circuit breaker 상태가 기준이다.
- 무제한 재시도, 타임아웃 부재, 배포 중 스키마 불일치, 종료 중 메시지 처리 중단이 장애를 확대한다.
- Timeout, Retry Backoff, Circuit Breaker, Bulkhead, Graceful Shutdown, 하위 호환 배포, Readiness로 방어한다.
- 운영에서는 timeout 수, retry 수, circuit open 수, readiness 실패, 배포 후 에러율, Consumer rebalance를 본다.

#### 질문에 답하는 방식

좋은 답변은 용어 정의에서 멈추지 않고 다음 순서로 이어져야 한다.

1. 어떤 데이터나 요청을 보호하려는 개념인지 말한다.
2. 그 데이터의 진실의 원천이 어느 서비스에 있는지 말한다.
3. 장애, 중복, 동시성, 지연 중 어떤 상황에서 문제가 생기는지 설명한다.
4. 코드 레벨의 방어 수단과 운영 레벨의 확인 수단을 함께 말한다.

이 개념이 없으면 Green 환경의 송금 API가 실패하는데 전환 후 롤백 절차가 없어 장애 시간이 길어진다.
그래서 코드에서는 Blue/Green 환경 헬스체크, 트래픽 스위치, DB 스키마 호환성, 롤백 절차를 준비하고,
운영에서는 전환 직후 오류율, health 상태, 롤백 소요 시간을 확인해야 한다.

### PayFlow에 대입해보기

1. 이 개념이 가장 직접적으로 연결되는 PayFlow 서비스 하나를 고른다.
2. 그 서비스가 소유한 데이터의 "진실의 원천"이 무엇인지 적어본다.
3. 동시에 두 요청이 들어오거나, 네트워크가 끊기거나, 프로세스가 죽는 상황을 가정한다.
4. 그 상황에서 데이터가 깨지지 않으려면 어떤 제약조건, 트랜잭션, 락, 이벤트, 재처리 장치가 필요한지 설명한다.
5. 마지막으로 운영자가 문제를 발견할 수 있는 로그나 지표가 무엇인지 적어본다.
