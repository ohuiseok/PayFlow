# Audit Log

> 도메인 전환 안내: 현재 PayFlow는 **청년 정책 참여 미션 및 지원금 지급 플랫폼**으로 설명한다. 내부 구현 호환성을 위해 `PARENT`/`CHILD`, `/api/families`, `/api/missions`, `/api/cashbook`, `reward-service` 같은 명칭은 유지하지만, 문서와 발표에서는 각각 **기관 담당자**, **청년 참여자**, **참여자 연결**, **정책 미션**, **지원금 사용 내역**, **정책 미션/지원금 서비스**로 해석한다.

## 핵심 개념

Audit Log는 누가, 언제, 어떤 작업을 했고, 결과가 무엇이었는지 추적하기 위한 기록이다. 보안, 장애 분석, 분쟁 대응, 규제 대응에 사용된다.

## PayFlow 연결

PayFlow에서는 송금 요청 생성, 잔액 변경, 로그인, 관리자성 작업, 정산 실행, 보상 처리 같은 이벤트를 감사 로그로 남길 수 있다.

감사 로그는 비즈니스 원장과 목적이 다르다. 원장은 금전 흐름을 설명하고, 감사 로그는 행위와 접근을 설명한다.

## 실무 포인트

- 사용자 ID, 요청 ID, IP, 시간, 작업 결과를 남긴다.
- 민감 정보는 마스킹한다.
- 감사 로그는 조작하기 어렵게 보관한다.
- 실패한 요청도 중요한 감사 대상이다.
- 로그 보관 기간과 접근 권한을 정한다.

## 체크 질문

- Audit Log와 Ledger의 차이는 무엇인가
- 실패한 송금 요청도 감사 로그로 남겨야 하는 이유는 무엇인가
- 감사 로그에 남기면 안 되는 정보는 무엇인가

## 실무 설계 참고

### 대표 장애 시나리오

관리자나 사용자의 민감 작업이 기록되지 않아 사고 후 누가 무엇을 했는지 알 수 없다.

### 잘못된 구현 예시

~~~text
비즈니스 원장이 있으니 감사 로그는 필요 없다고 생각한다.
~~~

### 좋은 구현 예시

~~~text
행위자, 요청 ID, IP, 작업, 결과, 시간을 마스킹된 감사 로그로 남긴다.
~~~

### 대안과 선택 이유

현재 잔액이나 최종 집계값만 저장하는 단순한 방식도 있지만, 결제 시스템에서는 나중에 왜 그런 결과가 나왔는지 설명할 수 있어야 한다. PayFlow는 금액 정책, 원장, 이력, 정산 결과, 대사 리포트를 남기는 방식이 분쟁 대응과 복구에 더 강하다.

### PayFlow에서 확인할 위치

gateway access log, service audit log, admin/internal APIs

### 면접에서 설명하기

Audit Log는 돈의 이동이 아니라 행위의 책임을 기록한다. 원장과 목적이 다르다.

### 관련 문서

11, 46, 75

## 심화 참고

### Audit Log와 일반 로그의 차이

일반 애플리케이션 로그는 장애 분석을 위해 남기는 실행 기록이다. 반면 Audit Log는 책임 추적을 위해 남기는 보안성 기록이다. 따라서 "나중에 검색할 수 있으면 된다"가 아니라 "나중에 누가 무엇을 했는지 신뢰할 수 있어야 한다"가 핵심이다.

| 구분 | 일반 로그 | 감사 로그 |
| --- | --- | --- |
| 목적 | 디버깅, 장애 분석 | 책임 추적, 분쟁 대응, 규제 대응 |
| 보관 기준 | 운영 편의와 비용 | 정책, 법무, 보안 기준 |
| 내용 | 예외, 상태, 내부 값 | 행위자, 대상, 작업, 결과, 근거 |
| 변경 가능성 | 재수집/삭제 가능 | 임의 변경이 어려워야 함 |

### 필드 설계 예시

```text
audit_log
- audit_id
- occurred_at
- actor_type        // USER, ADMIN, SYSTEM
- actor_id
- action            // TRANSFER_REQUESTED, SETTLEMENT_APPROVED
- resource_type     // TRANSFER, WALLET, SETTLEMENT
- resource_id
- request_id
- trace_id
- client_ip
- user_agent
- result            // SUCCESS, FAILURE, DENIED
- reason_code
- masked_payload
```

감사 로그에는 원문 카드번호, 비밀번호, 토큰, 주민등록번호 같은 민감 정보를 넣지 않는다. 필요한 경우 원문 대신 마스킹 값, 해시, 내부 식별자만 남긴다.

### PayFlow에서 특히 남겨야 할 행위

- 로그인 성공/실패, 토큰 재발급, 권한 거부
- 송금 요청 생성, 취소, 보상 처리 요청
- 지갑 잔액 수동 보정 같은 관리자성 작업
- 정산 배치 실행, 승인, 재실행, 실패 처리
- DLQ 메시지 재처리, 보상 트랜잭션 실행
- 설정 변경: 수수료율, 한도, 차단 계정, API 키

### 설계 체크리스트

- 감사 로그 저장 실패가 핵심 비즈니스 트랜잭션을 막아야 하는지 정책을 정한다.
- 관리자성 작업은 감사 로그 실패 시 작업 자체를 거부하는 편이 안전하다.
- 사용자 일반 요청은 감사 로그를 Outbox처럼 비동기로 남기되 누락 감지 지표를 둔다.
- 로그 조회 권한을 별도로 제한하고, 감사 로그 조회 행위 자체도 감사 대상으로 남긴다.
- 삭제 정책이 필요하면 물리 삭제보다 보존 기간, 익명화, 별도 아카이브를 우선 검토한다.

### 테스트 포인트

- 실패한 송금 요청과 권한 거부 요청도 감사 로그에 남는지 확인한다.
- 민감 정보가 감사 로그에 평문으로 남지 않는지 확인한다.
- 같은 `request_id`로 비즈니스 로그, 감사 로그, trace를 연결할 수 있는지 확인한다.
- 관리자 API 호출 후 행위자, 대상, 결과가 빠짐없이 남는지 확인한다.

## 질문의 질문으로 더 깊게 보기

### 이 문서에서 잡아야 할 CS 감각

이 구간의 핵심은 결제 도메인의 불변식이다. 금액은 정확해야 하고, 잔액은 설명 가능해야 하며, 원장과 정산은 나중에 검증 가능해야 한다. 좋은 결제 시스템은 성공 처리보다 설명 가능한 기록을 더 중요하게 본다.

### 질문의 질문

아래 질문들은 답을 외우기 위한 질문이 아니다. 하나씩 답하면서 "왜 그런 설계가 필요한지"를 설명하는 연습을 하기 위한 질문이다.

- Audit Log 개념이 없다면 PayFlow에서 가장 먼저 어떤 장애가 생기는가?
- 이 개념은 정확성, 성능, 보안, 운영성 중 무엇을 가장 크게 개선하는가?
- 반대로 이 개념을 잘못 적용하면 어떤 복잡도가 추가되는가?
- 이 개념을 적용하면 어떤 문제가 해결되지만, 어떤 비용이나 복잡도가 새로 생기는가?
- 장애가 발생했을 때 이 개념은 문제를 예방하는가, 감지하는가, 복구하는가?
- PayFlow 코드에서 이 개념을 확인하려면 어떤 서비스, 테이블, 로그, 테스트를 봐야 하는가?

#### 핵심 답변

Audit Log 개념은 PayFlow에서 다음 이유로 중요하다.

- 결제 도메인 개념은 PayFlow에서 돈의 이동을 정확하고 설명 가능하며 나중에 검증 가능하게 만들기 위해 필요하다.
- 지갑 잔액, 잔액 변경 이력, 원장 엔트리, 정산 결과, 대사 리포트가 진실을 검증하는 기준이다.
- 금액 반올림 오류, 중복 정산, 원장 누락, 잔액과 원장 불일치가 발생하면 분쟁과 금전 손실로 이어진다.
- 정수 금액 모델, append-only 원장, 수수료 정책 버전, 재처리 가능한 배치, 대사로 방어한다.
- 운영에서는 잔액-원장 차이, 정산 검증 실패 수, 수수료 합계 차이, 중복 정산 차단 건수, 대사 실패 리포트를 본다.

#### 질문에 답하는 방식

좋은 답변은 용어 정의에서 멈추지 않고 다음 순서로 이어져야 한다.

1. 어떤 데이터나 요청을 보호하려는 개념인지 말한다.
2. 그 데이터의 진실의 원천이 어느 서비스에 있는지 말한다.
3. 장애, 중복, 동시성, 지연 중 어떤 상황에서 문제가 생기는지 설명한다.
4. 코드 레벨의 방어 수단과 운영 레벨의 확인 수단을 함께 말한다.

#### PayFlow 예시 답변

```text
Audit Log 개념은 PayFlow에서 누가 언제 어떤 결제 관련 작업을 했는지 추적 가능하게 하기 위해 필요하다.
이 개념이 없으면 관리자성 API나 실패한 송금 요청의 행위자가 남지 않아 사고 분석과 분쟁 대응이 어려워진다.
그래서 코드에서는 userId, requestId, IP, 작업 결과를 감사 로그로 남기고 민감 정보는 마스킹하며,
운영에서는 감사 로그 누락, 실패 요청 로그, 관리자 작업 이력을 확인해야 한다.
```

#### 더 생각해볼 점

이 답안은 하나의 예시다. 실제 설계에서는 성능, 복잡도, 장애 복구 난이도 사이의 trade-off를 함께 봐야 한다. 특히 PayFlow처럼 돈을 다루는 시스템에서는 빠른 성공보다 명확한 실패, 추적 가능한 상태, 재처리 가능한 구조가 더 중요하다.

### PayFlow에 대입해보기

1. 이 개념이 가장 직접적으로 연결되는 PayFlow 서비스 하나를 고른다.
2. 그 서비스가 소유한 데이터의 "진실의 원천"이 무엇인지 적어본다.
3. 동시에 두 요청이 들어오거나, 네트워크가 끊기거나, 프로세스가 죽는 상황을 가정한다.
4. 그 상황에서 데이터가 깨지지 않으려면 어떤 제약조건, 트랜잭션, 락, 이벤트, 재처리 장치가 필요한지 설명한다.
5. 마지막으로 운영자가 문제를 발견할 수 있는 로그나 지표가 무엇인지 적어본다.
