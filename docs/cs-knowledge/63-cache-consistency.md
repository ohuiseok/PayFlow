# Cache Consistency

> 도메인 전환 안내: 현재 PayFlow는 **청년 정책 참여 미션 및 지원금 지급 플랫폼**으로 설명한다. 내부 구현 호환성을 위해 `PARENT`/`CHILD`, `/api/families`, `/api/missions`, `/api/cashbook`, `reward-service` 같은 명칭은 유지하지만, 문서와 발표에서는 각각 **기관 담당자**, **청년 참여자**, **참여자 연결**, **정책 미션**, **지원금 사용 내역**, **정책 미션/지원금 서비스**로 해석한다.

## 핵심 개념

### 캐시가 존재하는 이유와 정합성 비용

캐시는 느린 저장소(DB, 외부 API)의 결과를 빠른 저장소(메모리, Redis)에 임시 보관해서 성능을 높인다. DB 조회 1회가 10ms라면, 캐시 조회는 0.1ms 이하다. 1000배 차이다.

하지만 캐시는 원본 데이터의 복사본이다. 원본이 바뀌면 복사본은 시간이 지날 때까지 오래된(stale) 상태가 된다. 이 시간 동안 캐시에서 읽으면 정확하지 않은 데이터를 보게 된다.

정합성 비용이란, 캐시를 빠르게 원본과 동기화할수록 캐시의 성능 이점이 줄어든다는 것이다. 모든 쓰기마다 캐시를 즉시 무효화하면 캐시 miss가 잦아지고 DB 부하가 다시 높아진다.

### 캐시 패턴 네 가지와 정합성

**Cache-Aside (Lazy Loading)**

```
읽기:
1. 캐시에서 조회
2. 캐시 miss → DB에서 조회
3. 결과를 캐시에 저장 후 반환

쓰기:
1. DB에 쓰기
2. 캐시 삭제 (또는 업데이트)
```

가장 일반적인 패턴이다. 쓰기 시 캐시를 삭제하면, 다음 읽기가 DB에서 최신 값을 가져와 캐시를 갱신한다. 일시적 stale 윈도우(삭제~다음 읽기)가 존재한다.

**Write-Through**

```
쓰기:
1. 캐시에 쓰기
2. DB에 쓰기 (동기적)

읽기:
1. 캐시에서 조회 (항상 최신)
```

쓸 때 캐시와 DB를 동시에 업데이트한다. 캐시가 항상 최신 상태지만 쓰기 지연이 증가한다. 캐시된 데이터가 읽히지 않아도 캐시에 저장되므로 불필요한 캐시 공간을 사용할 수 있다.

**Write-Behind (Write-Back)**

```
쓰기:
1. 캐시에만 쓰기 (즉시 응답)
2. 비동기로 나중에 DB에 쓰기
```

쓰기 성능이 가장 높지만, 캐시가 죽으면 DB에 반영되지 않은 데이터가 사라진다. 금전 시스템에서는 사용하면 안 된다.

**Read-Through**

캐시가 DB 조회를 대신 수행한다. Cache-Aside와 비슷하지만 캐시 레이어가 DB 조회를 처리한다. 애플리케이션 코드가 단순해진다.

### Cache Invalidation의 어려움

Phil Karlton의 유명한 말이 있다: "컴퓨터 과학에서 어려운 것은 두 가지다. 캐시 무효화와 이름 짓기."

캐시 무효화가 어려운 이유는 분산 환경에서 여러 서버가 각자 캐시를 가지거나, 하나의 데이터가 여러 캐시 키에 영향을 줄 수 있기 때문이다.

```
문제 시나리오:
1. 사용자 정보가 user:{userId} 키로 캐시됨
2. 이름과 이메일로도 각각 검색 캐시가 있음
3. 사용자 이름 변경 시 어떤 캐시를 모두 무효화할 수 있는가?
```

**패턴 기반 삭제**는 복잡하고 성능 비용이 높다. `KEYS user:*` 같은 명령은 Redis를 블로킹할 수 있다. 대신 Tag 기반 무효화나 버전 기반 캐시 키를 사용한다.

```
버전 기반 캐시 키:
user:{userId}:v{version}

업데이트 시 version 증가 → 이전 키는 자연스럽게 TTL로 만료
새 요청은 새 버전 키를 사용
```

### TTL 설계 전략

TTL(Time To Live)은 캐시가 자동으로 만료되는 시간이다. TTL이 짧으면 DB 부하가 높고 stale 기간이 짧다. TTL이 길면 DB 부하가 낮고 stale 기간이 길다.

데이터 변경 빈도에 따라 TTL을 달리 설정해야 한다.

| 데이터 | 변경 빈도 | 권장 TTL |
|---|---|---|
| 시스템 설정 | 드물게 | 수 시간 ~ 수일 |
| 사용자 프로필 | 가끔 | 수 분 ~ 수 시간 |
| 상품 목록 | 자주 | 수 십초 ~ 수 분 |
| 실시간 재고 | 매우 자주 | 수 초 또는 캐시 없음 |
| 지갑 잔액 | 트랜잭션마다 | 캐시 사용 금지 (또는 검증 미사용) |

### 결제 시스템에서의 캐시 사용 원칙

**캐시를 절대 사용하면 안 되는 곳**

잔액 검증, 권한 검증, 중복 결제 확인. 이 데이터는 반드시 DB에서 직접 읽어야 한다. 오래된 캐시 값으로 판단하면 실제 잔액보다 많은 금액이 출금되거나, 이미 취소된 결제가 유효한 것처럼 처리될 수 있다.

```java
// 잘못된 패턴: 캐시로 잔액 검증
public void withdraw(String walletId, BigDecimal amount) {
    BigDecimal balance = cache.get("wallet:" + walletId);  // stale 가능!
    if (balance.compareTo(amount) < 0) {
        throw new InsufficientBalanceException();
    }
    // 실제 DB 잔액이 더 적어도 통과됨
}

// 올바른 패턴: DB에서 잔액 검증
public void withdraw(String walletId, BigDecimal amount) {
    // SELECT ... FOR UPDATE로 락을 잡고 DB에서 직접 읽음
    BigDecimal balance = walletRepository.findForUpdate(walletId).getBalance();
    if (balance.compareTo(amount) < 0) {
        throw new InsufficientBalanceException();
    }
}
```

**캐시를 써도 되는 곳**

사용자 프로필 표시용 정보, 환율 정보(주기적 갱신), 공지사항, 거래 내역 페이지(이미 확정된 과거 데이터).

### Cache Stampede (Thunder Herd) 문제

동일 캐시 키의 TTL이 만료되는 순간, 동시에 많은 요청이 DB로 몰리는 현상이다.

```
09:00:00 - cache 만료
09:00:01 - 1000개 요청이 동시에 cache miss → 1000개 DB 쿼리
DB 과부하 → 응답 지연 → 캐시 갱신 지연 → 더 많은 DB 쿼리
```

**해결 방법**
- TTL에 랜덤 jitter 추가: `TTL = 300 + random(0, 60)` 초
- 뮤텍스(분산 락): 한 번에 하나만 DB 조회하고 나머지는 대기
- Probabilistic Early Expiration: TTL 만료 전에 일부 요청이 미리 갱신

## PayFlow 연결

PayFlow에서 사용자 정보나 조회용 데이터는 캐시할 수 있다. 하지만 지갑 잔액처럼 금전 정합성이 중요한 데이터는 캐시 사용에 신중해야 한다.

잔액 조회에 캐시를 쓰면 출금 직후 오래된 잔액이 보일 수 있다. 더 위험하게는 캐시 값을 기준으로 출금 검증을 하면 안 된다.

## 실무 포인트

- 쓰기 후 캐시 무효화 전략을 정한다.
- 잔액 검증은 원본 DB 기준으로 한다.
- TTL은 데이터 성격에 맞게 설정한다.
- 캐시 장애 시 DB 부하 증가를 고려한다.
- Cache Aside 패턴을 이해한다.

## 체크 질문

- 캐시가 데이터 정합성을 깨뜨릴 수 있는 이유는 무엇인가
- 지갑 잔액 검증에 캐시를 쓰면 왜 위험한가
- TTL은 어떤 기준으로 정해야 하는가

## 실무 설계 참고

### 대표 장애 시나리오

오래된 캐시 잔액을 기준으로 출금 검증을 수행한다.

### 잘못된 구현 예시

~~~text
캐시는 빠른 DB라고 생각하고 정합성 요구가 높은 데이터에도 그대로 쓴다.
~~~

### 좋은 구현 예시

~~~text
잔액 검증은 원본 DB 기준으로 하고 캐시는 조회 보조로만 제한한다.
~~~

### 대안과 선택 이유

서비스 간 REST 호출만으로 후속 처리를 연결하는 방식도 있지만, 원장이나 정산 서비스 장애가 송금 응답에 직접 영향을 준다. PayFlow는 Kafka와 Outbox로 시간적 결합을 낮추고, Consumer 멱등성과 DLQ로 중복/실패를 감당하는 방식이 더 적합하다.

### PayFlow에서 확인할 위치

wallet balance read path, Redis cache usage, cache invalidation logic

### 면접에서 설명하기

캐시는 성능을 주지만 진실의 원천이 아니다. 결제에서는 캐시가 판단 기준이 되면 위험하다.

#### 핵심 답변

Cache Consistency 개념은 PayFlow에서 다음 이유로 중요하다.

- Kafka와 메시징 개념은 PayFlow에서 송금 완료 이후 원장과 정산을 느슨하게 연결하면서도 이벤트를 잃지 않기 위해 필요하다.
- Kafka topic/partition/offset, outbox_event, processed_event, DLQ가 이벤트 처리의 기준이다.
- 이벤트 유실, 중복 소비, 순서 뒤바뀜, Consumer lag, DLQ 적체가 원장 누락이나 중복 기록으로 이어질 수 있다.
- partition key 설계, 처리 후 offset commit, Outbox, processed_event 유니크 제약, DLQ와 재처리 도구로 방어한다.
- 운영에서는 Kafka lag, rebalance 횟수, DLQ 메시지 수, Consumer 처리 시간, processed_event 중복 차단 건수를 본다.

1. 이 개념이 가장 직접적으로 연결되는 PayFlow 서비스 하나를 고른다.
