# Concurrency Control And Locking

## 핵심 개념

### 동시성 문제의 종류

여러 트랜잭션이 같은 데이터를 동시에 읽고 쓸 때 다음 문제가 발생할 수 있다.

**Lost Update (갱신 손실)**: 두 트랜잭션이 같은 값을 읽고 각자 수정하면, 나중에 커밋한 쪽의 변경이 앞서 커밋한 쪽의 변경을 덮어쓴다.

```text
잔액: 10,000원
T1: 읽기(10,000) -> 차감 -> 8,000으로 업데이트
T2: 읽기(10,000) -> 차감 -> 7,000으로 업데이트 (T1 결과 덮어씀)
최종 잔액: 7,000 (T1의 2,000원 차감이 사라짐)
```

결제 시스템에서 Lost Update는 직접적인 금전 사고다.

**Dirty Read**: 아직 커밋되지 않은 변경 값을 다른 트랜잭션이 읽는다. T1이 잔액을 변경했지만 롤백하면, T2는 존재하지 않는 잔액을 기준으로 처리한 셈이다.

**Non-Repeatable Read**: 같은 트랜잭션 안에서 같은 행을 두 번 읽었을 때 값이 달라진다. 잔액 확인과 출금 처리 사이에 다른 트랜잭션이 잔액을 바꿀 수 있다.

**Phantom Read**: 같은 조건으로 두 번 조회했을 때 결과 행 수가 달라진다. 정산 대상 조회 사이에 새 데이터가 삽입될 수 있다.

### 동시성 제어 방식

**DB 격리 수준**: MVCC와 락을 통해 격리 수준을 조정한다. 높은 격리 수준일수록 안전하지만 성능이 낮아진다.

**비관적 락(Pessimistic Lock)**: 데이터를 읽을 때부터 다른 트랜잭션이 변경하지 못하도록 락을 건다. 충돌이 잦거나 금전처럼 충돌 시 비용이 큰 데이터에 적합하다.

**낙관적 락(Optimistic Lock)**: 충돌이 드물다고 보고 수정할 때 버전이 바뀌었는지 확인한다. 충돌 시 예외가 발생하고 재시도한다.

**분산 락(Distributed Lock)**: 여러 서버 인스턴스에서 같은 자원에 대한 접근을 제어한다. Redis의 `SETNX`나 Redisson을 사용한다.

**DB 조건부 UPDATE**: 락 없이 조건을 걸어 원자적으로 업데이트한다.

```sql
-- 잔액이 충분할 때만 차감. 동시 요청 중 하나만 성공
UPDATE wallet
SET balance = balance - 5000
WHERE id = 1 AND balance >= 5000;
-- affected_rows = 0이면 잔액 부족 또는 다른 트랜잭션에 의해 이미 차감됨
```

### 잔액 차감의 동시성 제어

잔액 10,000원에 8,000원 출금 요청 두 개가 동시에 들어오는 시나리오:

**제어 없는 경우**:
```text
T1: SELECT balance FROM wallet WHERE id=1 -> 10,000
T2: SELECT balance FROM wallet WHERE id=1 -> 10,000
T1: 잔액 10,000 >= 8,000 조건 통과
T2: 잔액 10,000 >= 8,000 조건 통과
T1: UPDATE wallet SET balance = 2,000 WHERE id=1
T2: UPDATE wallet SET balance = 2,000 WHERE id=1
결과: 16,000원을 출금했는데 잔액은 2,000원 (잘못된 결과)
```

**비관적 락 사용**:
```text
T1: SELECT balance FROM wallet WHERE id=1 FOR UPDATE -> 행 락 획득, 10,000
T2: SELECT ... FOR UPDATE -> 대기
T1: 잔액 확인, UPDATE wallet SET balance = 2,000, 커밋 -> 락 해제
T2: 락 획득, 잔액 재조회 2,000 -> 2,000 < 8,000, 잔액 부족 오류
결과: 한 건만 성공
```

**조건부 UPDATE 사용**:
```text
T1: UPDATE wallet SET balance = balance - 8000 WHERE id=1 AND balance >= 8000
T2: UPDATE wallet SET balance = balance - 8000 WHERE id=1 AND balance >= 8000
-> 둘이 동시에 실행되면 DB가 직렬화해 처리
-> 첫 번째가 업데이트 완료, 두 번째는 balance가 2000이라 조건 불일치, affected_rows=0
결과: 한 건만 성공
```

### Redis 분산 락 패턴

여러 서버 인스턴스가 있을 때 DB 락만으로는 부족할 수 있다. Redis `SETNX`(Set If Not Exists)로 분산 락을 구현한다.

```text
1. Redis에 lock:wallet:1 키로 SETNX 시도 (TTL 10초)
2. 성공하면 락 획득 -> 잔액 변경 로직 실행
3. 실패하면 락 획득 실패 -> 재시도 또는 오류 반환
4. 작업 완료 후 Redis 키 삭제 (락 해제)
```

주의: 락을 획득한 서버가 작업 중 죽으면 TTL이 만료되어 다른 서버가 락을 획득할 수 있다. 이 시점에 이전 작업이 완료됐는지 확인하는 로직이 필요하다.

### 락 범위와 성능 Trade-off

락은 범위가 좁고 시간이 짧을수록 성능이 좋다. 하지만 너무 좁히면 정합성을 보장하기 어렵다.

**락 범위를 넓힐수록**: 안전하지만 처리량(Throughput)이 낮아지고 대기 시간이 늘어난다.
**락 범위를 좁힐수록**: 처리량이 높아지지만 보호되지 않는 영역이 생길 수 있다.

결제 시스템에서는 "성능보다 정합성"이 우선이지만, 그렇다고 필요 이상으로 넓은 락을 잡는 것도 피해야 한다. 잔액 변경에만 락을 잡고, 이후 이벤트 발행이나 응답 처리는 락 밖에서 한다.

### 흔한 오해와 함정

**"트랜잭션이 있으면 동시성이 자동으로 해결된다"**: 트랜잭션과 격리 수준이 있어도 Lost Update는 기본적으로 막지 못한다. 명시적인 락이나 조건부 UPDATE가 필요하다.

**"애플리케이션에서 if문으로 잔액 체크하면 된다"**: 단일 서버, 단일 스레드라면 동작하지만 동시 요청이나 다중 서버 환경에서는 Race Condition이 발생한다.

**"락을 많이 쓰면 무조건 안전하다"**: 락이 너무 많으면 데드락 가능성이 높아지고 처리량이 낮아진다. 꼭 필요한 곳에만 최소한의 락을 사용해야 한다.

**"동시성 테스트는 복잡하니 나중에 한다"**: 동시성 버그는 개발 환경에서 잘 재현되지 않는다. CountDownLatch 등을 사용해 초기부터 동시 요청 테스트를 작성해야 한다.

```java
@Test
void 동시_출금_테스트() throws InterruptedException {
    int threadCount = 10;
    CountDownLatch latch = new CountDownLatch(1);
    CountDownLatch done = new CountDownLatch(threadCount);

    for (int i = 0; i < threadCount; i++) {
        new Thread(() -> {
            try {
                latch.await();
                walletService.withdraw(walletId, amount);
            } finally {
                done.countDown();
            }
        }).start();
    }

    latch.countDown();  // 모든 스레드 동시 시작
    done.await();

    // 잔액이 음수가 아닌지, 정확히 처리된 건수만 성공했는지 확인
    Wallet wallet = walletRepository.findById(walletId).orElseThrow();
    assertThat(wallet.getBalance()).isGreaterThanOrEqualTo(BigDecimal.ZERO);
}
```


## PayFlow 연결

PayFlow에서 가장 중요한 동시성 지점은 지갑 잔액이다. 같은 지갑에서 동시에 두 건의 출금이 발생하면, 둘 다 같은 잔액을 보고 출금에 성공해 초과 출금이 발생할 수 있다.

예를 들어 잔액이 10,000원인데 8,000원 출금 요청 두 개가 동시에 들어오면 하나만 성공해야 한다.

## 실무 포인트

- 잔액 변경은 반드시 트랜잭션 안에서 처리한다.
- DB 락, 낙관적 락, 비관적 락, 분산 락 중 상황에 맞게 선택한다.
- 락 범위와 시간을 최소화한다.
- 락 실패 시 재시도 정책을 명확히 한다.
- 테스트에서 동시 요청 시나리오를 반드시 검증한다.

## 체크 질문

- Lost Update는 무엇인가
- 지갑 잔액 변경에 동시성 제어가 필요한 이유는 무엇인가
- 락을 오래 잡으면 어떤 문제가 생기는가

## 실무 설계 보강

### 대표 장애 시나리오

잔액 10,000원 지갑에 8,000원 출금 요청 두 개가 동시에 들어와 둘 다 성공한다.

### 잘못된 구현 예시

~~~text
단순히 현재 잔액을 읽고 애플리케이션에서 if문으로만 검사한다.
~~~

### 좋은 구현 예시

~~~text
락, 조건부 update, 트랜잭션, 동시성 테스트로 잔액 불변식을 지킨다.
~~~

### 대안과 선택 이유

애플리케이션 코드에서만 검증하는 방식도 있지만, 동시 요청과 장애 상황에서는 코드 검증만으로 부족하다. PayFlow는 DB 트랜잭션, 락, 유니크 제약, 인덱스, 실행 계획을 함께 사용해 데이터 계층에서도 불변식을 지키는 쪽이 더 안전하다.

### PayFlow에서 확인할 위치

wallet-service withdraw 로직, wallet table, 동시성 테스트

### 면접에서 설명하기

동시성 제어는 운이 나쁜 순간을 막는 기술이다. 결제에서는 그 순간이 곧 금전 사고다.

### 관련 문서
