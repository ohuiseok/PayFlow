# API Versioning

> 도메인 전환 안내: 현재 PayFlow는 **청년 정책 참여 미션 및 지원금 지급 플랫폼**으로 설명한다. 내부 구현 호환성을 위해 `PARENT`/`CHILD`, `/api/families`, `/api/missions`, `/api/cashbook`, `reward-service` 같은 명칭은 유지하지만, 문서와 발표에서는 각각 **기관 담당자**, **청년 참여자**, **참여자 연결**, **정책 미션**, **지원금 사용 내역**, **정책 미션/지원금 서비스**로 해석한다.

## 핵심 개념

API Versioning은 이미 배포된 API의 계약을 깨지 않으면서 기능을 변경하거나 확장하는 방법이다. API를 사용하는 클라이언트나 다른 서비스가 있다면, 필드 하나를 바꾸는 것도 장애가 될 수 있다.

버전 관리는 URI, 헤더, 쿼리 파라미터, 미디어 타입 등으로 할 수 있다.

```text
/api/v1/transfers
/api/v2/transfers
```

### 호환성이 깨지는 변경과 안전한 변경

API 변경은 하위 호환(Backward Compatible) 여부에 따라 두 가지로 나뉜다.

하위 호환이 유지되는 안전한 변경은 다음과 같다.

```text
- 응답에 새 필드 추가 (기존 클라이언트는 무시)
- 요청 바디에 선택 필드 추가
- 새 API 엔드포인트 추가
- 에러 메시지 문구 변경 (에러 코드는 유지)
- 새 enum 값 추가 (클라이언트가 unknown 처리를 한다면)
```

호환성이 깨지는 위험한 변경은 다음과 같다.

```text
- 기존 필드 이름 변경
- 기존 필드 타입 변경 (String -> Integer)
- 기존 필드 삭제
- 기존 enum 값 의미 변경
- 필수 요청 필드 추가
- 상태 코드 의미 변경 (200 -> 201)
- 에러 코드 변경
- 응답 구조 전면 재설계
```

결제 시스템에서는 특히 상태값 enum 변경이 위험하다. `transfer-service`가 `COMPLETED` 상태를 기준으로 후속 처리를 하는데, `wallet-service`가 `DONE`으로 상태값 이름을 바꾸면 transfer-service는 완료 응답을 인식하지 못하고 재시도를 무한 반복할 수 있다.

### 버전 관리 전략 비교

**URI 버전 관리**

가장 단순하고 가시적인 방법이다.

```text
GET /api/v1/wallets/{id}
GET /api/v2/wallets/{id}
```

장점은 URL만 보면 버전을 알 수 있고 캐시하기 쉽다는 것이다. 단점은 같은 리소스가 여러 URL로 나뉘어 관리 비용이 커진다.

**헤더 버전 관리**

```text
GET /api/wallets/{id}
Accept-Version: v2
```

또는

```text
X-API-Version: 2
```

URL은 깔끔하지만 버전을 보려면 헤더를 봐야 하며 브라우저에서 테스트하기 불편하다.

**미디어 타입 버전 관리**

```text
Accept: application/vnd.payflow.v2+json
```

REST 원칙에 가장 충실하지만 구현이 복잡하고 클라이언트 지원 부담이 크다.

내부 서비스 간 통신에서는 URI 버전 관리가 가장 명확하고 추적하기 쉽다. 외부 공개 API에서는 헤더나 미디어 타입 방식이 더 깔끔할 수 있지만, 운영 비용을 고려해야 한다.

### MSA에서의 배포 순서 문제

MSA에서는 서비스마다 독립적으로 배포된다. API 변경 시 어떤 서비스를 먼저 배포하느냐에 따라 일시적인 불일치가 발생한다.

```text
시나리오: wallet-service API v2 배포

방법 1: wallet-service 먼저 배포
  - wallet-service v2 배포됨
  - transfer-service는 아직 v1 응답을 기대함
  - 불일치 구간 발생

방법 2: transfer-service 먼저 배포
  - transfer-service가 v2 응답을 처리하도록 업데이트
  - wallet-service는 아직 v1 응답을 반환
  - transfer-service가 v1 응답도 처리할 수 있어야 함

안전한 방법: Expand/Contract 패턴
  1단계 (Expand): v1과 v2 모두 지원하도록 wallet-service 배포
  2단계: transfer-service가 v2를 사용하도록 배포
  3단계 (Contract): v1 지원 제거
```

이 패턴을 쓰면 배포 중 불일치 구간 없이 안전하게 마이그레이션할 수 있다. 결제 시스템에서는 배포 중 실패 없이 전환하는 것이 매우 중요하다.

### 소비자 주도 계약 테스트

API 변경이 다른 서비스를 깨지 않는지 확인하는 방법 중 하나가 Consumer Driven Contract Testing이다. 소비자(호출하는 서비스)가 기대하는 요청/응답 계약을 테스트 코드로 정의하고, 제공자(호출받는 서비스)가 이 계약을 만족하는지 CI에서 검증한다.

```text
transfer-service (소비자) 정의:
  wallet-service의 출금 API는 다음을 반환해야 한다.
  {
    "walletId": "string",
    "balance": "number",
    "status": "ACTIVE" | "FROZEN"
  }

wallet-service (제공자) 검증:
  실제 응답이 위 계약을 만족하는지 테스트
```

이 방식을 쓰면 wallet-service 개발자가 응답 구조를 바꾸기 전에 다른 서비스가 그 필드에 의존하고 있음을 미리 알 수 있다.

### 흔한 오해와 함정

버전을 올리면 기존 사용자를 지원하지 않아도 된다는 생각은 위험하다. 버전을 나눈 후에도 구버전을 사용하는 클라이언트나 서비스가 있으면 구버전 API를 유지해야 한다. 구버전 종료 시점을 명확히 공지하고, 충분한 마이그레이션 기간을 주어야 한다.

또 다른 함정은 내부 API는 버전 관리가 필요 없다고 생각하는 것이다. 내부 서비스라도 배포 시점이 다르므로 구버전 클라이언트와 신버전 서버가 동시에 운영될 수 있다. 내부 API도 Expand/Contract 패턴이나 명시적 버전 관리가 필요하다.

### trade-off 관점

버전 관리를 철저히 할수록 하위 호환성이 보장되지만 코드베이스가 복잡해진다. v1과 v2를 동시에 지원하면 비즈니스 로직이 두 벌 존재하게 된다. 이 비용을 줄이려면 내부 도메인 로직은 공유하고 외부 계약만 버전별로 분리하는 방법을 쓴다.

PayFlow처럼 소규모 MSA에서는 Expand/Contract 패턴과 계약 테스트 조합이 실용적이다. 과도한 버전 분기보다 배포 순서와 하위 호환 설계를 먼저 고려하는 편이 더 낫다.

## PayFlow 연결

PayFlow에서 `transfer-service`의 송금 요청 필드가 바뀌거나, `wallet-service`의 출금 API 응답 구조가 바뀌면 호출하는 서비스가 영향을 받는다. 특히 MSA에서는 서비스별 배포 시점이 다를 수 있으므로 하위 호환성이 중요하다.

예를 들어 기존 응답에 새 필드를 추가하는 것은 비교적 안전하지만, 기존 필드명을 바꾸거나 의미를 바꾸는 것은 위험하다.

## 실무 포인트

- 필드 삭제나 의미 변경은 가장 위험하다.
- 응답 필드 추가는 대체로 안전하지만 클라이언트 파서에 따라 다를 수 있다.
- 내부 API도 버전 전략이 필요하다.
- API 변경은 배포 순서를 고려해야 한다.
- 구버전 API 종료 시점은 명확히 공지해야 한다.

## 체크 질문

- API 변경 중 하위 호환성이 깨지는 변경은 무엇인가
- MSA에서 API 버전 관리가 중요한 이유는 무엇인가
- 내부 API는 버전 관리가 필요 없다고 말할 수 있는가

신규 송금 상태값이 추가됐지만 구버전 클라이언트나 서비스가 이를 처리하지 못한다.

### 잘못된 구현 예시


### 좋은 구현 예시

~~~text
추가 변경은 하위 호환으로, 삭제/의미 변경은 버전 분리와 단계적 마이그레이션으로 처리한다.
~~~

### 대안과 선택 이유

문서 없이 코드만 맞춰가는 방식도 가능하지만, 서비스가 나뉘면 배포 시점이 달라져 작은 응답 변경도 장애가 된다. PayFlow에서는 API 계약과 버전 전략을 명확히 두는 편이 내부 서비스 간 변경 비용을 줄인다.

## 질문의 질문으로 더 깊게 보기

### 이 문서에서 잡아야 할 CS 감각

이 구간의 핵심은 API가 단순한 입구가 아니라 시스템의 계약이자 보안 경계라는 점이다. 누가 호출할 수 있는지, 어떤 입력을 믿을 수 있는지, 실패를 어떤 형태로 돌려주는지가 서비스 전체의 안정성을 결정한다.

### 질문의 질문

아래 질문들은 답을 외우기 위한 질문이 아니다. 하나씩 답하면서 "왜 그런 설계가 필요한지"를 설명하는 연습을 하기 위한 질문이다.

- API Versioning 개념이 없다면 PayFlow에서 가장 먼저 어떤 장애가 생기는가?
- 이 개념은 정확성, 성능, 보안, 운영성 중 무엇을 가장 크게 개선하는가?
- 반대로 이 개념을 잘못 적용하면 어떤 복잡도가 추가되는가?
- 이 개념을 적용하면 어떤 문제가 해결되지만, 어떤 비용이나 복잡도가 새로 생기는가?
- 장애가 발생했을 때 이 개념은 문제를 예방하는가, 감지하는가, 복구하는가?
- PayFlow 코드에서 이 개념을 확인하려면 어떤 서비스, 테이블, 로그, 테스트를 봐야 하는가?

#### 핵심 답변

API Versioning 개념은 PayFlow에서 다음 이유로 중요하다.

#### 질문에 답하는 방식

좋은 답변은 용어 정의에서 멈추지 않고 다음 순서로 이어져야 한다.

1. 어떤 데이터나 요청을 보호하려는 개념인지 말한다.
2. 그 데이터의 진실의 원천이 어느 서비스에 있는지 말한다.
3. 장애, 중복, 동시성, 지연 중 어떤 상황에서 문제가 생기는지 설명한다.
4. 코드 레벨의 방어 수단과 운영 레벨의 확인 수단을 함께 말한다.

### PayFlow에 대입해보기

1. 이 개념이 가장 직접적으로 연결되는 PayFlow 서비스 하나를 고른다.
2. 그 서비스가 소유한 데이터의 "진실의 원천"이 무엇인지 적어본다.
3. 동시에 두 요청이 들어오거나, 네트워크가 끊기거나, 프로세스가 죽는 상황을 가정한다.
4. 그 상황에서 데이터가 깨지지 않으려면 어떤 제약조건, 트랜잭션, 락, 이벤트, 재처리 장치가 필요한지 설명한다.
5. 마지막으로 운영자가 문제를 발견할 수 있는 로그나 지표가 무엇인지 적어본다.

### 설명 연습

다음 문장을 자기 말로 완성해보자.

```text
API Versioning 개념은 PayFlow에서 ______ 문제를 막기 위해 필요하다.
이 개념이 없으면 ______ 상황에서 ______ 장애가 발생할 수 있다.
그래서 코드에서는 ______ 방식으로 방어하고,
운영에서는 ______ 지표나 로그로 확인해야 한다.
```

"정의"를 말할 수 있으면 1단계이고, "왜 필요한지"를 말할 수 있으면 2단계다. 실제 장애 상황에 대입해서 "어떻게 막고 어떻게 복구할지"까지 말할 수 있으면 그때부터 실무 지식이 된다.
