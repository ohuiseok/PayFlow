# MSA Service Boundaries

## 핵심 개념

### 서비스 경계란 무엇인가

MSA Service Boundary는 각 서비스가 어떤 책임과 데이터를 소유하는지 나누는 경계다. 좋은 경계는 변경 이유가 비슷한 기능과 데이터를 함께 두고, 다른 서비스의 내부 DB에 직접 접근하지 않게 한다.

서비스 경계를 잘못 나누면 코드를 물리적으로 분리했을 뿐, 실제로는 강하게 결합된 분산 모놀리스가 된다. 이 상태에서는 한 서비스를 변경하면 다른 서비스도 함께 바꿔야 하고, 배포와 장애 격리도 제대로 되지 않는다.

### 도메인 경계를 결정하는 기준: Bounded Context

마이크로서비스 경계는 기술 계층 분리가 아니라 도메인 개념의 자연스러운 경계를 따라야 한다. DDD(Domain-Driven Design)에서 이 단위를 Bounded Context라고 부른다.

Bounded Context 안에서는 같은 용어가 같은 의미를 갖는다. 예를 들어 `user`라는 단어가 `user-service`에서는 인증 주체를 의미하지만, `ledger-service`에서는 원장 항목의 참조 ID에 불과하다. 서비스를 나눌 때 이 의미의 경계를 따라 나누면, 내부 응집도가 높고 외부 결합도가 낮은 서비스 구조가 된다.

Bounded Context를 식별하는 실용적인 기준은 다음과 같다.

- 데이터 소유권: 누가 이 데이터를 생성하고 유일하게 변경할 권한을 갖는가
- 변경 이유: 이 기능이 바뀌는 이유가 다른 기능의 변경 이유와 독립적인가
- 팀 자율성: 이 서비스를 다른 팀이 독립적으로 배포하고 운영할 수 있는가
- 장애 격리: 이 서비스가 다운될 때 다른 서비스가 독립적으로 계속 동작할 수 있는가

### 서비스 간 통신 방식과 결합도

서비스 경계를 정하면 서비스 간 통신 방법도 결정해야 한다. 크게 두 가지 방식이 있다.

**동기 방식 (Synchronous)**

요청한 서비스가 응답을 기다리는 방식이다. REST HTTP 호출이 대표적이다.

```text
transfer-service ---> wallet-service (HTTP POST /debit)
                      [잔액 차감 응답 기다림]
```

장점은 응답 결과를 즉시 알 수 있다는 것이다. 단점은 wallet-service가 느려지거나 다운되면 transfer-service도 영향을 받는다는 것이다.

**비동기 방식 (Asynchronous)**

이벤트나 메시지를 발행하고 응답을 기다리지 않는다.

```text
transfer-service ---> Kafka (TransferCompleted 이벤트 발행)
ledger-service  <--- Kafka (이벤트 소비)
```

장점은 발행 서비스와 소비 서비스가 독립적으로 동작할 수 있다는 것이다. 단점은 처리 결과를 즉시 알 수 없고, 중복 소비, 순서 보장, 관측성이 복잡해진다.

### 데이터 소유권과 Anti-Corruption Layer

서비스 경계에서 가장 중요한 원칙은 데이터 소유권이다. 각 데이터는 단 하나의 서비스만이 그 데이터를 정의하고 변경할 수 있어야 한다.

- 잔액의 진실: `wallet-service`만 잔액을 변경할 수 있다.
- 송금 상태의 진실: `transfer-service`만 송금 상태를 변경할 수 있다.
- 원장 기록의 진실: `ledger-service`만 원장을 기록할 수 있다.

다른 서비스가 이 데이터를 필요로 할 때는 반드시 소유 서비스의 API를 통해 접근해야 한다. 소유 서비스의 DB에 직접 SELECT 하는 것조차 문제다. DB 스키마가 내부 구현이 되어야 하는데, 다른 서비스가 직접 의존하면 스키마 변경 시 모든 의존 서비스가 함께 바뀌어야 한다.

Anti-Corruption Layer(ACL)는 외부 서비스의 모델을 자신의 도메인 모델로 변환하는 중간 계층이다. 외부 서비스의 변경이 내부 도메인에 직접 전파되지 않도록 막는다.

```java
// ACL 예시: 외부 wallet-service 응답을 내부 도메인 모델로 변환
class WalletAdapter {
    WalletBalance getBalance(UserId userId) {
        WalletResponse response = walletClient.getWallet(userId.value());
        return new WalletBalance(
            Money.of(response.getAmount(), Currency.KRW),
            response.getUpdatedAt()
        );
    }
}
```

### 서비스 경계 설계 시 흔한 실수

**너무 잘게 나누는 경우 (Nano-services)**

서비스를 지나치게 작게 나누면 하나의 비즈니스 흐름에 수십 개의 서비스 호출이 필요해진다. 네트워크 지연이 누적되고, 분산 트랜잭션 처리가 복잡해지며, 운영 부담이 기하급수적으로 커진다.

**공유 DB로 인한 암묵적 결합**

서비스를 나눴지만 DB를 공유하면 서비스 경계가 없는 것이나 마찬가지다. 한 서비스가 다른 서비스의 테이블을 직접 수정하면 데이터 정합성 규칙이 무너진다.

```text
// 나쁜 예: transfer-service가 wallet DB를 직접 수정
UPDATE wallet_service.wallets SET balance = balance - 10000 WHERE user_id = 1;

// 이렇게 하면:
// - wallet-service의 잔액 검증 로직을 우회한다
// - wallet-service의 잔액 변경 이벤트가 발행되지 않는다
// - 잔액 이력이 남지 않는다
```

**동기 결합 체인**

A → B → C → D 처럼 동기 호출이 체인으로 연결되면, D가 느려지거나 다운될 때 A부터 D까지 모두 영향을 받는다. 서비스를 나눈 의미가 없어진다. 이런 경우 비동기 이벤트로 전환하거나 Circuit Breaker를 적용해야 한다.

### 서비스 크기와 팀 구조: Conway의 법칙

"시스템 설계는 그 시스템을 만든 조직의 커뮤니케이션 구조를 닮는다."

서비스 경계는 기술적 판단뿐 아니라 팀 구조와도 맞아야 한다. 두 팀이 하나의 서비스를 공동으로 소유하면 배포 충돌과 우선순위 갈등이 생긴다. 반대로 한 팀이 너무 많은 서비스를 소유하면 운영 부담이 커진다.

현실적으로는 초기에는 모놀리식으로 시작해 도메인 이해가 깊어지면 자연스럽게 경계가 보이는 부분부터 서비스를 분리하는 방식이 안전하다.

### 성능과 정합성 trade-off

서비스 경계를 나누면 다음 trade-off가 생긴다.

| 항목 | 모놀리식 | MSA |
|------|---------|-----|
| 데이터 일관성 | 트랜잭션으로 강하게 보장 | 최종 정합성, 보상 트랜잭션 필요 |
| 성능 | 메서드 호출 수준 | 네트워크 지연 누적 |
| 장애 격리 | 한 곳 장애가 전체 영향 | 서비스별 독립 장애 |
| 배포 유연성 | 전체 재배포 | 서비스별 독립 배포 |
| 운영 복잡도 | 낮음 | 높음 (서비스 수에 비례) |

결제 시스템에서는 데이터 정합성이 가장 중요하다. 서비스를 분리하면 하나의 트랜잭션으로 모든 것을 묶기 어렵기 때문에, Saga, Outbox, 멱등성, 보상 처리 같은 추가 메커니즘이 반드시 필요하다.

## PayFlow 연결

PayFlow의 주요 경계는 다음과 같다.

- `user-service`: 사용자와 인증
- `wallet-service`: 지갑과 잔액
- `transfer-service`: 송금 요청과 상태
- `ledger-service`: 원장 기록
- `settlement-service`: 정산
- `api-gateway`: 인증 검증과 라우팅

잔액의 진실은 `wallet-service`가 갖고, 송금 상태의 진실은 `transfer-service`가 갖는다. 다른 서비스는 API나 이벤트를 통해 접근해야 한다.

## 실무 포인트

- 다른 서비스의 DB를 직접 읽지 않는다.
- 서비스 경계는 데이터 소유권 기준으로 정한다.
- 너무 잘게 나누면 통신 비용과 운영 복잡도가 커진다.
- 공통 코드를 공유하더라도 도메인 책임은 섞지 않는다.

## 체크 질문

- 서비스별 DB 분리가 중요한 이유는 무엇인가
- `transfer-service`가 `wallet-service` DB를 직접 수정하면 어떤 문제가 생기는가
- 서비스 경계를 너무 잘게 나누면 어떤 단점이 있는가

## 실무 설계 참고

### 대표 장애 시나리오

transfer-service가 wallet DB를 직접 수정해 잔액 이력과 검증 규칙을 우회한다.

### 잘못된 구현 예시

~~~text
서비스를 나눴지만 DB는 공유해서 편하게 접근한다.
~~~

### 좋은 구현 예시

~~~text
각 서비스가 자기 데이터만 소유하고 다른 서비스에는 API나 이벤트로 요청한다.
~~~

### 대안과 선택 이유

처음에는 모든 서비스를 느슨하게 열어두고 편하게 호출하는 방식이 빠르지만, 운영에 가까워질수록 보안 경계와 설정 혼선이 문제가 된다. PayFlow는 Compose 네트워크, Gateway, 환경 설정 분리, Rate Limit으로 경계를 명확히 하는 방식이 더 낫다.

## 질문의 질문으로 더 깊게 보기

### 이 문서에서 잡아야 할 CS 감각

이 구간의 핵심은 서비스를 나누면 코드만 나뉘는 것이 아니라 네트워크, 배포, 설정, 라우팅, 장애 지점도 함께 늘어난다는 점이다. MSA의 장점은 경계가 명확할 때만 살아난다.

### 질문의 질문

아래 질문들은 답을 외우기 위한 질문이 아니다. 하나씩 답하면서 "왜 그런 설계가 필요한지"를 설명하는 연습을 하기 위한 질문이다.

- MSA Service Boundaries 개념이 없다면 PayFlow에서 가장 먼저 어떤 장애가 생기는가?
- 이 개념은 정확성, 성능, 보안, 운영성 중 무엇을 가장 크게 개선하는가?
- 반대로 이 개념을 잘못 적용하면 어떤 복잡도가 추가되는가?
- 이 개념을 적용하면 어떤 문제가 해결되지만, 어떤 비용이나 복잡도가 새로 생기는가?
- 장애가 발생했을 때 이 개념은 문제를 예방하는가, 감지하는가, 복구하는가?
- PayFlow 코드에서 이 개념을 확인하려면 어떤 서비스, 테이블, 로그, 테스트를 봐야 하는가?

2. 그 데이터의 진실의 원천이 어느 서비스에 있는지 말한다.
3. 장애, 중복, 동시성, 지연 중 어떤 상황에서 문제가 생기는지 설명한다.
4. 코드 레벨의 방어 수단과 운영 레벨의 확인 수단을 함께 말한다.

### PayFlow에 대입해보기

1. 이 개념이 가장 직접적으로 연결되는 PayFlow 서비스 하나를 고른다.
2. 그 서비스가 소유한 데이터의 "진실의 원천"이 무엇인지 적어본다.
3. 동시에 두 요청이 들어오거나, 네트워크가 끊기거나, 프로세스가 죽는 상황을 가정한다.
4. 그 상황에서 데이터가 깨지지 않으려면 어떤 제약조건, 트랜잭션, 락, 이벤트, 재처리 장치가 필요한지 설명한다.
5. 마지막으로 운영자가 문제를 발견할 수 있는 로그나 지표가 무엇인지 적어본다.
