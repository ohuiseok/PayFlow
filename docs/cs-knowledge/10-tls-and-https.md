# TLS And HTTPS

## 핵심 개념

HTTPS는 HTTP 위에 TLS 암호화를 적용한 프로토콜이다. TLS는 통신 내용을 암호화하고, 서버 신원을 검증하며, 중간자 공격을 줄이는 역할을 한다.

로그인, 결제, 개인정보 API는 반드시 HTTPS로 보호되어야 한다. HTTP 평문 통신은 네트워크 중간에서 토큰이나 개인정보가 노출될 수 있다.

### TLS Handshake 동작 원리

TLS는 데이터를 전송하기 전에 Handshake 과정을 통해 암호화 방식과 세션 키를 협상한다. TLS 1.3 기준으로 단순화하면 다음과 같다.

```text
클라이언트                           서버
    |                                 |
    |-- ClientHello (지원 암호 목록) -->|
    |                                 |
    |<-- ServerHello + 인증서 ---------|
    |   (서버 선택 암호 + 공개키)       |
    |                                 |
    |-- 인증서 검증 (CA 체인 확인)       |
    |                                 |
    |-- 세션 키 협상 (Diffie-Hellman) -->|
    |                                 |
    |<====== 암호화된 HTTP 통신 =======>|
```

TLS 1.3에서는 0-RTT 또는 1-RTT로 Handshake가 완료된다. TLS 1.2보다 더 빠르고 안전하다. 서버는 지원하는 최신 TLS 버전을 사용하고, 구버전(TLS 1.0, 1.1)은 비활성화해야 한다.

### 인증서의 역할과 CA 체계

TLS 인증서는 "이 서버가 정말 payflow.example.com인가"를 증명한다. 클라이언트가 서버 신원을 확인하지 않으면 중간자(Man-in-the-Middle) 공격에 취약해진다.

```text
인증서 신뢰 체계 (Certificate Chain):

Root CA (신뢰 앵커, OS/브라우저에 내장)
  └── Intermediate CA (중간 인증서)
        └── End-Entity 인증서 (실제 서버 인증서)

검증 과정:
  1. 서버가 자신의 인증서 + Intermediate CA 인증서 제공
  2. 클라이언트가 서명 체인을 Root CA까지 검증
  3. 인증서 유효 기간 확인
  4. 인증서 폐기 여부 확인 (CRL, OCSP)
  5. CN/SAN 필드와 실제 도메인 일치 확인
```

인증서 만료는 자동 갱신이 없으면 서비스 장애를 일으킨다. Let's Encrypt 같은 무료 CA와 certbot을 쓰면 자동 갱신이 가능하다. 운영 환경에서는 인증서 만료일을 모니터링하고 30일 전에 알림을 받아야 한다.

### 암호화 방식과 보안 수준

TLS에서 사용하는 암호화는 두 단계다.

```text
1. 비대칭 암호화 (Handshake 단계)
   - RSA, ECDH 같은 공개키 암호화
   - 세션 키 협상에 사용
   - 느리지만 키 교환에 안전

2. 대칭 암호화 (데이터 전송 단계)
   - AES-256-GCM 같은 대칭키 암호화
   - 협상된 세션 키로 실제 데이터 암호화
   - 빠르고 효율적
```

Cipher Suite는 TLS에서 사용할 암호화 알고리즘 조합이다. 서버는 강한 알고리즘만 허용하도록 설정해야 한다.

```nginx
# Nginx 예시: 안전한 Cipher Suite 설정
ssl_protocols TLSv1.2 TLSv1.3;
ssl_ciphers ECDHE-RSA-AES256-GCM-SHA384:ECDHE-RSA-AES128-GCM-SHA256;
ssl_prefer_server_ciphers on;
```

### HSTS와 보안 헤더

HTTPS를 강제하려면 HSTS(HTTP Strict Transport Security)를 사용한다. 클라이언트가 한 번 HTTPS로 접속하면 이후 일정 기간 동안 HTTP로 요청해도 자동으로 HTTPS로 전환한다.

```text
Strict-Transport-Security: max-age=31536000; includeSubDomains; preload
```

`max-age`는 HTTPS 강제 기간(초)이다. `includeSubDomains`는 서브도메인에도 적용한다. `preload`는 브라우저 내장 목록에 등록을 신청할 때 사용한다.

다른 보안 헤더들도 함께 설정해야 한다.

```text
X-Content-Type-Options: nosniff
  - MIME 타입 스니핑 방지

X-Frame-Options: DENY
  - 클릭재킹 방지

Content-Security-Policy: default-src 'self'
  - 허용된 리소스 출처 제한

X-XSS-Protection: 1; mode=block
  - 구형 브라우저 XSS 방어 (현대 브라우저는 CSP로 대체)
```

### mTLS (Mutual TLS)

일반 TLS는 서버만 인증서를 제공한다. mTLS는 서버와 클라이언트가 모두 인증서를 제공해 서로를 인증한다.

```text
일반 TLS:
  클라이언트 -> 서버 (서버 인증서만 검증)

mTLS:
  클라이언트 <-> 서버 (서로 인증서 검증)
```

MSA에서 서비스 간 통신에 mTLS를 적용하면 내부 네트워크에서도 각 서비스의 신원을 보장할 수 있다. Istio 같은 Service Mesh는 mTLS를 투명하게 관리한다. PayFlow 같은 소규모 시스템에서는 내부 네트워크 격리로 대신하지만, 제로 트러스트 보안 모델에서는 mTLS가 필수다.

### JWT와 HTTPS의 관계

JWT를 HTTPS 없이 전송하면 심각한 보안 문제가 생긴다.

```text
나쁜 예:
  HTTP 통신 중 JWT가 평문으로 네트워크를 지남
  -> 네트워크 감청으로 토큰 탈취
  -> 공격자가 유효한 JWT로 API 호출 가능
  -> 피해자가 로그아웃해도 토큰이 만료 전까지 유효

또 다른 나쁜 예:
  HTTPS이지만 JWT를 URL 쿼리 파라미터로 전송
  GET /api/transfers?token=eyJhbGc...
  -> URL이 서버 로그, 브라우저 히스토리, Referrer 헤더에 기록됨
  -> JWT가 로그에 남아 노출

올바른 예:
  Authorization: Bearer eyJhbGc...
  HTTPS로 전송 + Authorization 헤더 사용
```

### 흔한 오해와 함정

HTTPS를 쓰면 모든 통신이 안전하다는 오해가 있다. HTTPS는 전송 구간을 보호하지만 서버에 도달한 이후의 처리는 별도 보안이 필요하다.

또한 내부 서비스 간 통신은 HTTP를 써도 된다는 생각은 상황에 따라 다르다. 단일 Docker 네트워크 안에서는 괜찮지만, 데이터센터 간 통신이나 공유 인프라를 쓰는 경우 내부 통신도 암호화가 필요하다.

자체 서명 인증서(Self-Signed Certificate)를 운영 환경에 쓰는 것도 위험하다. 클라이언트가 인증서 검증을 건너뛰도록 설정하면 중간자 공격에 취약해진다.

## PayFlow 연결

PayFlow에서 외부 클라이언트는 Nginx 또는 API Gateway 앞단의 HTTPS 엔드포인트로 접근해야 한다. JWT, 사용자 정보, 송금 요청은 모두 민감한 데이터이므로 평문으로 전달되면 안 된다.

내부 Docker 네트워크에서는 HTTP를 사용할 수 있지만, 운영 환경이나 서비스 간 경계가 넓어지면 mTLS 같은 내부 암호화도 고려할 수 있다.

## 실무 포인트

- 외부 트래픽은 HTTPS를 기본으로 한다.
- JWT를 URL 쿼리 파라미터로 전달하지 않는다.
- TLS 인증서 만료를 모니터링한다.
- HTTP에서 HTTPS로 리다이렉트한다.
- 운영 환경에서는 HSTS도 고려한다.

## 체크 질문

- HTTPS가 막아주는 대표적인 공격은 무엇인가
- JWT를 평문 HTTP로 보내면 어떤 문제가 생기는가
- 내부 통신에도 TLS가 필요해질 수 있는 상황은 언제인가

## 실무 설계 참고

### 대표 장애 시나리오

공용 네트워크에서 JWT가 평문으로 노출되어 공격자가 송금 요청을 재현한다.

### 잘못된 구현 예시

~~~text
로컬에서는 HTTP도 되니까 운영도 비슷하게 두거나, 토큰을 URL query로 넘긴다.
~~~

### 좋은 구현 예시

~~~text
외부 트래픽은 HTTPS로 강제하고 인증서, HSTS, 토큰 전달 위치를 관리한다.
~~~

### 대안과 선택 이유

Gateway에서 한 번만 보안을 처리하는 방식도 있지만, 결제 시스템에서는 단일 방어선이 뚫렸을 때 피해가 크다. PayFlow는 Gateway 인증, 서비스 내부 인가, 입력 검증, 로그 마스킹, Secret 분리를 겹겹이 두는 방식이 더 안전하다.

### PayFlow에서 확인할 위치

nginx 설정, api-gateway 외부 endpoint, docker-compose/nginx, 배포 문서

### 면접에서 설명하기

TLS는 보안 옵션이 아니라 인증 토큰과 결제 요청을 보호하는 기본 전제다.

### 이 문서에서 잡아야 할 CS 감각

이 구간의 핵심은 API가 단순한 입구가 아니라 시스템의 계약이자 보안 경계라는 점이다. 누가 호출할 수 있는지, 어떤 입력을 믿을 수 있는지, 실패를 어떤 형태로 돌려주는지가 서비스 전체의 안정성을 결정한다.

### 질문의 질문

아래 질문들은 답을 외우기 위한 질문이 아니다. 하나씩 답하면서 "왜 그런 설계가 필요한지"를 설명하는 연습을 하기 위한 질문이다.

- 이 요청에서 서버가 믿어도 되는 값과 믿으면 안 되는 값은 무엇인가?
- 인증된 사용자라는 사실만으로 이 리소스에 접근해도 되는가?
- 로그, 에러 응답, 헤더 중 어디로 민감 정보가 새어나갈 수 있는가?
- 이 개념을 적용하면 어떤 문제가 해결되지만, 어떤 비용이나 복잡도가 새로 생기는가?
- 장애가 발생했을 때 이 개념은 문제를 예방하는가, 감지하는가, 복구하는가?
- PayFlow 코드에서 이 개념을 확인하려면 어떤 서비스, 테이블, 로그, 테스트를 봐야 하는가?

#### 핵심 답변

TLS And HTTPS 개념은 PayFlow에서 다음 이유로 중요하다.

- API와 보안 개념은 PayFlow에서 외부 요청이 내부 금전 명령으로 바뀌는 경계를 안전하게 만들기 위해 필요하다.
- Gateway가 인증 정보를 검증하지만, 실제 리소스 소유권의 진실은 각 도메인 서비스의 DB에 있다.

- JWT 검증, 소유권 확인, 입력 검증, 내부 헤더 재생성, 민감 정보 마스킹, Secret 분리로 방어한다.
- 운영에서는 인증 실패율, 인가 실패율, 4xx 비율, 비정상 요청 패턴, 민감 정보 로그 노출 여부를 확인한다.

#### 질문에 답하는 방식

좋은 답변은 용어 정의에서 멈추지 않고 다음 순서로 이어져야 한다.

1. 어떤 데이터나 요청을 보호하려는 개념인지 말한다.
2. 그 데이터의 진실의 원천이 어느 서비스에 있는지 말한다.
3. 장애, 중복, 동시성, 지연 중 어떤 상황에서 문제가 생기는지 설명한다.
4. 코드 레벨의 방어 수단과 운영 레벨의 확인 수단을 함께 말한다.

### PayFlow에 대입해보기

1. 이 개념이 가장 직접적으로 연결되는 PayFlow 서비스 하나를 고른다.
2. 그 서비스가 소유한 데이터의 "진실의 원천"이 무엇인지 적어본다.
3. 동시에 두 요청이 들어오거나, 네트워크가 끊기거나, 프로세스가 죽는 상황을 가정한다.
4. 그 상황에서 데이터가 깨지지 않으려면 어떤 제약조건, 트랜잭션, 락, 이벤트, 재처리 장치가 필요한지 설명한다.
5. 마지막으로 운영자가 문제를 발견할 수 있는 로그나 지표가 무엇인지 적어본다.
