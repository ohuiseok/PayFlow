# Rate Limiting

> 도메인 전환 안내: 현재 PayFlow는 **청년 정책 참여 미션 및 지원금 지급 플랫폼**으로 설명한다. 내부 구현 호환성을 위해 `PARENT`/`CHILD`, `/api/families`, `/api/missions`, `/api/cashbook`, `reward-service` 같은 명칭은 유지하지만, 문서와 발표에서는 각각 **기관 담당자**, **청년 참여자**, **참여자 연결**, **정책 미션**, **지원금 사용 내역**, **정책 미션/지원금 서비스**로 해석한다.

## 핵심 개념

### Rate Limiting이 필요한 이유

Rate Limiting은 사용자가 일정 시간 동안 보낼 수 있는 요청 수를 제한하는 기법이다. 목적은 크게 세 가지다.

- **시스템 보호**: 과도한 요청이 DB, 캐시, 내부 서비스에 부하를 주지 않도록 한다.
- **보안 공격 완화**: Brute Force(비밀번호 반복 시도), Credential Stuffing, DDoS 공격을 어렵게 만든다.
- **공정한 사용 보장**: 일부 사용자의 과도한 요청이 다른 사용자의 서비스 품질을 떨어뜨리지 않도록 한다.

결제 시스템에서 Rate Limiting이 없으면 공격자가 송금 API를 초당 수천 번 호출해 DB 커넥션 풀을 소진시키거나, Redis 락을 압박하거나, Kafka 프로듀서를 과부하 시킬 수 있다.

### 핵심 알고리즘 4가지

**1. Fixed Window (고정 윈도우)**

시간을 고정된 구간으로 나누고 각 구간 내 요청 수를 카운트한다.

```text
[00:00 - 01:00] 요청 카운트: 0
00:30에 요청 → 카운트: 1
00:45에 요청 → 카운트: 2
...한도 초과 시 거부

01:00이 되면 카운트 리셋
```

구현이 단순하지만 경계 부근에서 버스트 문제가 있다. 00:59에 100개, 01:01에 100개 요청이 오면 2초 안에 200개 요청이 처리된다.

```python
# Redis 기반 Fixed Window 예시
def is_allowed(user_id: str, limit: int, window_seconds: int) -> bool:
    key = f"rate:{user_id}:{int(time.time() / window_seconds)}"
    count = redis.incr(key)
    if count == 1:
        redis.expire(key, window_seconds)
    return count <= limit
```

**2. Sliding Window Log (슬라이딩 윈도우 로그)**

각 요청의 타임스탬프를 로그에 저장하고 현재 시점 기준 과거 N초 내 요청 수를 세는 방식이다. 정확하지만 메모리 사용량이 크다.

**3. Token Bucket (토큰 버킷)**

버킷에 일정 속도로 토큰이 채워지고, 요청이 올 때마다 토큰 하나를 소비한다. 버킷이 비어 있으면 요청을 거부한다.

```text
버킷 용량: 10개 토큰
토큰 보충 속도: 초당 2개
현재 토큰: 7개

요청 → 토큰 1개 소비 → 남은 토큰: 6개
```

장점은 짧은 시간 동안 버스트 트래픽을 허용하면서도 평균 처리 속도를 제어할 수 있다는 점이다. Stripe, AWS API Gateway 등에서 많이 사용된다.

**4. Leaky Bucket (누수 버킷)**

요청이 들어오면 버킷에 쌓이고, 일정한 속도로만 처리된다. 버킷이 가득 차면 새 요청을 거부한다. 출력 속도가 일정하므로 백엔드 시스템에 안정적인 부하를 줄 수 있다.

| 알고리즘 | 버스트 허용 | 구현 복잡도 | 메모리 사용 | 특징 |
|---------|-----------|-----------|-----------|------|
| Fixed Window | 경계 부근 버스트 | 낮음 | 낮음 | 단순, 경계 취약 |
| Sliding Window Log | 없음 | 중간 | 높음 | 정확, 메모리 비효율 |
| Token Bucket | 있음 (버킷 용량까지) | 중간 | 낮음 | 버스트 친화적 |
| Leaky Bucket | 없음 | 중간 | 낮음 | 안정적 처리율 |

### 제한 기준 선택

Rate Limit의 기준을 무엇으로 할지 선택해야 한다.

- **IP 기반**: 구현이 단순하지만 NAT 뒤에 있는 사용자들이 같은 IP를 공유해 정상 사용자가 피해를 볼 수 있다.
- **사용자 ID 기반**: 인증된 사용자를 정확히 제어할 수 있다. 로그인 후에만 적용 가능하다.
- **API Key 기반**: B2B API에서 클라이언트별로 제한할 때 사용한다.
- **사용자 ID + API 경로 조합**: 특정 API에 더 엄격한 제한을 적용할 때 사용한다.

```text
// 제한 키 예시
rate:user:{userId}:transfer          → 사용자별 송금 API 제한
rate:user:{userId}:login             → 사용자별 로그인 시도 제한
rate:ip:{ipAddress}:global           → IP 기반 전체 제한
```

### 분산 환경에서의 Rate Limiting

단일 서버에서는 메모리에 카운터를 저장하면 된다. 하지만 여러 Gateway 인스턴스가 실행되는 분산 환경에서는 각 인스턴스가 카운터를 공유해야 한다.

Redis를 중앙 카운터 저장소로 사용하는 것이 일반적인 해결책이다.

```text
Gateway 인스턴스 1 Gateway 인스턴스 2  --> Redis (공유 카운터) --> 일관된 Rate Limit 적용
Gateway 인스턴스 3 /
```

Redis의 INCR 명령은 원자적이므로 여러 인스턴스가 동시에 카운터를 증가시켜도 정확한 값을 유지한다.

### 응답 형식과 클라이언트 안내

Rate Limit 초과 시 `429 Too Many Requests`를 반환한다. 클라이언트가 언제 재시도할 수 있는지 알 수 있도록 헤더를 함께 제공하는 것이 좋다.

```http
HTTP/1.1 429 Too Many Requests
Retry-After: 30
X-RateLimit-Limit: 100
X-RateLimit-Remaining: 0
X-RateLimit-Reset: 1716998400

{
  "error": "RATE_LIMIT_EXCEEDED",
  "message": "요청 한도를 초과했습니다. 30초 후 다시 시도하세요."
}
```

### API별 차등 정책

모든 API에 같은 제한을 적용하면 안 된다. 리스크와 비용에 따라 다르게 설정해야 한다.

```text
로그인 API:        분당 10회 (Brute Force 방지)
비밀번호 재설정:   시간당 5회 (OTP 발송 비용)
송금 API:          분당 5회 (오류 방지 + 비용 제어)
잔액 조회 API:     분당 60회 (읽기 전용이므로 관대하게)
공개 API (헬스):   무제한
```

### 멱등성과 Rate Limiting의 관계

멱등성이 있는 API라도 Rate Limiting은 필요하다. 멱등성은 "같은 결과를 만든다"는 보장이지, "처리 비용이 없다"는 의미가 아니다. 같은 Idempotency-Key로 1000번 요청이 오면 1000번 모두 DB에서 기존 결과를 조회하는 비용이 발생한다.

## PayFlow 연결

송금 API, 로그인 API, 지갑 충전 API는 무제한 호출을 허용하면 위험하다. 공격자가 반복 요청을 보내면 DB, Redis, Kafka, 내부 서비스가 과부하될 수 있다.

특히 결제 요청은 멱등성이 있어도 요청 처리 비용이 발생하므로 제한이 필요하다.

## 실무 포인트

- 사용자 ID, IP, API Key 단위로 제한할 수 있다.
- 로그인과 송금은 더 엄격한 제한이 필요하다.
- Rate Limit 초과 시 `429 Too Many Requests`를 반환한다.
- 내부 서비스 보호용 제한도 고려한다.
- 정상 사용자를 과도하게 막지 않도록 기준을 조정한다.

## 체크 질문

- Rate Limiting은 보안과 성능 중 어디에 더 가까운가
- 송금 API에 Rate Limit이 필요한 이유는 무엇인가
- `429` 상태 코드는 언제 사용하는가

## 실무 설계 참고

### 대표 장애 시나리오

짧은 시간에 동일 사용자가 송금 API를 반복 호출해 DB와 Redis 락을 압박한다.

### 잘못된 구현 예시

~~~text
멱등성이 있으니 호출이 많아도 안전하다고 생각한다.
~~~

### 좋은 구현 예시

~~~text
사용자/IP/API 단위 제한과 429 응답, 중요 API 별도 정책을 둔다.
~~~

### 대안과 선택 이유

처음에는 모든 서비스를 느슨하게 열어두고 편하게 호출하는 방식이 빠르지만, 운영에 가까워질수록 보안 경계와 설정 혼선이 문제가 된다. PayFlow는 Compose 네트워크, Gateway, 환경 설정 분리, Rate Limit으로 경계를 명확히 하는 방식이 더 낫다.

### PayFlow에서 확인할 위치

api-gateway filter, Redis rate limit key, access log

### 면접에서 설명하기

Rate Limit은 보안과 안정성을 동시에 위한 방어선이다. 멱등성은 비용 폭증까지 막아주지는 않는다.

### PayFlow에 대입해보기

1. 이 개념이 가장 직접적으로 연결되는 PayFlow 서비스 하나를 고른다.
2. 그 서비스가 소유한 데이터의 "진실의 원천"이 무엇인지 적어본다.
3. 동시에 두 요청이 들어오거나, 네트워크가 끊기거나, 프로세스가 죽는 상황을 가정한다.
4. 그 상황에서 데이터가 깨지지 않으려면 어떤 제약조건, 트랜잭션, 락, 이벤트, 재처리 장치가 필요한지 설명한다.
5. 마지막으로 운영자가 문제를 발견할 수 있는 로그나 지표가 무엇인지 적어본다.
