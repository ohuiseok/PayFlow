# CQRS

> 도메인 전환 안내: 현재 PayFlow는 **청년 정책 참여 미션 및 지원금 지급 플랫폼**으로 설명한다. 내부 구현 호환성을 위해 `PARENT`/`CHILD`, `/api/families`, `/api/missions`, `/api/cashbook`, `reward-service` 같은 명칭은 유지하지만, 문서와 발표에서는 각각 **기관 담당자**, **청년 참여자**, **참여자 연결**, **정책 미션**, **지원금 사용 내역**, **정책 미션/지원금 서비스**로 해석한다.

## 핵심 개념

### CQRS의 기본 아이디어

CQRS(Command Query Responsibility Segregation)는 데이터를 변경하는 작업(Command)과 데이터를 읽는 작업(Query)의 모델을 분리하는 패턴이다. Bertrand Meyer의 CQS(Command-Query Separation) 원칙을 아키텍처 수준으로 확장한 것이다.

전통적인 CRUD에서는 같은 모델로 읽기와 쓰기를 모두 처리한다.

```text
// 하나의 모델로 읽기/쓰기를 모두 처리 (전통적)
TransferRepository.save(transfer)    // 쓰기
TransferRepository.findById(id)      // 읽기 - 같은 모델 사용
```

CQRS에서는 쓰기 모델과 읽기 모델을 분리한다.

```text
// CQRS: 쓰기와 읽기 모델 분리
TransferCommandService.createTransfer(command)  // 쓰기: 검증 중심 모델
TransferQueryService.getTransferSummary(query)  // 읽기: 조회 최적화 모델
```

### Command와 Query의 차이

**Command (명령)**

시스템 상태를 변경하는 작업이다. 비즈니스 규칙 검증이 핵심이다.

```java
// Command 예시: 상태 변경 + 검증 중심
public class CreateTransferCommand {
    private String senderId;
    private String receiverId;
    private Money amount;
    private String idempotencyKey;
}

public TransferId handle(CreateTransferCommand command) {
    // 검증: 잔액 충분한가?
    // 검증: 일일 한도 초과하지 않는가?
    // 검증: 수신자가 존재하는가?
    // 도메인 이벤트 발행
    // 결과: 송금 ID만 반환
}
```

Command는 복잡한 응답 모델이 필요 없다. 성공 여부와 생성된 ID 정도면 충분하다.

**Query (조회)**

시스템 상태를 읽는 작업이다. 성능과 사용성이 핵심이다.

```java
// Query 예시: 화면에 최적화된 응답
public class TransferSummaryQuery {
    private String userId;
    private LocalDate from;
    private LocalDate to;
    private int page;
}

public Page<TransferSummaryDTO> handle(TransferSummaryQuery query) {
    // 인덱스 최적화된 쿼리
    // 조인 없이 비정규화된 데이터 조회
    // 페이지네이션
    // 결과: 화면에서 바로 사용할 수 있는 형태
}
```

Query는 DB 트랜잭션이 필요 없고, 읽기 전용이므로 더 자유롭게 최적화할 수 있다.

### 왜 쓰기와 읽기 요구사항이 다른가

결제 시스템에서 쓰기와 읽기의 요구사항은 본질적으로 다르다.

**쓰기 작업의 특성**:
- 복잡한 비즈니스 규칙 검증 필요
- 강한 일관성 요구
- 동시성 충돌 방지 (낙관적/비관적 락)
- 트랜잭션 안에서 처리
- 도메인 모델이 복잡함
- 처리량보다 정확성이 우선

**읽기 작업의 특성**:
- 다양한 화면/API에 맞는 형태 필요
- 빠른 응답 시간 우선
- 복잡한 조인, 집계, 정렬
- 트랜잭션 불필요
- 비정규화가 성능에 유리
- 정확성보다 성능이 우선인 경우도 있음

하나의 모델로 두 요구사항을 모두 만족하려면 모델이 복잡해지거나 성능 타협이 생긴다.

### CQRS 구현 수준

CQRS를 어느 수준까지 적용할지는 필요에 따라 선택한다.

**수준 1: 코드 레벨 분리**

같은 DB를 쓰지만 Command와 Query 핸들러를 분리한다. 가장 단순한 형태다.

```java
// 같은 DB, 다른 핸들러
@Service class TransferCommandHandler { ... }  // 쓰기
@Service class TransferQueryHandler { ... }    // 읽기
```

**수준 2: 읽기 모델 분리**

쓰기 DB와 별도로 읽기에 최적화된 테이블을 만든다. 이벤트나 동기화로 읽기 테이블을 갱신한다.

```text
쓰기: transfers 테이블 (정규화, 트랜잭션)
읽기: transfer_summaries 뷰/테이블 (비정규화, 조회 최적화)

Transfer 저장 시 → transfer_summaries 갱신 (동기 또는 이벤트)
```

**수준 3: DB 분리**

쓰기 DB와 읽기 DB를 물리적으로 분리한다. 읽기 DB는 레플리카이거나, Elasticsearch 같은 다른 종류의 저장소일 수 있다.

```text
쓰기: PostgreSQL (쓰기 마스터)
읽기: PostgreSQL 읽기 레플리카 + Elasticsearch (검색 최적화)
```

### 읽기 모델 동기화와 최종 정합성

쓰기 모델과 읽기 모델이 분리되면 두 모델 사이에 지연이 생길 수 있다. 쓰기 후 읽기 모델이 즉시 갱신되지 않는 최종 정합성 상태가 된다.

```text
// 쓰기 완료 직후 읽기 시 최신 데이터가 없을 수 있음
1. 송금 완료 (쓰기 DB 업데이트)
2. 사용자가 즉시 조회 → 읽기 DB가 아직 갱신 안 됨 → 이전 상태 표시
3. 100ms 후 → 읽기 DB 갱신 완료
4. 재조회 → 최신 상태 표시
```

이 지연을 허용할 수 없는 경우(예: 방금 처리한 송금의 결과를 즉시 확인해야 하는 경우)에는 쓰기 DB에서 직접 조회하거나, 응답에서 방금 처리한 결과를 직접 반환하는 방식이 필요하다.

### CQRS 적용 시 주의점

**복잡도 증가**

CQRS는 읽기와 쓰기 모델을 별도로 유지해야 하므로 코드와 인프라 복잡도가 증가한다. 단순한 CRUD에 CQRS를 적용하면 과도한 엔지니어링이 된다.

**읽기 모델 동기화 실패**

이벤트 기반으로 읽기 모델을 갱신하는 경우, 이벤트 소비 실패로 읽기 모델이 쓰기 모델과 오랫동안 불일치할 수 있다. 동기화 실패를 모니터링하고 재처리하는 메커니즘이 필요하다.

**CQRS가 항상 필요한 것은 아님**

CQRS는 읽기와 쓰기의 요구사항이 크게 다를 때, 또는 읽기 성능이 병목일 때 도입을 고려한다. 트래픽이 작거나 요구사항이 단순한 경우에는 전통적인 CRUD가 더 적합하다.

### Event Sourcing과의 관계

CQRS와 Event Sourcing은 자주 함께 언급되지만 독립적인 패턴이다.

- CQRS 없는 Event Sourcing: 이벤트를 저장하고 재생하지만 읽기 모델은 분리하지 않는다.
- Event Sourcing 없는 CQRS: 현재 상태를 저장하면서 읽기와 쓰기 모델만 분리한다.
- CQRS + Event Sourcing: 이벤트 저장소를 쓰기 모델로, 이벤트를 소비해 생성한 프로젝션을 읽기 모델로 사용한다. 복잡하지만 강력한 감사 추적과 시점 재현이 가능하다.

## PayFlow 연결

PayFlow에서 송금 생성은 강한 검증과 트랜잭션이 필요한 명령이다. 반면 송금 내역 조회나 정산 통계 조회는 빠른 검색과 정렬이 중요하다.

나중에 트래픽이 커지면 쓰기 DB와 조회용 모델을 분리하거나, 이벤트를 기반으로 조회 테이블을 구성할 수 있다.

## 실무 포인트

- CQRS는 복잡도를 증가시키므로 필요할 때만 도입한다.
- 조회 모델은 최종 정합성이 될 수 있다.
- 명령 API는 비즈니스 규칙을 엄격히 검증한다.
- 조회 API는 성능과 사용성에 맞게 모델링한다.

## 체크 질문

- Command와 Query의 차이는 무엇인가
- 송금 생성 API와 송금 목록 조회 API의 요구사항은 어떻게 다른가
- CQRS가 항상 필요한 것은 아닌 이유는 무엇인가

## 실무 설계 참고

### 대표 장애 시나리오

송금 목록 조회 최적화 요구가 송금 생성 도메인 모델을 복잡하게 만든다.

### 잘못된 구현 예시

~~~text
쓰기 모델 하나로 모든 조회 화면 요구를 해결하려고 한다.
~~~

### 좋은 구현 예시

~~~text
명령 모델은 검증 중심으로, 조회 모델은 화면과 성능 중심으로 분리한다.
~~~

### 대안과 선택 이유

분산 트랜잭션이나 모든 후속 처리를 동기 호출로 묶는 방식도 있지만, 서비스 하나의 장애가 전체 송금 실패로 번지기 쉽다. PayFlow는 로컬 트랜잭션, Saga, 도메인 이벤트, Outbox, 멱등성을 조합해 부분 실패를 상태로 남기고 복구하는 방식을 선택하는 것이 적합하다.

### PayFlow에서 확인할 위치

transfer command API, transfer query API, read model table

## 질문의 질문으로 더 깊게 보기

### 이 문서에서 잡아야 할 CS 감각

이 구간의 핵심은 여러 서비스의 작업을 하나의 로컬 트랜잭션처럼 다룰 수 없다는 사실이다. 그래서 Saga, Domain Event, Outbox, 멱등성이 등장한다. 이들은 모두 "부분 실패를 어떻게 추적하고 복구할 것인가"에 대한 답이다.

### 질문의 질문

아래 질문들은 답을 외우기 위한 질문이 아니다. 하나씩 답하면서 "왜 그런 설계가 필요한지"를 설명하는 연습을 하기 위한 질문이다.

- CQRS 개념이 없다면 PayFlow에서 가장 먼저 어떤 장애가 생기는가?
- 이 개념은 정확성, 성능, 보안, 운영성 중 무엇을 가장 크게 개선하는가?
- 반대로 이 개념을 잘못 적용하면 어떤 복잡도가 추가되는가?
- 이 개념을 적용하면 어떤 문제가 해결되지만, 어떤 비용이나 복잡도가 새로 생기는가?
- 장애가 발생했을 때 이 개념은 문제를 예방하는가, 감지하는가, 복구하는가?
- PayFlow 코드에서 이 개념을 확인하려면 어떤 서비스, 테이블, 로그, 테스트를 봐야 하는가?

### PayFlow에 대입해보기

1. 이 개념이 가장 직접적으로 연결되는 PayFlow 서비스 하나를 고른다.
2. 그 서비스가 소유한 데이터의 "진실의 원천"이 무엇인지 적어본다.
3. 동시에 두 요청이 들어오거나, 네트워크가 끊기거나, 프로세스가 죽는 상황을 가정한다.
4. 그 상황에서 데이터가 깨지지 않으려면 어떤 제약조건, 트랜잭션, 락, 이벤트, 재처리 장치가 필요한지 설명한다.
5. 마지막으로 운영자가 문제를 발견할 수 있는 로그나 지표가 무엇인지 적어본다.
