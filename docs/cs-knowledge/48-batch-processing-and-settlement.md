# Batch Processing And Settlement

> 도메인 전환 안내: 현재 PayFlow는 **청년 정책 참여 미션 및 지원금 지급 플랫폼**으로 설명한다. 내부 구현 호환성을 위해 `PARENT`/`CHILD`, `/api/families`, `/api/missions`, `/api/cashbook`, `reward-service` 같은 명칭은 유지하지만, 문서와 발표에서는 각각 **기관 담당자**, **청년 참여자**, **참여자 연결**, **정책 미션**, **지원금 사용 내역**, **정책 미션/지원금 서비스**로 해석한다.

## 핵심 개념

Batch Processing은 대량 데이터를 일정 단위로 묶어 처리하는 방식이다. Settlement, 정산은 거래 내역을 기준으로 수수료, 지급액, 집계 결과를 확정하는 과정이다.

## PayFlow 연결

PayFlow의 `settlement-service`는 일별 송금 데이터를 집계하고 수수료를 계산하며 정산 결과를 생성한다. 실시간 API와 달리 정산은 기준일, 처리 상태, 재실행 가능성이 중요하다.

정산 배치는 중간에 실패할 수 있으므로 어디까지 처리했는지 상태를 남겨야 한다.

## 실무 포인트

- 기준일을 명확히 정의한다.
- 배치는 재실행 가능해야 한다.
- 같은 거래가 두 번 정산되지 않게 한다.
- 처리 단위를 chunk로 나눈다.
- 실패 내역을 추적 가능하게 저장한다.

## 체크 질문

- 정산 배치가 재실행 가능해야 하는 이유는 무엇인가
- 실시간 API와 배치 처리는 어떤 점이 다른가
- 기준일이 정산에서 중요한 이유는 무엇인가

## 실무 설계 참고

### 대표 장애 시나리오

일별 정산 배치가 중간에 실패해 일부 거래만 정산된다.

### 잘못된 구현 예시

~~~text
배치는 한 번 성공한다고 가정하고 중간 상태를 저장하지 않는다.
~~~

### 좋은 구현 예시

~~~text
chunk 처리, 처리 상태, 배치 실행 ID, 재시작 가능 구조를 사용한다.
~~~

### 대안과 선택 이유

현재 잔액이나 최종 집계값만 저장하는 단순한 방식도 있지만, 결제 시스템에서는 나중에 왜 그런 결과가 나왔는지 설명할 수 있어야 한다. PayFlow는 금액 정책, 원장, 이력, 정산 결과, 대사 리포트를 남기는 방식이 분쟁 대응과 복구에 더 강하다.

### PayFlow에서 확인할 위치

settlement-service batch job, settlement tables, scheduler

### 면접에서 설명하기

배치는 오래 걸리는 트랜잭션이 아니라 재시작 가능한 처리 흐름이어야 한다.

### 관련 문서

44, 50, 52

## 질문의 질문으로 더 깊게 보기

### 이 문서에서 잡아야 할 CS 감각

이 구간의 핵심은 결제 도메인의 불변식이다. 금액은 정확해야 하고, 잔액은 설명 가능해야 하며, 원장과 정산은 나중에 검증 가능해야 한다. 좋은 결제 시스템은 성공 처리보다 설명 가능한 기록을 더 중요하게 본다.

### 질문의 질문

아래 질문들은 답을 외우기 위한 질문이 아니다. 하나씩 답하면서 "왜 그런 설계가 필요한지"를 설명하는 연습을 하기 위한 질문이다.

- 이 데이터가 틀렸을 때 돈이 실제로 잘못 움직이는가, 아니면 후속 기록만 늦어지는가?
- 원천 데이터와 집계 데이터가 다를 때 무엇을 진실로 볼 것인가?
- 나중에 분쟁이 생겼을 때 이 기록만 보고 거래를 설명할 수 있는가?
- 이 개념을 적용하면 어떤 문제가 해결되지만, 어떤 비용이나 복잡도가 새로 생기는가?
- 장애가 발생했을 때 이 개념은 문제를 예방하는가, 감지하는가, 복구하는가?
- PayFlow 코드에서 이 개념을 확인하려면 어떤 서비스, 테이블, 로그, 테스트를 봐야 하는가?

#### 핵심 답변

Batch Processing And Settlement 개념은 PayFlow에서 다음 이유로 중요하다.

- 정산과 대사는 PayFlow에서 거래 결과가 최종적으로 맞는지 검증하고 확정하기 위해 필요하다.
- 지갑 잔액, 잔액 변경 이력, 원장 엔트리, 정산 결과, 대사 리포트가 진실을 검증하는 기준이다.
- 원장 누락, 중복 정산, 수수료 계산 오류가 발견되지 않으면 실제 지급액이 틀어질 수 있다.
- 기준일, 배치 실행 ID, 유니크 제약, 정산 검증, 대사 리포트, 재처리 절차로 방어한다.
- 운영에서는 잔액-원장 차이, 정산 검증 실패 수, 수수료 합계 차이, 중복 정산 차단 건수, 대사 실패 리포트를 본다.

#### 질문에 답하는 방식

좋은 답변은 용어 정의에서 멈추지 않고 다음 순서로 이어져야 한다.

1. 어떤 데이터나 요청을 보호하려는 개념인지 말한다.
2. 그 데이터의 진실의 원천이 어느 서비스에 있는지 말한다.
3. 장애, 중복, 동시성, 지연 중 어떤 상황에서 문제가 생기는지 설명한다.
4. 코드 레벨의 방어 수단과 운영 레벨의 확인 수단을 함께 말한다.

#### PayFlow 예시 답변

```text
Batch Processing And Settlement 개념은 PayFlow에서 일별 송금 데이터를 안정적으로 집계하고 정산 결과를 확정하기 위해 필요하다.
이 개념이 없으면 정산 배치 중간 실패 후 재실행 시 일부 거래가 두 번 반영될 수 있다.
그래서 코드에서는 기준일, 배치 실행 ID, chunk 처리, 처리 상태, 유니크 제약을 사용하고,
운영에서는 정산 성공/실패, 처리 건수, 재실행 결과, 중복 정산 차단을 확인해야 한다.
```

#### 더 생각해볼 점

이 답안은 하나의 예시다. 실제 설계에서는 성능, 복잡도, 장애 복구 난이도 사이의 trade-off를 함께 봐야 한다. 특히 PayFlow처럼 돈을 다루는 시스템에서는 빠른 성공보다 명확한 실패, 추적 가능한 상태, 재처리 가능한 구조가 더 중요하다.

### PayFlow에 대입해보기

1. 이 개념이 가장 직접적으로 연결되는 PayFlow 서비스 하나를 고른다.
2. 그 서비스가 소유한 데이터의 "진실의 원천"이 무엇인지 적어본다.
3. 동시에 두 요청이 들어오거나, 네트워크가 끊기거나, 프로세스가 죽는 상황을 가정한다.
4. 그 상황에서 데이터가 깨지지 않으려면 어떤 제약조건, 트랜잭션, 락, 이벤트, 재처리 장치가 필요한지 설명한다.
5. 마지막으로 운영자가 문제를 발견할 수 있는 로그나 지표가 무엇인지 적어본다.
